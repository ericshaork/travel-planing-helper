# MVP v1.2 分阶段实施计划

## 1. 使用方式

本文把 v1.2 拆为 0—6 七个阶段。每次只进入一个阶段；上一阶段验收通过并完成汇报后，才允许开始下一阶段。

当前文档已同步到阶段 6 收尾状态，用于记录 v1.2 实际完成范围、已知限制和部署前检查。

## 2. 总体范围

版本主题：

> Mobile First + Cabinet Identity Upgrade + Pending Changes

三条主线：

1. `/plan` 与 `/result` 的移动端体验真正以手机为先；
2. 行程柜从普通卡片升级为稳定的柜门 / 格层 / 积木结构；
3. 多个修改意图先进入 Pending Changes，再统一写入现有修改框。

总原则：Mobile Flow First。

- 从首页输入、`/plan` 补信息到 `/result` 浏览结果，移动端都优先采用一屏一任务；
- 移动端优先使用步骤切换、分页浏览、分屏浏览或 Tab 切换；
- 不把桌面端长页面直接压缩成手机长页面；
- 桌面端继续保持完整展开式布局；
- 移动端和桌面端允许不同布局，但必须复用同一套数据和业务逻辑。

固定技术边界：

- 复用当前 `TripPlan`、`DayCabinetView`、`ItineraryBlock`；
- 不改 `TripPlan` Schema；
- 不改 `/api/generate-trip` 协议；
- 不做局部 Patch；
- Pending Changes 只是前端 UI 状态；
- 最终仍使用一个 `modificationRequest`；
- `RegenerateBox` 仍是唯一提交入口。

## 3. 阶段依赖

```text
阶段 0 规划与冻结
  ↓
阶段 1 /plan 手机问卷
  ↓
阶段 2 /result 手机分页式浏览
  ↓
阶段 3 Cabinet Identity
  ↓
阶段 4 Pending Changes
  ↓
阶段 5 轻工作台雏形
  ↓
阶段 6 总验收、文档与部署
```

阶段 1 与阶段 2 技术上可独立，但为减少并行 UI 变量，按顺序实施。

## 9. 阶段 4：Pending Changes 修改篮

### 目标

- 支持多个积木动作与快捷修改统一收集；
- 提供查看、删除单项、清空和数量提示；
- 确定性处理重复与同积木冲突；
- 一次合并为 `modificationRequest` 并写入现有 `RegenerateBox`；
- 写入不等于提交，最终仍由用户手动重排。

### 允许修改文件

- `app/result/page.tsx`
- `components/trip/BlockActions.tsx`
- `components/trip/ModificationQuickActions.tsx`
- `components/trip/RegenerateBox.tsx`

- `tests/trip/modification-intents.test.ts`
- `tests/trip/pending-changes.test.ts`（新增）
- `app/globals.css`

### 不做什么

- 不把 Pending Changes 写入 localStorage；
- 不实现持久化 constraints；
- 不新增 API 字段；
- 不新增 API route；
- 不自动提交；
- 不做局部 Patch；
- 不反向解析用户自由文本；
- 不做历史版本或撤销栈；
- 不引入全局状态库。

### 验收标准

- [ ] A 保留、B 替换、C 加类似可同时存在；
- [ ] 不同 block 的操作互不覆盖；
- [ ] 完全重复意图只保留一条；
- [ ] 同 block 的 remove / replace / lock 后选覆盖前选；
- [ ] addSimilar 可按规则与 lock 并存；
- [ ] 可删除单项和清空；
- [ ] 空清单不能写入；
- [ ] 写入后生成一个可读的 `modificationRequest`；
- [ ] 原修改框已有文字时不会被静默覆盖；
- [ ] 超过 1000 字时不截断、不改两边状态；
- [ ] 写入不请求网络；
- [ ] `RegenerateBox` 仍是唯一提交入口；
- [ ] 重排成功后旧 Pending Changes 被清空；
- [ ] 重排失败后旧 `TripPlan` 和修改框文本保留。

- 验证成功、失败、重复点击和生成中禁用；
- 阶段结束运行 lint、typecheck、build、test。

- PendingChange 类型与状态所有者；
- 去重和冲突规则；

## 13. 阶段通用退出条件

每阶段结束都必须：

1. 只修改该阶段允许文件，额外文件要在汇报中解释；
2. 验收标准逐项确认；
3. 运行相关测试与全量四命令；
4. 记录移动端和桌面端结果；
5. 明确列出没有做的内容；
6. 无阻塞问题才进入下一阶段。

若出现以下任一情况，应停在当前阶段：

- 需要修改 `TripPlan` Schema 才能继续；
- 需要修改 `/api/generate-trip` 协议才可完成；
- 移动端改动导致桌面主流程明显退化；
- 修改篮与修改框形成无法解释的双数据源；
- 必须引入真实 API 才能兑现当前文案；
- 自动化检查失败且原因未查明。
