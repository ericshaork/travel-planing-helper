# 产品路线图：从 AI 行程生成器到 AI 旅行计划工作台

## 1. 路线图目的

本文描述“漫游草稿”从 MVP 1.0 到 2.0+ 的中期演进方向，用于统一产品判断、版本边界和开发顺序。它不是功能愿望清单：每个版本只解决一个主要问题，后一阶段是否启动，要以前一阶段的用户反馈和技术稳定性为依据。

当前判断：

- MVP 1.0 已完成“先生成一版”的闭环；
- MVP 1.1 已完成点击式行程编辑 MVP，并上线测试；
- MVP 1.2 已完成 Mobile Flow、Cabinet Identity、Pending Changes 与轻工作台雏形；
- MVP 1.2.5 再接入真实 AI 的最小冒烟能力；
- MVP 1.3 再接入国内真实性 API 与地图雏形；
- 保存分享与可拖拽工作台继续后置，不提前塞进 v1.2。

补充判断：竞品分析已经形成明确分层。

- Wanderlog / Mindtrip 的工作台与地图联动更适合进入 2.0；
- Roadtrippers 的路线、出行方式与地图基础更适合进入 v1.3；
- TripIt 的“已有计划整理”更适合进入 v1.4；
- Layla 的图片、短视频与路线动效只保留为中长期增强方向，不进入 v1.2。

## 2. 产品中期愿景

产品从“AI 行程生成器”升级为“强调个性化、省时间、可编辑的 AI 旅行积木板”。

用户输入旅行需求后，AI 先生成一版可执行行程。用户不需要从零查攻略，也不必被一份死板的 AI 行程绑住，而是可以像整理柜子、搭积木一样调整每天安排：删掉不喜欢的积木，换掉不合适的积木，锁定必去积木，加入类似地点或美食夜市，最后快速得到一份真正适合自己的旅行方案。

中期定位可以概括为：

> AI + Notion / 日历式旅行计划工具，但重点不是“能聊天”，而是让用户更快得到一份适合自己的、可调整的旅行计划。

长期持续坚持四个价值：

1. 个性化：方案能体现用户的兴趣、预算、节奏和明确取舍；
2. 省时间：用户不用从零查攻略，也不用写复杂提示词；
3. 可编辑：用户能看懂当前安排，并用低门槛操作表达修改意图；
4. 可执行：真实性边界清楚，路线、天气、预算和注意事项便于实际参考。

## 3. 版本总览

| 版本 | 主题 | 主要解决的问题 | 核心交付 |
|---|---|---|---|
| MVP 1.0 | AI 生成完整行程 | 从零查攻略太慢 | 输入、补充、生成、结果、复制、导出、简单重排 |
| MVP 1.1 | 点击式行程编辑 MVP（已完成） | AI 方案不好快速改；手机端太长 | Day Cabinet、积木操作、快捷修改、缺失提示、移动端 Day 导航 |
| MVP 1.2 | Mobile Flow + Cabinet Identity + Pending Changes + 轻工作台 | 手机上不好浏览；行程柜识别弱；修改意图会互相覆盖 | 首页 mobile 收口、`/plan` 问卷式、`/result` 分页式、柜子视觉、修改篮、修改工作台 |
| MVP 1.2.5 | 真实 AI 冒烟版 | Mock AI 的表达与多条修改稳定性有限 | 真实 AI 接入、主流程冒烟、多条修改重点复测 |
| MVP 1.3 | 真实性 API 与地图雏形 | 距离、路线、天气和开放时间缺少外部依据 | 高德 POI / 路线、天气、出行方式、地图基础展示、卡片与点位基础联动 |
| MVP 1.4 | 保存 / 分享 / 历史版本 | 方案无法跨会话留存或发给同行人；已有计划难以接入 | 保存、分享、历史版本、已有计划导入、补全完整方案、弱商业入口 |
| 2.0 | AI 旅行计划工作台 | 编辑、验证、保存仍然分散 | 左侧行程柜、右侧地图、修改篮、保存分享、地图选点、双向高亮、Explore |
| 2.0+ | 旅行生态延伸 | 复杂旅程与真实履约 | 平台联动、长期偏好、模板库、跨城跨国、社区与预订 |

## 4. MVP 1.0：AI 生成完整行程（已完成）

### 主题

AI 先帮用户生成一版完整行程。

### 已完成能力

- 首页自然语言输入与快捷标签；
- `/plan` 分步补充和标准化旅行信息；
- `/api/parse-trip` 与 `/api/generate-trip`；
- `WeatherProvider`、Mock Weather 与天气失败降级；
- `LLMProvider`、`MockLLMProvider` 与 OpenAI-Compatible Provider；
- 结构化 `TripPlan` 与 Zod 校验；
- `/result` 卡片式结果页；
- 复制完整方案与 Markdown 导出；
- 输入补充要求后重新生成完整 `TripPlan`；
- `localStorage` 草稿、需求和最近结果恢复；
- 移动端基础适配；
- GitHub + Netlify 部署；
- README 与 MVP 1.0 手动验收文档。

### 已验证价值

用户不用从零查攻略，可以先拿到一版结构化、可参考的旅行方案。

### 暴露的问题

1. `/plan` 最终无法提交时，用户不能快速知道缺什么、缺项在哪；
2. 手机端结果页和多日行程过长，用户需要持续下滑；
3. 简单自然语言重排可用，但用户仍要自己组织修改句，修改门槛偏高。

## 5. MVP 1.1：点击式积木编辑 MVP（已完成）

### 主题

用户能通过点几下，把 AI 生成的方案快速改成自己的。

### 主线：点击式行程编辑

- 一天是一个 `Day Cabinet`；
- 每天固定上午、下午、晚上三个 `Time Slot`；
- 每个景点、活动、美食或交通建议是一个 `Itinerary Block`；
- 每个积木提供“不想要这个 / 换一个 / 一定保留 / 加类似”操作；
- 操作只生成修改意图并填入 `RegenerateBox`，不自动提交；
- 用户确认后继续调用现有 `/api/generate-trip`；
- 服务端仍返回完整、经过 Schema 校验的新 `TripPlan`；
- 不做局部 Patch，不直接把客户端局部修改当作最终方案。

### 辅线：轻量自然语言修改

- 提供“轻松一点 / 少走路 / 预算低一点 / 加美食或夜市 / 不早起”快捷修改；
- 优化 `RegenerateBox` 的占位提示和修改句；
- 清楚展示 `appliedChanges`；
- 只服务于点击编辑后的确认与补充，不发展成完整聊天系统。

### 同期修复

- `/plan` 顶部缺失信息摘要；
- 字段级错误、高亮和点击定位；
- 提交失败滚到第一个错误；
- 手机端默认多日摘要，点某天展开；
- 顶部 Day 导航与“修改”入口前置；
- 让总览、天气、预算和当前 Day 更快可达。

### 进入下一版本前的证据

- 用户能在不学习提示词的情况下完成至少一次积木修改；
- 大多数积木操作生成的修改句无需手工重写；
- 二轮测试中，用户能快速找到缺失字段和指定某一天；
- 重新生成后，锁定、避开和新增意图基本得到体现；
- 复制与导出仍和当前完整方案一致。

## 6. MVP 1.2：Mobile Flow + Cabinet Identity + Pending Changes + 轻工作台

### 主题

让 v1.1 的点击式编辑在手机端真正好用，同时把多个修改意图先收进修改篮，再整理成一个更清楚的轻工作台雏形。

### 已完成交付

- `/plan` 手机端问卷式分页，桌面端保持 v1.1 为主；
- 中文友好的日期提示与辅助文案；
- `/result` 手机端按总览、Day、Slot、Block、Detail 分层浏览；
- 天气、预算、攻略、住宿、交通等次级模块按需展开；
- Day / Slot / Block 强化“旅行柜”视觉识别；
- 详情入口升级为明显可触控按钮；
- Pending Changes 统一收集多个积木动作与快捷修改；
- 支持查看、删除单项、清空和写入修改框；
- 最终仍合并为一个 `modificationRequest`；
- `overview / day / edit` 更清楚地组织成轻工作台节奏；
- 继续使用完整重新生成，不引入局部 Patch。

### 竞品洞察对 v1.2 的影响

v1.2 只吸收竞品里已经能直接提升当前主链路、且不需要地图或持久化支撑的部分：

- 吸收移动端一屏一任务的分层浏览；
- 吸收柜门、抽屉、积木这种更稳定的编辑识别；
- 吸收“结构化编辑为主、打字修改为辅”的编辑心智；
- 不吸收地图、商业按钮、图片视频、保存分享和 Explore。

### 启动条件

v1.1 已完成并上线测试。当前观察已明确暴露手机端层级、日期显示、详情触控和单次修改覆盖问题，因此进入 v1.2 规划与范围冻结。

### 明确不做

- 高德地图、真实 POI、真实路线与开放时间；
- 保存、分享、登录、数据库与历史版本；
- 拖拽；
- 格子高低调整；
- 隔板增删与自定义 Time Slot；
- 精确时间轴；
- 完整聊天系统；
- 局部 Patch；
- `TripPlan` Schema 或 `/api/generate-trip` 协议变更；
- constraints 持久化。

## 7. MVP 1.2.5：真实 AI 冒烟版

### 主题

在不进入地图和保存分享的前提下，把生成主流程从 Mock AI 过渡到真实 AI。

### 建议能力

- 接入真实 AI 的最小主流程；
- 保持现有 `/api/generate-trip` 协议与 `TripPlan` 结构；
- 优先验证单条、两条与三条以上 Pending Changes 的稳定性；
- 保持 Mock AI 作为可回退模式；
- 不把这一步扩成地图、POI、保存分享或工作台重构。

## 8. MVP 1.3：国内真实性 API 与地图雏形

### 主题

通过国内可用 Provider，让行程在 POI、路线、天气和开放时间上更接近真实可执行。

### 建议能力

- 接入高德 POI / 路线 API 的最小能力；
- 景点名称标准化和歧义处理；
- 景点之间的大概距离与通勤时间；
- 路线顺序合理性提示；
- 标记“这天可能太赶”；
- 将距离、通勤时间和路线顺序作为生成上下文；
- 让天气影响更可靠；
- 自驾 / 公交 / 步行等出行方式；
- 地图基础展示；
- 地点卡与地图点位的基础联动；
- 对开放时间提供弱支持与明确降级；
- 继续明确“估算”和“出行前核实”的可信边界。

### 竞品洞察对 v1.3 的影响

- 吸收 Wanderlog / Mindtrip / Roadtrippers 的地图基础价值；
- 先做“验证路线是否合理”的基础地图，而不是一步做到完整工作台；
- 自驾等出行方式优先进入这一版，因为它和路线、时间、距离是一层问题。

### 暂不做

- 真实票务；
- 酒店库存；
- 支付与下单；
- 多平台价格比价；
- 复杂全局路线优化。

## 9. MVP 1.4：保存 / 分享 / 历史版本

### 主题

让用户能把方案留住、回看、发给同行人。

### 建议能力

- 分享链接；
- 保存当前方案；
- 本地历史版本；
- 多版本基础对比；
- 更完整的 Markdown / PDF 导出；
- 同行人只读查看链接；
- 已有计划导入入口；
- 用户带着机票、酒店、想去地点进入后，系统补全完整方案；
- 商业按钮弱占位；
- 根据实际跨设备需求决定是否引入轻量登录。

### 竞品洞察对 v1.4 的影响

- 吸收 TripIt 的“已有计划整理 / 补全”入口；
- 吸收 Wanderlog 的商业入口预期，但只做弱占位，不做真实预订闭环；
- 把这一版定位成“让计划能留住、能继续改、能带着已有安排进来”，而不是提前做交易平台。

### 排期原则

本版本可以根据二轮朋友测试提前。如果用户强烈反馈“想保存”“想换设备看”“想发给同行人”，可把分享与本地历史中的最小闭环提前；但不要因此把账号、权限、协作和数据库整体前移。

## 10. 2.0：AI 旅行计划工作台

### 主题

从生成器变成可编辑、可验证、可保存、可分享的旅行工作台。

### 建议能力

- 左侧行程柜；
- 右侧地图与地点详情；
- Pending Changes 修改篮；
- 保存与分享；
- 拖拽积木；
- 调整格子高低；
- 新增和删除隔板；
- 精确时间轴；
- 时间冲突检测；
- 地图路线校验；
- 地图拖动、缩放和点位详情；
- 地图选点加入左侧工作台；
- 左侧卡片与右侧地图双向高亮；
- Explore 发现页；
- Saved / Inspiration / Notes / Trips 等工作台导航；
- 登录与云端保存；
- 分享与多人协作；
- 多版本和回滚；
- 移动端 PWA 化；
- 更接近日历、Notion 和旅行计划 App 的工作台体验。

### 竞品洞察对 2.0 的影响

- Wanderlog 提供了“左侧工作台 + 右侧地图”的总体结构启发；
- Mindtrip 提供了卡片 hover / 选中与地图高亮联动的交互方向；
- 本项目在 2.0 里不追求照抄，而是形成“柜子 + 积木 + 修改篮 + 地图”的自有工作台；
- Layla 的路线连线动效、图片和短视频可作为 2.0+ 增强项，不作为 2.0 首批门槛。

### 2.0 的前提

2.0 不是把所有愿望同时开发，而是把 1.1—1.4 已验证的心智模型、真实性数据和保存机制组合成稳定工作台。每项复杂能力都应有真实使用证据，不以“看起来像完整产品”为理由提前建设。

## 11. 2.0+ 远期方向

以下仅作为愿景，不进入当前实现计划：

- 多人实时协作规划；
- 与地图、日历、票务、酒店平台深度联动；
- 用户长期偏好记忆；
- 公开行程模板库；
- 多城市 / 跨国复杂旅行；
- App 化或更完整的 PWA；
- 社区行程分享；
- 真实预订链路。

## 12. 不应提前进入 v1.1 的功能

v1.1 明确不做：

- 拖拽；
- 格子高低调整；
- 隔板增删和自定义时间段；
- 精确时间轴；
- 地图 API 与真实距离计算；
- 实时开放时间；
- 真实票务、酒店价格、库存、预订或支付；
- 登录、收藏、数据库、跨设备保存；
- 真正的局部 Patch；
- 完整多轮聊天系统与长期对话记忆；
- 历史版本管理；
- 多人协作；
- 为未来版本提前重写项目架构。

判断规则：如果一项功能不能直接提升“发现缺项”“快速找到某天”或“点几下表达修改意图”三件事之一，它默认不进入 v1.1。

## 13. 如何避免头痛医头式开发

### 12.1 一版只解决一个主问题

每个版本设置一句主题和一条主线。v1.1 的主线是点击式积木编辑，缺失提示与移动端导航是完成这条主线所需的基础体验；聊天、地图、保存不是并列主线。

### 12.2 用稳定领域对象承接多个版本

v1.1 继续复用 `TripPlan`、`DailyItinerary`、`ItineraryItem`，只增加展示映射和轻量修改约束。只有经过用户验证、确实无法由现有结构表达的需求，才在后续版本调整核心 Schema。

### 12.3 每阶段都有“不做什么”

阶段计划必须同时写目标、修改文件、功能范围、不做项、验收标准和测试命令。没有明确“不做”的阶段，不进入开发。

### 12.4 先用用户反馈验证，再提高技术复杂度

- v1.1 验证点击修改是否降低表达门槛；
- v1.2 验证移动端分层、柜子识别与多意图收集；
- v1.3 再用国内地图 / POI / 天气 / 开放时间提高真实性；
- v1.4 根据保存分享需求决定账号与持久化范围；
- 2.0 最后组合成工作台。

### 12.5 保持单一真实来源

当前完整 `TripPlan` 始终是页面展示、复制和导出的唯一来源。点击操作和 constraints 只是生成下一版方案的意图，不与完整方案形成两个互相漂移的数据源。

### 12.6 设置版本出口

每个版本在开发前冻结范围，在朋友测试后记录：

- 哪个问题被解决；
- 哪个问题仍高频；
- 哪些能力无人使用；
- 哪项需求值得提前；
- 下一版本是否应扩展、保持或缩减范围。

这样路线图用于做取舍，而不是事后给零散功能补标签。
