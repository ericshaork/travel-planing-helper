本次审计目标：

- 盘清当前 AI 生成入口和数据流

## 3. Current AI Generation Chain

当前 AI 生成主链路是：

1. 用户在 `/create` 输入自然语言需求，附带快捷兴趣和风格标签。
2. `NaturalLanguageInput` 调 `/api/parse-trip`，把解析结果保存到本地 `parsed-trip` session。
3. 用户进入 `/plan`，基于解析结果补全 `TripRequestDraft`。
4. `/plan` 提交时通过 `normalizeTripRequestDraft()` 生成完整 `TripRequest`。
5. 页面先把 `TripRequest` 存进本地 `trip-request`，然后 `POST /api/generate-trip`，请求体只有：
   - `tripRequest`
6. `app/api/generate-trip/route.ts` 只负责校验后调用 `planTrip()`。
7. `lib/trip/planner.ts` 会：
   - 用 `tripRequest.destinationCity` 等字段取天气
   - 把 `tripRequest + weatherForecast + weatherWarning` 交给 `generateTripWithProvider()`
8. `lib/ai/generateTrip.ts` 再把这些输入传给 provider。
9. `lib/ai/prompts.ts` 当前 prompt 只注入：
   - `tripRequest`
   - `weatherForecast`
   - `weatherWarning`
10. 生成成功后，前端把 `tripPlan` 写进本地 `trip-plan`，补 `workspace-session`，然后跳 `/workspace`。

当前结论：

- AI 生成链路里还没有读取 `user_settings`
- 也没有单独的 `userDefaults` / `tripSpecificPreferences` / `effectivePreferences` 层
- 当前 AI 真正吃到的用户偏好，只有 `TripRequest` 里已经填写的那些字段

## 4. Current TripRequest And Trip-Specific Preferences

当前“单篇计划偏好”并不是独立对象，而是分散在 `TripRequest` 里。主要字段包括：

- `budget`
- `interests`
- `travelStyles`
- `mustVisitPlaces`
- `avoidPlaces`
- `accommodationPreference`
- `localTransportPreference`
- `schedulePreference`
- `specialRequirements`

这意味着当前单篇计划偏好的事实来源就是：

- `/create` 的自然语言输入和快捷标签
- `/plan` 的表单补充
- 最终归一化后的 `TripRequest`

补充结论：

- 当前没有独立的 `tripSpecificPreferences` 类型
- `trip_plans.trip_preferences_json` 虽然已经存在，但当前保存链路仍然写 `{}`，还没有参与 AI 生成和恢复
- 当前也没有“从单篇计划反推长期偏好”的逻辑

## 6. Which Defaults Should Enter AI Context

建议进入 AI 生成上下文的长期偏好：

- `travelPreferences.budget`
- `travelPreferences.pace`
- `travelPreferences.interests`
- `travelPreferences.companions`
- `travelPreferences.wakeUpPreference`
- `travelPreferences.transportPreference`
- `aiPreferences.detailLevel`
- `aiPreferences.preferHiddenGems`
- `aiPreferences.preferLessWalking`
- `aiPreferences.preferConvenientTransport`

建议不要进入 AI prompt 的偏好：

- `workspacePreferences.defaultMode`
- `workspacePreferences.mapLayout`
- `workspacePreferences.mapOverlay`

原因很明确：

- `workspacePreferences` 只影响 Workspace 的阅读/编辑体验，不影响行程内容本身
- 它们属于界面偏好，不属于路线生成偏好

启用条件建议：

- 仅当用户已登录时尝试读取长期偏好
- 仅当 `aiPreferences.useLongTermPreferences === true` 时参与合并
- 未登录、读取失败或设置关闭时，都应优雅降级为“不使用长期偏好，但继续生成”

## 7. Field Mapping And Gaps

当前长期偏好与单篇计划偏好的对应关系并不是一一完全相同。

相对好合并的字段：

- `travelPreferences.budget` -> `TripRequest.budget`
- `travelPreferences.interests` -> `TripRequest.interests`
- `travelPreferences.transportPreference` -> `TripRequest.localTransportPreference`

部分可映射但不是完全等价的字段：

- `travelPreferences.pace` -> `TripRequest.schedulePreference`
- `travelPreferences.wakeUpPreference` -> `TripRequest.schedulePreference` 或补充说明文本

当前 `TripRequest` 里没有直接承载的长期偏好：

- `companions`
- `aiPreferences.detailLevel`
- `aiPreferences.preferHiddenGems`
- `aiPreferences.preferLessWalking`
- `aiPreferences.preferConvenientTransport`

这类字段更适合在 Step 9B 作为“生成上下文摘要”注入 prompt，而不是硬塞进 `TripRequest` 结构。

## 8. Recommended Merge Rules

Step 9B 建议正式引入三层概念：

### 8.1 `userDefaults`

来源：

- 已登录用户的 `user_settings`
- 仅在 `useLongTermPreferences = true` 时有效

### 8.2 `tripSpecificPreferences`

来源：

- 当前一次创建计划时填写的 `TripRequest`
- 永远优先于长期默认偏好

### 8.3 `effectivePreferences`

来源：

- `userDefaults + tripSpecificPreferences` merge 后得到的生成参考

推荐 merge 规则：

1. 单值字段：
   - 当前计划有值就用当前计划
   - 当前计划没值才回退到长期默认

2. 数组字段：
   - 当前计划如果明确选了值，就只用当前计划
   - 当前计划为空时，才使用长期默认数组

3. 布尔型 AI 偏好：
   - 如果当前计划没有对应字段，允许直接使用长期默认 AI 偏好

4. 不参与 merge 的字段：
   - `workspacePreferences.*`

5. 严格禁止：
   - 不把单篇计划偏好回写进 `user_settings`
   - 不因为用户这次生成了某种偏好，就自动改长期设置
   - 不让 Settings 修改自动覆盖已经创建的历史计划

## 9. Prompt Injection Recommendation

Step 9B 不建议大改 `TripRequest` 和现有 prompt 主体，更稳的方式是增加一层 generation context。

建议新增两个轻量 helper：

- `lib/settings/preference-merge.ts`
- `lib/ai/generation-context.ts`

推荐生成的数据层：

- `userDefaults`
- `tripSpecificPreferences`
- `effectivePreferences`

推荐 prompt 接法：

- 保持现有 `tripRequest`、`weatherForecast`、`weatherWarning` 不变
- 额外注入一段“偏好摘要”文本，而不是把整份 `user_settings` 原样扔进 prompt

摘要方向建议：

- 用户长期偏好：预算更偏适中、节奏偏慢、少走路、优先交通方便
- 本次计划偏好：本次更看重美食和城市漫步
- 如果本次计划偏好和长期默认冲突，以本次计划为准

这样做的好处：

- 对现有 prompt 侵入最小
- 不需要暴露一堆内部字段名
- 不需要改 `TripRequest` schema
- 便于后续继续扩展而不破坏旧链路

## 10. Guest And Authenticated Boundaries

推荐边界如下：

### 10.1 游客

- 仍然可以正常 AI 生成
- 不读取 `user_settings`
- 直接按当前 `TripRequest` 生成

### 10.2 已登录用户

- 生成前可尝试读取 `user_settings`
- 如果读取失败，不阻断生成
- 如果 `useLongTermPreferences = false`，等同于不使用长期偏好

### 10.3 错误处理原则

- 设置读取失败不应让 `/api/generate-trip` 整体失败
- 不要因为长期偏好缺失而影响已有 AI 生成主链路
- 天气、生成、Workspace 跳转逻辑保持现状

## 11. trip_preferences_json Recommendation

Step 9B 对 `trip_plans.trip_preferences_json` 建议谨慎处理。

更稳的建议是：

- Step 9B 可以先不接这个字段
- 继续让 AI 生成只依赖 `TripRequest + userDefaults summary`

原因：

- 当前 `TripRequest` 已经承载了大部分单篇计划偏好
- 这一阶段的重点是“生成时合并长期偏好”，不是“重构保存结构”
- 如果现在把“合并后的有效偏好”写入 `trip_preferences_json`，容易和 `user_settings` 职责混淆

如果后续确实要写，建议只写“单篇计划偏好快照”或“生成时参考快照”，并明确区分：

- 不能把长期默认设置直接当成单篇计划偏好落库
- 不能让 `trip_preferences_json` 反向驱动 `user_settings`

## 12. Recommended Step 9B Implementation

Step 9B 推荐只做最小稳定接入：

1. 在生成链路中增加读取长期偏好的能力
2. 提取当前 `TripRequest` 对应的单篇计划偏好
3. 生成 `effectivePreferences`
4. 把偏好摘要注入现有 prompt
5. 不改现有 `/workspace` 进入链路
6. 不改 Settings UI
7. 不改 My Trips
8. 不改 `TripRequest` schema
9. 不改 `trip_plans` schema

建议改动文件范围：

- `app/api/generate-trip/route.ts`
- `lib/trip/planner.ts`
- `lib/ai/prompts.ts`
- `lib/ai/types.ts`
- `lib/settings/server.ts`
- 新增 `lib/settings/preference-merge.ts`
- 新增 `lib/ai/generation-context.ts`
- 按需补测试到 `tests/ai/*`、`tests/trip/*` 或现有 generate route tests

推荐实现顺序：

1. 新增 server helper，安全读取当前登录用户 settings
2. 从 `TripRequest` 提取 `tripSpecificPreferences`
3. merge 出 `effectivePreferences`
4. 生成中文偏好摘要
5. 让 prompt 追加这段摘要
6. 补未登录、读取失败、关闭长期偏好的测试

## 13. Recommended Tests For Step 9B

至少应覆盖：

1. 未登录生成时，不读取长期偏好，也能正常生成
2. 已登录且 `useLongTermPreferences=true` 时，会把长期偏好并入上下文
3. 已登录但读取 settings 失败时，不阻断生成
4. `useLongTermPreferences=false` 时，不应注入长期偏好摘要
5. 当前计划字段与长期偏好冲突时，以当前计划为准
6. `workspacePreferences` 不进入 AI prompt
7. 不改现有天气注入和生成返回结构

## 14. Risks

主要风险有四个：

1. 如果直接修改 `TripRequest` schema，范围会迅速变大，容易误伤 `/plan`、保存链路和测试。
2. 如果把长期偏好直接硬写进 `TripRequest`，会模糊“长期默认偏好”和“单篇计划偏好”的边界。
3. 如果把 settings 读取失败视为生成失败，会破坏当前稳定的 AI 主链路。
4. 如果现在顺手接 `trip_preferences_json`，容易把 Step 9B 变成保存结构重构，而不是最小合并接入。

## 15. Out Of Scope For Now

以下内容建议继续留在后续阶段：

- 修改 `TripRequest` 结构
- 把长期偏好写回 `trip_plans.trip_preferences_json`
- 从某次生成结果反推长期偏好
- Settings 改动自动影响历史计划
- 让 Workspace 编辑直接实时改 AI 偏好
- 重写 AI API、模型配置或 prompt 架构

## 16. Summary

当前 AI 生成链路已经稳定，但它只吃单次 `TripRequest`，还没有吃到 `user_settings`。长期偏好数据层已经就绪，真正缺的是一层轻量的 server-side merge 和 prompt 摘要注入。

因此，Step 9B 最稳的做法不是重构 schema，而是：

- 保留现有 `TripRequest` 主链路
- 增加 `userDefaults / tripSpecificPreferences / effectivePreferences`
- 用最小侵入方式把偏好摘要并入现有 prompt
- 保证未登录、读取失败、关闭长期偏好时都能继续正常生成
