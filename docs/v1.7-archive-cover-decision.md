# v1.7 Archive Cover Decision

## Decision

Do not add a dedicated `archive/cover/` runtime image contract in the current phase.

## Why

The current `ArchiveCover` component already has enough structure to assemble the archive opening surface from existing assets:

- `archive/template/archive-template-main.png`
- `archive/paper/archive-paper-light.png`
- `archive/frame/archive-photo-frame-01.png`
- Explore city or archive cover candidates

That means the current viewer does not need a separate:

- `archive/cover/archive-cover-default.png`

to become visually functional.

## Component Impact

Primary component:

- `components/explore/archive/ArchiveCover.tsx`

Current behavior:

1. left large panel uses archive template
2. right upper panel uses city-aware or archive fallback cover candidates
3. right lower panel uses paper / frame support imagery

This already gives the component:

- a background identity
- a content image slot
- a material layer

without introducing another asset category.

## Why This Is Better

- avoids duplicating responsibility between `archive/template/*` and `archive/cover/*`
- keeps the archive surface more composable
- reduces one more asset branch the resolver would need to maintain
- matches the current implementation direction, where Archive is assembled rather than painted from one monolithic cover file

## When To Reopen This Decision

Only reopen `archive/cover/*` if a later UI phase introduces one of these needs:

1. a single branded archive hero image must always appear regardless of city content
2. the archive opening panel becomes a fixed visual composition independent of city/archive list covers
3. the design system requires a distinct “archive jacket” layer separate from template and frame

## Current Result

For v1.7 Phase 3.1.2:

- `archive/cover/*` is not required
- it should not remain in the blocking missing list
