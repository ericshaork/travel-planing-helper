# PRD v1.3：真实地点与路线可信度增强

## 1. 文档定位

本文是 `v1.3` 的产品需求文档，只定义新阶段的目标、范围和页面形态。

- `v1.0`、`v1.1`、`v1.2`、`v1.2.5` 已完成能力继续有效；
- `v1.3` 阶段 0 只做文档与范围冻结；
- 本文不授权写业务代码、不授权接真实 API、不授权改 schema。

## 2. 一句话定位

让 AI 行程从“像攻略”进一步走向“有真实地点、有路线依据、有地图辅助的自由行草稿”。

## 3. 为什么现在做

`v1.2.5` 已经把真实 AI 主链路跑通，但当前结果页仍有几个明显短板：

1. 地点可能像真的，但还没被真实校验。
2. 路线顺序看起来合理，但没有距离和通勤时间依据。
3. 单日是否太赶，缺少明确提醒。
4. 用户看不到地点大概分布在哪里。
5. 后续桌面和移动端工作台需要先有真实性基础层。

所以 `v1.3` 的核心不是重做 UI，而是补真实性底座。

## 4. 产品目标

### 4.1 P0 目标

- 地点真实：AI 方案中的核心地点可以被 POI 能力校验。
- 路线可信：相邻地点之间有距离、时长或路线估算。
- 天气有依据：天气提醒尽量基于真实 Provider，而不是模型自由发挥。
- 节奏有提醒：能提示某一天可能过满、过赶或通勤过长。
- 地图有基础展示：用户能看到当前 Day 的点位和路线摘要。

### 4.2 不是本版目标

- 不是完整桌面工作台；
- 不是地图重交互产品；
- 不是商业预订或交易平台；
- 不是拖拽式编辑器；
- 不是重新设计 `TripPlan` 的大版本。

## 5. 用户价值

`v1.3` 要给用户带来的直接价值：

1. 减少 AI 编地点。
2. 减少路线不顺和明显绕路。
3. 提醒一天是不是太赶。
4. 能看到地点大概在哪，不只是一串文字。
5. 为后续桌面与移动端工作台打下可信数据基础。

## 6. 核心功能范围

### 6.1 POI 校验

- 对行程中的关键地点做真实地点解析与校验；
- 返回地点名称、地址、坐标、类型和可信度；
- 查不到或歧义高时，给出 warning，而不是假装确认成功。

### 6.2 Route Estimate

- 基于相邻地点估算路线距离与通勤时间；
- 优先支持最基础的 driving / walking；
- transit 可以后置或先做轻量化；
- 结果首先用于展示和提醒，不承诺精确到分钟级导航。

### 6.3 Pace Warning

- 根据单日地点数、总通勤时长、长距离 leg 等信息做节奏判断；
- 输出“这天可能略赶”“通勤偏多”“建议少排一点”等提示；
- 重点是提醒，不是自动重排。

### 6.4 Weather Enhancement

- 优先增强现有天气 Provider 的真实性；
- 若天气失败，不阻断生成；
- 结果中必须明确区分“有天气依据”和“暂时无法获取实时天气”。

### 6.5 Map Preview

- 展示当前 Day 的点位和路线摘要；
- 重点是帮助用户判断顺不顺、远不远；
- 不做复杂地图交互。

### 6.6 Desktop Light Split Layout

- 保留现有 `/result` 主体；
- 在桌面端增加轻量左右分区雏形；
- 左侧继续是现有 Day Cabinet / 行程柜；
- 右侧放地图基础区、当前 Day 地点列表、距离/时间摘要；
- 这是雏形，不是最终工作台。

### 6.7 Mobile Map Page

- mobile 不做左右分栏；
- 保持 `overview / day-N / budget / more / edit`；
- 新增 `map` 或 `route` 页面；
- Day 页展示地点真实校验和到下一站的距离信息；
- map/route 页展示当前 Day 点位和路线摘要。

## 7. 页面形态

## 7.1 Desktop

桌面端结果页仍然以现有阅读体验为主，不重做成竞品式工作台。

页面形态：

- 左：现有行程柜、Day Cabinet、结果内容；
- 右：地图 / 地点列表 / 路线摘要；
- 保持结果页可读性、可复制性、可导出性优先；
- 地图区主要承担真实性验证辅助，不承担编辑主入口。

## 7.2 Mobile

移动端继续保持分页式体验，不做左右分。

页面形态：

- 继续保留 `overview / day-N / budget / more / edit`；
- 新增 `map` 或 `route` 页面；
- Day 页内可看到每个地点的校验状态和到下一站的大概距离；
- 独立地图页只展示当前 Day 点位、顺序和路线摘要。

## 8. 成功标准

`v1.3` 完成后，应满足：

1. 关键地点具备真实校验结果或明确 warning。
2. 当前 Day 具备基础路线距离 / 时间估算。
3. 单日节奏异常时有明确提醒。
4. 天气信息不再只依赖模型自由文本。
5. 桌面端能看到轻量右侧真实性辅助区。
6. 移动端能看到独立 map/route 页面。
7. 复制与 Markdown 导出仍以当前完整方案为唯一来源。
8. 没有越界做保存、拖拽、完整工作台或商业闭环。

## 9. 明确不做

`v1.3` 明确不做：

- 不拖拽；
- 不保存分享；
- 不登录；
- 不上数据库；
- 不做完整地图工作台；
- 不大改整体 UI；
- 不做地图选点加入行程；
- 不做复杂地图卡片双向联动；
- 不做商业闭环；
- 不做真实票务、酒店、支付或下单；
- 不做完整时间轴；
- 不做完整桌面工作台；
- 不做移动端工作台大改；
- 不改 `TripPlan` schema，除非单独发起最小扩展确认。

## 10. 与后续版本关系

- `v1.3`：先补真实性基础层。
- `v1.4`：再做 Desktop Workspace UI。
- `v1.5`：再做 Mobile Workspace UI。
- `v2.0`：再整合成完整 AI 旅行工作台。

本版的目标是打基础，不是一次性把工作台做完。
