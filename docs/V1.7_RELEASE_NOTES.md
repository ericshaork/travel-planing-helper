# Wanderly v1.7 Release Notes

## 1. 文档用途

- 本文记录 `v1.7` 在当前仓库中的实际完成状态。
- 原始规划仍保留在 `docs/PRD_v1.7.md`、`docs/V1.7_PHASE_PLAN.md` 和 `docs/UI_AUDIT_v1.7.md` 中。
- 如果原计划与当前代码状态冲突，应以当前代码和本文记录的最终状态为准。
- 本文不代表 `v1.8` 的 PRD 或阶段计划，只用于 `v1.7` 历史收档。

## 2. v1.7 版本定位

`v1.7` 最初定位为桌面端 UI/UX 大修，但实施过程中逐步发现，现有技术、数据和页面结构不足以直接支撑目标视觉与交互，因此实际开发同时补入了必要的技术基础、数据结构、页面入口和运行时资产。

基于当前仓库状态，`v1.7` 更准确的描述应为：

> UI/UX 升级 + 必要技术补全形成的可运行基线。

这不是正式上线成品，也不是最终展示质量版本；它更接近后续 `v1.8` 继续精修所依赖的稳定基础版。

## 4. v1.7 实际完成内容

以下内容根据当前代码、路由、接口、迁移文件、测试目录与现有文档整理。

### 4.1 产品与页面

- 首页 `/` 已不是纯占位页，存在第一版品牌化 Landing：`app/page.tsx` 接入 `components/landing/LandingPage.tsx`，并实际使用 hero 主视觉图与导航入口。
- 创建链路 `/create -> /plan -> /result` 仍是当前核心主链路，页面均已存在并可联通。
- `/result` 与 `/workspace` 已共用 `WorkspaceRoutePage` 和 `TripWorkspace`，说明工作台基础已经统一为同一套结果/编辑承载结构。
- Explore 第一版页面已存在：`/explore` 首页、`/explore/[id]` 档案详情页、Explore 入口到 Create 的流转逻辑均已落地。
- `/login`、`/trips` 已是可用页面，不再只是文档层面的待办。
- `/favorites` 页面也已存在，但更接近 Explore 辅助页，不属于已收口完成的核心主链路页面。
- 当前路由状态可确认包括：`/`、`/create`、`/plan`、`/result`、`/workspace`、`/explore`、`/explore/[id]`、`/login`、`/trips`、`/favorites`。

### 4.2 核心功能

- 已存在 AI 旅行需求解析接口：`app/api/parse-trip/route.ts`。
- 已存在 AI 行程生成接口：`app/api/generate-trip/route.ts`，并由 `lib/trip/planner.ts` 串联 LLM 与天气能力。
- 已存在重新生成相关前端能力，如 `components/trip/RegenerateBox.tsx`，说明“在当前方案基础上重生”已进入实际工作台。
- 已存在天气能力，且保留“失败不阻断生成”的兜底逻辑：`lib/weather/client.ts`。
- 已存在 POI Provider 与 Route Provider 抽象，并支持 mock / AMap 双实现：`lib/poi/*`、`lib/route/*`。
- 已接入高德前端地图加载器与地图组件：`lib/map/amap-loader.ts`、`components/map/AmapBaseMap.tsx`。
- Day 与地图的基础联动已存在，相关工作台与地图/点位测试也已存在于 `tests/workspace`、`tests/map`、`tests/trip`。
- 本地草稿、本地工作台状态与恢复逻辑已存在：`lib/trip/storage.ts`。

### 4.3 数据与接口

- Trip 相关服务端 CRUD 已存在：

## 5. v1.7 未完全完成的内容

以下内容按“已有基础但完成度不足”与“尚未实现/尚未形成闭环”区分记录。

### 5.1 Landing

- 已有第一版品牌化首页和 hero 主视觉，但仍未达到正式展示级收口。
- Hero 图片比例、标题文案、信息层级、动效、装饰与下游页面衔接仍有继续精修空间。
- 当前首页结构已经脱离纯占位，但仍明显处于第一版可用基础，而非最终版。

### 5.2 Explore

- Explore 首页、详情页、收藏与档案转换流程已经存在，但“完整产品闭环”仍未完成。
- 直接搜索体验目前更像基础筛选/定位，不是成熟的搜索产品。
- 灵感探索多选、二级卡牌、筛选结果回流搜索与 AI 创建的关系还在第一版结构阶段。
- AI 创建入口虽然已能从 Explore 带入草稿，但入口文案、流程表达和用户心智仍需产品化。
- 平台旅行档案的规模化正式内容接入仍不足，当前同时依赖数据库结构与 mock/静态内容。
- 档案详情阅读器已经存在，但是否达到既定的“右侧 65% 阅读器”完成度仍不能视为正式完成。
- “用此行程创建”已有转换逻辑基础，但尚不能认定为完全成熟闭环。

### 5.3 Workspace

- `result/workspace` 双栏基础已经存在，且当前桌面端默认比例已写死为 `55% / 45%`，但仍是基础版而非成熟工作台。
- 当前代码体现为固定双栏布局，尚未实现规划文档中提到的用户可拖拽分隔线。
- Read Mode 与 Edit Mode 都已有结构，但整体还未形成成熟、稳定、统一的产品体验。
- Day 与地图联动已有基础实现与测试，但仍需要继续验收与细节修正。
- 手帐积木编辑、拖拽编辑和完整 AI 修改闭环尚未成熟；至少拖拽能力当前不能视为已完成。
- 地图搜索、地点详情扩展、两点距离/通勤增强等能力仍属于后续方向。

### 5.4 Settings 与长期记忆

- 当前路由中没有 `settings` 页面。
- 当前“偏好”仍主要停留在单次旅行请求、工作台草稿和 Explore 草稿层，不等于长期用户记忆。
- 长期用户偏好、跨行程记忆和正式设置中心尚未形成。

### 5.5 登录系统

- 当前已采用 Supabase Magic Link 邮箱登录，短期可用。
- 登录成功提示、回流路径和整体叙事仍有产品化精修空间。
- 更完整的登录方式体系不是当前代码中的完成项。

### 5.6 移动端

- 移动端并非空白，当前工作台包含移动端分页浏览结构。
- 但从代码与规划边界看，`v1.7` 仍以桌面端为主，移动端没有完成正式的深度设计与全面收口。
- 后续版本仍需要区分桌面端精修与移动端正式推进范围。

## 6. v1.7 遗留原则

- `v1.7` 已形成后续继续开发可以依赖的真实技术基线，而不是只停留在 UI 设想。
- 后续不应把当前 AI、天气、POI、路线、地图、登录、保存、Explore 数据结构整体推翻重做。
- 当前视觉、图片和品牌资源虽已大量进入仓库，但多数仍处于第一版接入与整理状态，不应误判为最终视觉定稿。
- 后续页面精修应继续同时考虑技术补全、视觉收口、图片资产替换与跨页面状态流转，而不是只做表层样式改动。

## 7. 转入 v1.8 的工作

这里只做收档概括，不展开新的 `v1.8` 规划。

后续继续推进时，建议沿以下顺序进入：

1. Landing 技术与视觉收口；
2. Explore 三类入口、档案平台与阅读器深化；
3. Workspace 的 Read/Edit 模式精修；
4. Settings 与长期偏好体系补全；
5. 全站桌面端收口与上线前验收。

需要特别强调：

- 各页面的推进不是彼此孤立的。
- 对任一页面继续修改前，都应同时考虑导航、状态传递、Create/Explore/Workspace 之间的入口关系。
- 后续每页都仍会同时涉及技术补全、UI 精修和视觉资产处理三个维度。

## 9. v1.7 最终结论

`v1.7` 不是纯 UI 版本，也不是最终上线成品。

更准确地说，它已经把以下几类能力结合到了同一条可运行基线上：

- 主链路旅行生成；
- 天气、POI、路线、地图基础能力；
- Supabase 登录与保存；
- Explore 第一版档案与入口；
- Landing / Workspace / Explore 的第一版视觉资产。

因此，`v1.7` 应被视为后续 `v1.8` 精修使用的稳定功能与页面基线。后续工作重点不是重新定义它是否“做过这些能力”，而是逐页把这些已存在的能力收口到更成熟的产品完成度。
