# Wanderly v1.8 Explore Archive Normalization Plan

## 1. 审计范围

本步骤只做 Explore 档案源数据标准化方案设计、dry-run 校验和文档整理，不修改 UI、不改图片、不导入数据库、不覆盖原始 JSON、不 commit、不 push。

源数据目录：

- `data/explore/source/explore_plans_production/`

目标：

- 为 200 篇 AI 生成档案建立稳定标准字段

## 5. theme 推导规则

原始 JSON 没有 `theme`，第一版建议按以下优先级推导：

1. 先看 tags 中是否命中强主题
2. 再看 title / summary 的关键词
3. 再结合 `trip_type`
4. 最后回退到城市主气质

建议第一版 theme bucket：

- `citywalk`
- `food`
- `heritage`
- `seaside`
- `jiangnan`
- `night-view`
- `nature`
- `family`
- `romance`
- `student-budget`
- `solo-walk`
- `other`

示例规则：

- tags 含 `citywalk / canal-walk / old-street` -> `citywalk`
- tags 含 `food / street-food / local-food / cantonese-food` -> `food`
- tags 含 `heritage / imperial-history / ancient-capital / silk-road` -> `heritage`
- tags 含 `coastal-city / beach / bay-view / sunset` -> `seaside`
- tags 含 `jiangnan / canal-walk / lake-view` -> `jiangnan`
- tags 含 `night-view / nightlife / river-evening / evening-scene` -> `night-view`
- tags 含 `nature / karst-scenery / mountain-view / lake-view` -> `nature`

## 6. recommendedFor 推导规则

原始 JSON 没有 `recommendedFor`，建议第一版按 `trip_type` 主导，tags 辅助：

| `trip_type` | 推荐输出 |
| --- | --- |
| `couple` | `["情侣"]` |
| `family` | `["亲子", "家庭"]` |
| `solo` | `["独旅"]` |
| `student` | `["学生", "朋友"]` |

再按 tags 补充：

- 含 `family-friendly / family / 亲子 / 家庭友好` -> 增加 `家庭`
- 含 `romantic / romance / couple / 情侣` -> 增加 `情侣`
- 含 `solo / solo-travel` -> 增加 `独旅`
- 含 `student / budget-friendly / youthful-energy` -> 增加 `学生`

最后去重。

## 7. externalId 生成规则

建议规则：

- 前缀固定：`explore_`
- 主体使用原始文件名 stem
- 格式：`explore_${sourceFileStem}`

示例：

- `beijing_couple_2d_001.json` -> `explore_beijing_couple_2d_001`
- `hangzhou_student_3d_062.json` -> `explore_hangzhou_student_3d_062`

为什么选文件名 stem：

- 比直接基于 slug 更稳定
- 对重复 slug 天然安全
- 可追溯到原始文件
- 不需要随机数

dry-run 结果：

- `externalId` 全部唯一
- 冲突数量：`0`

如果未来引入更多批次，建议扩展为：

- `explore_${batch}_${sourceFileStem}`

但本批 `v1` 暂时不需要。

## 8. slug 去重规则

建议规则：

1. 保留原始 `slug` 到 `sourceSlug`
2. 若 `sourceSlug` 全局唯一，则 `slug = sourceSlug`
3. 若 `sourceSlug` 重复，则在末尾追加源文件尾号

格式：

- 唯一 slug：`sourceSlug`
- 重复 slug：`${sourceSlug}-${sourceFileSeq}`

示例：

- `beijing-student-3d-food-citywalk`
  - `beijing_student_3d_002.json` -> `beijing-student-3d-food-citywalk-002`
  - `beijing_student_3d_005.json` -> `beijing-student-3d-food-citywalk-005`

优点：

- 可读
- 稳定
- 不引入随机串
- 与源文件可追溯

## 10. 是否会影响已有路由

会有兼容问题，但可控。

原因：

- 当前前端与 `/explore/[id]` 兼容读取里，很多地方默认把 `slug` 当成主 key
- 如果后续正式数据导入使用新的 `canonicalSlug`，旧 `sourceSlug` 的重复项会分叉

建议兼容策略：

1. normalized 数据正式落库后，以 `canonicalSlug` 作为主 slug
2. 同时保留 `sourceSlug`
3. 在数据读取层支持：
   - 先按 `canonicalSlug` 查
   - 查不到再按 `sourceSlug` 查
4. 对重复 `sourceSlug` 的旧链接：
   - 可以返回第一条
   - 或后续补一个 alias / redirect 规则

## 12. 校验规则

后续正式标准化时，建议至少校验：

1. `externalId` 必须唯一
2. `slug` 必须唯一
3. `sourceSlug` 必须保留
4. `title` 必须存在
5. `cityCode` 必须存在
6. `city` 必须存在
7. `days` 必须是正数
8. `tags` 必须是数组
9. `dailyItinerary` 必须是数组
10. `food` 必须是数组
11. `places` 必须是数组
12. `coverImageUrl` 可以为空，但字段必须存在
13. `galleryImages` 可以为空数组，但字段必须存在
14. `source.type` / `source.sourceFile` / `source.sourceVersion` 必须存在
15. `tripType` 只能落在允许集合内
16. `theme` 必须落在受控主题集合内
17. 不允许 v1.8 正式 normalized 数据混入国外城市

建议城市 allowlist 沿用当前 v1.8 Explore 逻辑。

## 13. 后续落地建议

建议顺序如下，本步骤不执行：

### 13.1 保留 source 原始目录只读

保留：

- `data/explore/source/explore_plans_production/`

原则：

- 原始 JSON 不覆盖
- 原始目录只读
- 后续任何治理都在 normalized 层完成

### 13.2 新增 normalized 输出目录

建议新增：

- `data/explore/normalized/`

建议产物：

- `data/explore/normalized/explore_archives_v1.json`
- `data/explore/normalized/explore_archive_image_manifest.json`

### 13.3 前端与导入脚本只读 normalized

建议后续：

- 前端 mock / dry-run 逻辑逐步切到 normalized 结构
- import 脚本改为读取 normalized
- 补图 manifest 使用 normalized 的 `externalId / slug`

### 13.4 是否需要数据库 migration

短期不一定需要 schema migration。

原因：

- 当前 DB 模型本身已经能容纳多数目标字段
- 更现实的做法是先做 normalized 输出
- 再由导入层把标准化数据映射进现有 DB 字段

如果后续要正式引入：

- gallery
- image manifest
- alias slug

那时再考虑 migration 更合适。

## 14. 风险与待确认问题

1. 当前 `city.name` 是英文城市名，是否要在 normalized 阶段补中文名，需要单独规则。
2. `theme` 当前完全依赖推导，不是源数据原生字段，后续最好固化受控枚举。
3. `highlights` 和 `featuredReason` 如果要做得更像 editorial，可能还需要后续二次加工。
4. 7 组重复 slug 虽然能修掉，但旧链接兼容策略要提前定。
5. `coverImageUrl`、`galleryImages` 为空是源数据层结构缺口，不是前端字段判断能解决的问题。

## 15. 结论

这 200 篇数据已经足够进入“标准化输出”阶段。

最合理的下一步不是继续直接读 source，也不是立刻导库，而是：

1. 保留 source 只读
2. 生成 normalized 输出
3. 后续前端、补图 manifest、导入脚本统一只读 normalized

这样可以把：

- `externalId`
- `canonicalSlug`
- `theme`
- `recommendedFor`
- `highlights`
- `coverImageUrl`
- `galleryImages`

这些目前不稳定或缺失的字段，集中治理到一层，而不是散落在前端组件里临时判断。
