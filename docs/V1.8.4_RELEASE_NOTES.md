# V1.8.4 Release Notes

## 1. 版本目标

v1.8.4 的目标是补齐 Wanderly 的用户系统底座，但不重做原有 AI 生成和 Workspace 主链路。核心关注点是：

- 统一注册 / 登录 / 退出体验
- 补上用户长期默认偏好设置
- 让 My Trips 成为可用的行程管理页
- 让 Workspace 具备稳定的登录保存链路
- 在不破坏现有生成流程的前提下，把长期默认偏好接入 AI 生成上下文

## 2. 完成内容

本版本已完成：

- 登录页改为中文“注册 / 登录”包装，保留邮箱链接登录能力，但隐藏技术实现细节
- Header、UserMenu、退出登录后的游客态 UI 已统一
- `/settings` 已增加登录保护，未登录访问会跳转 `/login?returnTo=/settings`
- Settings 页面已接入真实 `user_settings` 读取与保存
- 新增 `user_settings` 数据表、RLS、默认设置和用户设置 API
- `trip_plans` 已补齐基础 metadata 字段：`source_type`、`status`、`trip_preferences_json`、`local_draft_id`、`last_opened_at`
- Trips API 已支持搜索、状态筛选、来源筛选和 metadata patch
- My Trips 页面已支持搜索、筛选、中文标签、重命名、删除确认、最近打开时间展示
- 打开历史计划时会尽力更新 `last_opened_at`，但不会阻断进入 `/workspace`
- Workspace 保存链路已补齐稳定 `localDraftId`、`savedTripId` 和基础 metadata 写入
- 游客点击保存会跳登录，且不会清空本地草稿
- 已登录用户再次保存同一份 Workspace 时会更新已有 trip，而不是重复新建
- AI 生成链路已支持在登录状态下按需读取长期默认偏好，并以“当前计划偏好优先”的规则合并进 prompt 上下文

## 3. 数据库变更

### 3.1 `user_settings`

新增 migration：

- `supabase/migrations/005_user_settings.sql`

新增表：

- `id`
- `user_id`
- `travel_preferences_json`
- `workspace_preferences_json`
- `ai_preferences_json`
- `created_at`
- `updated_at`

关键规则：

- `user_id` 唯一，一个用户只有一份设置
- 开启 RLS
- `select / insert / update / delete` 都是 own-only
- 复用通用 `updated_at` trigger
- 新用户注册时会自动插入默认 `user_settings`

### 3.2 `trip_plans` metadata

新增 migration：

- `supabase/migrations/006_trip_plans_metadata.sql`

新增字段：

- `source_type text not null default 'ai_generated'`
- `status text not null default 'saved'`
- `trip_preferences_json jsonb not null default '{}'::jsonb`
- `local_draft_id text null`
- `last_opened_at timestamptz null`

约束与索引：

- `source_type` check：`ai_generated / blank_manual / explore_import`
- `status` check：`draft / saved / archived`
- 列表查询索引：`user_id + status`
- 最近打开排序索引：`user_id + last_opened_at desc`

## 4. API 变更

### 4.1 User Settings API

- `GET /api/user-settings`
  - 未登录返回 `401`
  - 已登录返回当前用户 settings
  - 如果缺失则按默认值创建或补齐后返回

- `PATCH /api/user-settings`
- `PUT /api/user-settings`
  - 未登录返回 `401`
  - 支持更新：
    - `travelPreferences`
    - `workspacePreferences`
    - `aiPreferences`
  - 使用轻量校验和 merge 更新

### 4.2 Trips API

- `GET /api/trips`
  - 支持 `search`
  - 支持 `status`
  - 支持 `source_type`
  - 返回 trip metadata，供 My Trips 页面展示和管理

- `PATCH /api/trips/[tripId]`
  - 保留整条 trip 更新能力
  - 新增 metadata 级部分更新能力，支持：
    - `title`
    - `status`
    - `source_type`
    - `trip_preferences_json`
    - `local_draft_id`
    - `last_opened_at`

### 4.3 AI Generate API

- `POST /api/generate-trip`
  - 仍保持原有生成入口与返回结构
  - 登录用户在带 token 请求时，会尝试读取 `user_settings`
  - `useLongTermPreferences = false` 时不会注入长期偏好
  - settings 读取失败不会阻断生成

## 5. 用户可见行为变化

### 5.1 游客

- 仍可查看 Landing 和 Explore
- 仍可完成 AI 生成并进入 `/workspace`
- 仍可从空白或 Explore 进入 `/workspace`
- 仍可本地编辑草稿
- 点击“保存”会跳转 `/login?returnTo=/workspace`
- 登录流程不会主动清空本地草稿

### 5.2 已登录用户

- 可以进入 `/settings` 管理长期默认偏好
- 可以进入 `/trips` 管理自己的行程
- 可以真实保存 Workspace 到 `trip_plans`
- 可以再次保存同一份 Workspace 并覆盖已有 trip
- 生成新计划时，可按设置决定是否参考长期默认偏好

## 6. 长期默认偏好与单篇计划偏好的边界

本版本已明确这两个概念分开：

- Settings 里的偏好是“长期默认偏好”
- 每次创建计划时填写的偏好是“当前计划偏好”

当前规则：

- 长期默认偏好只作为以后新建计划的参考
- 不会自动覆盖某一篇已经创建的历史计划
- AI 生成时，如果长期默认偏好与当前计划偏好冲突，以当前计划偏好为准
- `workspacePreferences` 只影响 Workspace 体验，不进入 AI prompt

## 7. 关键链路验收结论

基于代码检查、路由逻辑检查和测试结果，当前结论是：

- 注册 / 登录 / 退出链路已规范化，`returnTo` 机制可用
- `/settings` 登录保护、读取与保存链路已接通
- `/trips` 登录保护、搜索、筛选、重命名、删除确认和打开历史计划链路已接通
- Workspace 本地草稿、登录后保存、再次保存更新已有 trip 的链路已接通
- `/result` 兼容入口仍保留
- AI 生成已可按登录状态选用长期默认偏好，且不会影响游客生成
