# PRD v1.2.5：真实 AI 冒烟版 + mobile 表单补丁版

## 1. 文档定位

本文是 MVP v1.2.5 的增量产品需求，建立在已经完成的 MVP v1.0、v1.1 与 v1.2 之上。

- MVP v1.0 的完整生成闭环继续以 `docs/PRD.md` 为准；
- v1.1 的点击式行程编辑继续以 `docs/PRD_v1.1.md` 为准；
- v1.2 的 Mobile Flow、Cabinet Identity、Pending Changes 与轻工作台继续以 `docs/PRD_v1.2.md` 为准；
- 本文只定义 mobile 选项显示补丁和 Qwen 真实 AI 冒烟接入；
- 本文不覆盖旧文档，也不授权提前实现 v1.3、v1.4 或 2.0。

当前为阶段 0：规划文档与范围冻结。本阶段只新增和同步文档，不修改业务代码。

## 2. 一句话定位

v1.2.5 是一个小而明确的过渡版本：

> 修好手机端“选项要横着拖”的表单问题，并用 Qwen 跑通现有完整生成与重排链路。

它不是新的大功能版本。它只处理 v1.2 之后已经得到明确反馈的一个 mobile 表单问题，以及进入 v1.3 前必须先验证的真实模型链路。

## 3. 为什么现在做

### 3.1 mobile 选项仍不适合完整比较

v1.2 已把 `/plan` 改成三步问卷，但兴趣、旅行风格和同类固定选项仍采用横向滚动 chips。用户需要左右拖动才能看完一组选择，容易出现：

- 后面的选项不可见，不知道还有多少；
- 看完后面忘了前面，难以横向比较；
- 已选项散在滚动区域里，不容易确认；
- 问卷虽然分步了，但单步内仍有“横向抽屉”感。

这不是选项内容或字段 Schema 的问题，而是小屏展示方式的问题。

### 3.2 Mock AI 无法继续代表真实生成质量

v1.2 的 Pending Changes 已能连续收集多个积木修改，但 Mock AI 在 3 条以上修改同时重排时可能不稳定。现阶段需要用真实模型确认：

- 首次生成是否能稳定返回完整 `TripPlan`；
- 重新生成是否能保留原约束并应用新增要求；
- 3—5 条 Pending Changes 是否能被同时理解；
- 非法 JSON 是否能被现有一次修复流程处理；
- 模型自由文本中的未经验证天气说法是否仍会被服务端清洗。

## 4. 版本目标

### 4.1 P0：mobile 选项网格

- `/plan` mobile 下的兴趣和旅行风格改为网格展示；
- 同一表单中的住宿、当地交通、作息等固定选项组同步检查，存在横向滚动时一并改为网格；
- 一组选择尽量在当前任务卡内完整可见，不再依赖横向拖动发现选项；
- 选中、未选中、禁用、错误和焦点状态清楚；
- 文字可换行，不通过缩小字号或裁切文字硬塞进一屏；
- 360px、390px、430px 无页面横向溢出；
- desktop 保持现有换行式布局和视觉节奏；
- 不破坏 `/plan` 三步问卷、缺失字段定位、首错滚动和日期 / 天数自动补全。

“尽量一屏”是信息密度目标，不是裁切目标。选项较多或系统字体放大时允许正常纵向增长，但不得退回横向滚动。

### 4.2 P0：Qwen 真实 AI 冒烟

- 复用现有 `OpenAICompatibleProvider`；
- 通过 `USE_MOCK_AI=false` 和既有 `LLM_*` 环境变量启用；
- 支持把 `qwen-turbo`、`qwen-plus` 等兼容模型名填入 `LLM_MODEL`；
- 不引入 Qwen / DashScope 平台 SDK；
- 不在代码中硬编码平台、Base URL、Key 或模型名；
- 保持 Mock AI 可通过环境变量手动切回；
- 跑通自然语言解析、首次生成和重新生成；
- 重点验证 3—5 条 Pending Changes；
- 保持结构化 JSON 提取、Zod Schema 校验和一次 `repairJson`；
- 保持天气可信边界与服务端清洗。

这里的“回退”指部署或本地通过 `USE_MOCK_AI=true` 明确切回 Mock，不代表真实请求失败后在同一次请求里静默改用 Mock。静默回退会让用户误以为真实模型已经成功，不进入 v1.2.5。

## 5. mobile 选项问题定义

### 5.1 本版本处理的选项组

主要范围：

- 兴趣 `InterestSelector`；
- 旅行风格 `TravelStyleSelector`；
- `/plan` 中住宿偏好、当地交通偏好、作息偏好等同类固定按钮组。

这些选项在 mobile 下应使用可换行的自适应网格。建议优先两列，短文案且空间充足时可使用三列；具体列数由实现阶段结合真实文案和 360px 验收决定。

### 5.2 不属于本补丁的横向区域

以下不因本版本自动改成网格：

- `MissingFieldsSummary` 的缺失字段定位入口；
- `/result` 的 Day 导航；
- 其他明确承担导航而非多选比较的横向控件。

如果回归发现这些区域造成页面级横向溢出，可以做最小缺陷修复，但不借机重构导航信息架构。

### 5.3 交互要求

- 每个选项仍是原生 `button`；
- 多选继续使用 `aria-pressed`；
- 字段错误继续通过现有 `aria-invalid`、`aria-describedby` 关联；
- 键盘焦点可见；
- 触控目标高度以 44px 为目标；
- 长文案允许换行并保持按钮内对齐；
- 已选项不因换行或列宽变化产生明显跳动；
- 不改变选项值、字段名、校验规则或提交数据。

## 6. Qwen 接入边界

### 6.1 现有链路

```text
浏览器
→ /api/parse-trip 或 /api/generate-trip
→ LLMProvider
→ OpenAICompatibleProvider
→ {LLM_BASE_URL}/chat/completions
→ JSON 提取
→ Zod 校验
→ 必要时 repairJson 一次
→ 天气可信内容清洗
→ 现有 API 响应
```

Qwen 只作为现有 OpenAI-Compatible 边界后的一个可配置模型服务，不进入页面和业务 Schema。

### 6.2 环境变量

```env
USE_MOCK_AI=false
LLM_BASE_URL=
LLM_API_KEY=
LLM_MODEL=qwen-plus
```

规则：

- `LLM_BASE_URL` 填兼容接口的版本根路径，Provider 继续统一追加 `/chat/completions`；
- `LLM_API_KEY` 只存在于服务端环境；
- `LLM_MODEL` 是部署配置，不成为前端选项；
- `.env.example` 不写真实 Key；
- Mock 默认安全模式继续保留。

### 6.3 必须保持不变

- `POST /api/generate-trip` 请求与响应协议；
- `TripPlan`、`GenerateTripRequest`、`GenerateTripResponse` Schema；
- 首次生成和重新生成的完整方案模式；
- 当前完整 `TripPlan` 作为展示、复制和 Markdown 导出的唯一真实来源；
- WeatherProvider 的选择与失败降级；
- Prompt 集中在 `lib/ai/prompts.ts`；
- API Key 只在服务端使用；
- 对实时天气、价格、库存、开放状态等不确定信息的保守边界。

## 7. 核心验收场景

### 7.1 mobile 表单

1. 在 360px 打开 `/plan` 第 2 步；
2. 无需横向拖动即可看见完整兴趣组选项；
3. 选择多个兴趣后，选中态仍一眼可辨；
4. 进入第 3 步，无需横向拖动即可比较旅行风格和其他固定偏好；
5. 点击缺失摘要仍能切步并聚焦正确字段；
6. 日期与天数组合仍按现有规则自动补全；
7. desktop 仍保持稳定布局。

### 7.2 首次真实生成

使用：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

验收：

- 解析结果符合用户明确输入；
- 生成响应通过现有 Schema；
- `dailyItinerary` 天数正确；
- 必去 / 避开 / 预算 / 节奏约束得到体现；
- 没有把估算写成实时事实；
- 复制和 Markdown 导出正常。

### 7.3 真实重新生成

至少覆盖：

- 单条修改；
- 两条修改；
- 3 条 Pending Changes；
- 5 条 Pending Changes；
- 同时包含“保留、替换、删除、加类似”和一条自由文本补充。

验收：

- 请求仍使用 `tripRequest + previousPlan + modificationRequest`；
- 返回完整新 `TripPlan`；
- 未修改约束尽量保留；
- `appliedChanges` 可核对；
- 失败时旧方案仍保留；
- 不产生局部 Patch。

### 7.4 JSON 与天气安全

- 合法 JSON 直接通过；
- JSON 包在代码块或附带少量文字时仍能提取；
- 首次 Schema 校验失败时最多修复一次；
- 修复后仍不合法时返回 `AI_OUTPUT_INVALID`；
- WeatherProvider 不可用时，模型不能把自由文本天气说法变成可信事实；
- 修复后的结果也必须经过同一套天气清洗。

## 8. 成功标准

v1.2.5 成功需同时满足：

1. mobile `/plan` 固定选项组不再依赖横向滚动；
2. 360px、390px、430px 和 desktop 主流程无回归；
3. Qwen 可通过既有环境变量完成解析、首次生成和重新生成；
4. 3—5 条 Pending Changes 有记录清楚的真实模型测试结果；
5. Schema 校验、一次修复和天气清洗仍有效；
6. Mock AI 可明确切回；
7. 未修改核心 API 与 `TripPlan` Schema；
8. 自动化检查和手动冒烟通过。

真实模型偶发网络失败不要求用 Mock 掩盖，但必须返回现有友好错误，并能安全重试。

## 9. 明确不做

- 不接高德地图；
- 不接 POI；
- 不接路线规划；
- 不新增真实天气能力，不改现有 WeatherProvider；
- 不做景点实时开放时间；
- 不做真实票价、酒店价格、库存、预订或支付；
- 不做保存分享；
- 不做已有计划导入；
- 不做登录、账户或数据库；
- 不做拖拽；
- 不做格子高低调整；
- 不做加减格子或隔板；
- 不做精确时间轴；
- 不做地图工作台；
- 不做完整聊天系统；
- 不做局部 Patch；
- 不改 `TripPlan` Schema；
- 不改 `/api/generate-trip` 协议；
- 不引入平台 SDK；
- 不做模型选择 UI；
- 不做自动多模型路由；
- 不做真实失败后的静默 Mock 回退；
- 不借 mobile 补丁重做整套 `/plan`。

## 11. v1.2.5 完成定义

只有同时满足以下条件，v1.2.5 才算完成：

1. 两条目标都已完成，而不是只完成 Qwen 或只完成 mobile；
2. mobile 固定选项在目标宽度下可完整比较且无横向滚动；
3. 真实模型模式经过首次生成、重新生成和 3—5 条修改冒烟；
4. Mock 模式仍能零 Key 跑通主流程；
5. 核心 Schema、API、天气 Provider 和完整重排架构未改变；
6. 不做项逐项核对无越界；
7. lint、typecheck、build、test 全部通过；
8. README、手动验收和阶段状态同步到真实结果。
