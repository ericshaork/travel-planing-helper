# Wanderly PRD v1.8

## 1. 文档状态

- 文档用途：定义 `v1.8` 的产品范围、全局约束、核心页面目标和验收边界。
- 当前状态：准备阶段文档，不代表功能已经完成。
- 依赖基线：以 `v1.7` 可运行基线和 `docs/V1.7_RELEASE_NOTES.md` 为事实基础。
- 生效方式：后续各阶段实现、截图验收和用户确认应以本文为准。

## 2. 项目背景

Wanderly 是一个 AI 自由行规划网站。

`v1.7` 已形成可运行基线，当前仓库中已经具备：

- AI 旅行需求解析与行程生成；
- 天气、POI、路线和地图基础能力；
- Landing、Create、Plan、Result/Workspace、Login、Trips、Explore 第一版页面；
- Supabase Magic Link 登录；
- 旅行计划保存、打开、更新、删除；
- Explore 第一版档案结构、接口和部分视觉资产；
- 工作台第一版双栏结构与 Day/地图基础联动。

但这些能力仍存在两个问题：

1. 页面完成度不一致，很多地方仍停留在“第一版可用”而不是“可正式展示和试用”；
2. Landing、Explore、Workspace、Settings 之间的导航、状态、数据和视觉关系还没有收口成统一产品。

因此，`v1.8` 的目标不是推翻 `v1.7`，也不是只做表层换皮，而是在既有可运行基线上完成产品级收口。

## 3. v1.8 版本定位

`v1.8` 的核心定位是：

> 旅行杂志式阅读体验 + 私人旅行手帐式编辑体验 + AI 旅行规划能力。

具体来说：

- Landing 负责第一次打开时的品牌欢迎与入口分流；
- Explore 负责日常发现、搜索、灵感探索与 AI 创建分流；
- Workspace 负责同一份旅行计划的阅读与编辑；
- Settings 负责账户、基础偏好与最小可用设置中心。

`v1.8` 不是：

- 重做 AI 主链路；
- 重写地图、天气、POI、路线接口；
- 重构登录体系；
- 推翻既有 Trip 数据结构；
- 一次性同时深改桌面端和移动端；
- 国际版内容上线版本。

## 4. 核心目标

`v1.8` 需要在 `v1.7` 基线上逐页完成以下目标：

1. 完成关键页面的技术补全，消除阻塞后续产品收口的结构问题。
2. 完成桌面端 UI、交互和文案精修，让核心流程达到可公开展示和真实试用的完成度。
3. 正式梳理并接入图片与视觉资产，让视觉资源服务产品，而不是停留在素材堆积状态。
4. 收口页面之间的导航、数据流和状态衔接，避免每个页面各自为战。
5. 形成一版可以给朋友演示、邀请实际试用的桌面端产品。

## 5. 非目标

`v1.8` 明确不做：

- 移动端正式体验重构；
- 国际目的地正式上线；
- 长期用户记忆完整产品；
- 登录方式大改；
- 地图高级增强能力的全部实现；
- 分享、协作、支付、下单、历史版本、多人协作；
- 数据库和主 API 的大规模重写；
- 把未来方向全部塞进本版本。

## 6. 目标用户体验

用户在 `v1.8` 中应当感受到：

- 第一次打开时，Landing 让人一眼记住这是什么产品，并愿意开始探索；
- 日常进入时，Explore 是清楚、好逛、能直接开始的默认首页；
- 打开一篇旅行档案时，像在读一份有图有路线的旅行杂志，而不是后台卡片集合；
- 进入 Workspace 后，像在看和改自己的旅行手帐，而不是在操作工程面板；
- 登录、保存、打开、更新、删除、重新生成这些动作都清楚、可信、不迷路；
- 即使去掉装饰性英文，整站也仍然完全可理解、可操作。

## 7. 全局用户流程

`v1.8` 的全局用户流程分为两条主线：

### 7.1 发现与创建主线

`Landing /welcome`
→ `Explore`
→ `直接搜索` 或 `灵感探索` 或 `AI 直接创建`
→ `Create / Plan`
→ `Workspace`
→ `保存 / 返回 / 再次编辑`

### 7.2 已有计划主线

`Explore` 或 `My Trips`
→ 打开已有旅行档案或个人计划
→ `Workspace Read Mode`
→ `Edit Mode`
→ `保存 / 放弃 / 更新`

这两条主线共享同一套登录状态、导航、Trip 数据和保存机制。

## 8. 首次访问与老用户逻辑

### 8.1 首次访问

- Landing 只作为新用户首次访问时的品牌欢迎页。
- 仅“打开 Landing、刷新、关闭页面”不算完成首次引导。
- 只有用户点击 `Explore`、点击 `Create/AI 规划` 或完成登录后，才记录首次引导完成。

### 8.2 已完成引导用户

- 已完成首次引导或已登录用户，再访问根路径时默认进入 `Explore`。
- 保留 `/welcome` 路由，允许用户主动回看 Landing。
- 直接访问深层链接时，不强制跳回 Landing。

### 8.3 实现原则

- 首次访问状态应是全局产品状态的一部分，而不是 Landing 单页内部私有逻辑。
- 该逻辑需要与登录状态、根路径跳转和 Header 导航统一设计。

## 9. Landing 范围

### 9.1 页面定位

- 只服务于第一次打开产品的品牌欢迎与分流。
- 必须建立“惊艳、有记忆点、可信”的第一印象。
- 不承担老用户日常首页职责。

### 9.2 主入口

- 主按钮：进入 `Explore`。
- 次按钮：进入 `Create / AI 规划`。

### 9.3 需要完成的工作

- 首次访问状态与根路径跳转逻辑；
- `/welcome` 正式路由与回看入口；
- Hero、布局、文案、动效和桌面适配；
- 图片筛选、替换、裁切或重生；
- 与 `Explore`、`Create`、登录状态和全局 Header 的衔接。

### 9.4 不在本页单独解决的问题

- 不在 Landing 中复制完整 Create 表单；
- 不在 Landing 中承载老用户日常浏览；
- 不把 Explore 内容本身塞回 Landing。

## 10. Explore 范围

Explore 是老用户日常首页，也是发现与创建分流中心。

### 10.1 直接搜索

- 搜索平台现有旅行计划。
- 当前内容少是内容规模问题，不等于搜索逻辑错误。
- `v1.8` 中搜索优先命中现有内容，不直接即时生成新计划。
- 无结果时，可引导用户带着搜索词进入 AI 创建。
- 后续版本可以考虑 AI 解释自然语言搜索意图，但不作为 `v1.8` 必做。

### 10.2 灵感探索

- 面向没有明确目的地、但已有模糊想法的用户。
- 当前一级分类为：地点、同行、美食、季节。
- 一级分类展开二级类别，支持图片、标题、简短说明。
- 支持多选、跨一级分类组合和已选条件展示/删除。
- 已选条件可用于：
  - 搜索平台现有旅行计划；
  - 带入 AI 创建。

### 10.3 AI 直接创建

- 面向不想浏览和不想自己组合条件的用户。
- Explore 只提供清晰入口，不复制完整 Create 表单。
- 进入后仍走现有解析、补充和生成主链路。

### 10.4 旅行档案书架

- 三类入口下方展示平台现有旅行档案。
- 顶部为热门 5 篇横向重点展示；
- 下方为电脑端每行 4 篇的持续浏览书架；
- 第一版正式范围为约 40 篇中国旅行档案；
- 点击任意档案后进入阅读器。

### 10.5 范围说明

- `v1.8` 要明确书架结构、阅读器结构与创建转换流程；
- 但不能把未来更大内容生态误写成当前版本已完成；
- 不能凭空定义此前约 200 篇例文的最终字段结构，应在 Explore 阶段审计真实数据后确定。

## 11. 旅行档案阅读器

### 11.1 定位

- 这是 Explore 的阅读层，不是独立站外详情页。
- 体验目标是“像在翻一份带地图和配图的旅行杂志档案”。

### 11.2 目标结构

- 从右侧打开约占屏幕 65% 的档案纸抽屉；
- 左侧保留 Explore 背景并增加遮罩；
- 抽屉内部独立滚动；
- 支持关闭、收起、恢复；
- 打开另一篇档案时替换当前档案，而不是叠多个抽屉；
- 支持浏览器返回关闭抽屉；
- 刷新详情 URL 后仍可恢复对应档案。

### 11.3 数据原则

在正式进入 Explore 阶段前，需要审计此前生成的约 200 篇例文，确认：

- 列表卡片字段；
- 热门大屏字段；
- 档案正文结构；
- 图片需求；
- “用此行程创建我的旅行”所需模板数据。

在未审计前，不将这些字段写死为既定数据结构。

### 11.4 档案转个人计划

阅读器中提供：

`用此行程创建我的旅行`

目标流程为：

档案文章
→ 复制为个人模板草稿
→ 用户补充出发地、日期、天数、人数、预算和偏好
→ AI 依据模板适配
→ 进入个人 Workspace
→ 用户确认后保存

## 12. Workspace 范围

Workspace 不是普通结果页，而是同一份个人旅行计划的两种使用状态。

### 12.1 共通前提

- 使用同一份 Trip 数据；
- 左侧计划、右侧地图；
- 当前 Day 与地图同步；
- 左右占比可调整；
- 可从 Explore 模板、Create 主链路和 My Trips 打开进入。

### 12.2 Read Mode

- 默认进入；
- 定位为“像阅读一本带地图的旅行杂志”；
- 重点包括：大图、Day Timeline、景点卡、交通信息、天气建议、路线阅读、克制的手帐装饰；
- 地图跟随当前 Day 变化。

### 12.3 Edit Mode

- 点击修改后进入；
- 定位为“像使用手帐和积木块编辑旅行计划”；
- 目标包括：景点积木块、修改内容、调整顺序、AI 辅助修改、待修改清单、保存或放弃、地图同步。

### 12.4 范围说明

- 当前 Read Mode 和 Edit Mode 完成度都不高。
- `v1.8` 进入 Workspace 阶段时，需要先审计现状，再细化实现边界。
- 后续地图增强能力可暂不实现，但组件边界、状态边界和数据边界不能写死，避免阻塞未来扩展。

## 13. Settings 范围

### 13.1 当前判断

- 当前长期用户记忆按 0 处理。
- 当前已有的是“单次旅行的填写偏好”，不等于“跨多次旅行长期保存的用户记忆”。

### 13.2 v1.8 基础版

`v1.8` 先完成最小可用 Settings：

- 正式 Settings 路由；
- 当前登录邮箱；
- 登录状态；
- 退出登录；
- 查看欢迎页入口；
- 基础显示设置；
- 基础动效设置；
- 必要地图显示偏好；
- 清晰的保存、成功和失败状态。

### 13.3 不在本版本完成

- 不把长期用户记忆写成现有能力；
- 不做隐式行为学习；
- 不做复杂偏好自动推断。

### 13.4 后续方向

后续可以逐步加入：

- 常用出发城市；
- 默认预算；
- 默认旅行节奏；
- 常见同行类型；
- 兴趣标签；
- 饮食偏好；
- 特殊出行要求；
- Create 自动预填；
- 更多透明、可编辑、可关闭的长期记忆能力。

## 14. 登录系统范围

### 14.1 当前方式

- `v1.8` 短期保留 Supabase Magic Link。

### 14.2 v1.8 需要精修的内容

- 中文说明；
- 邮件已发送状态；
- 重新发送；
- 链接过期提示；
- 错误提示；
- 登录完成后返回原目标页面；
- 避免登录后迷路。

### 14.3 非目标

- 邮箱密码；
- 邮箱验证码；
- 其他登录方式；
- 登录核心协议重写。

## 15. 内容与地域范围

### 15.1 正式内容范围

`v1.8` 正式内容仅包含：

- 中国大陆；
- 香港；
- 澳门；
- 台湾。

### 15.2 非正式范围处理

- 国外图片和内容不要求直接删除；
- 可以归档或保留为未来国际版候选；
- 但不能进入 `v1.8` 正式 Explore 页面。

## 16. 语言规范

### 16.1 功能性内容

全部功能性内容使用中文，包括：

- 导航；
- 按钮；
- 表单；
- 错误提示；
- 空状态；
- 加载状态；
- 旅行档案正文；
- Workspace 操作；
- Settings。

### 16.2 允许的少量外文

- `Wanderly`
- 邮票文字
- 档案编号
- 品牌格言
- 很短的装饰栏目字

### 16.3 禁止项

禁止使用工程式或内部开发式外文作为主要界面内容，例如：

- `PLAN BAR`
- `CURRENT DAY`
- `Desktop Workspace`
- `active day sync`

原则是：去掉装饰性外文后，用户仍能完整理解和操作网站。

## 17. 图片与视觉资产原则

每个页面都必须包含独立的图片资产工作：

1. 扫描当前引用；
2. 判断保留、删除、归档、裁切、透明化、替换或重生；
3. 生成候选素材；
4. 用户确认；
5. 处理正式尺寸与格式；
6. 接入代码；
7. 检查页面效果；
8. 检查性能；
9. 清理无用引用。

同时必须避免：

- 把总图直接作为运行时单图；
- 该裁切的问题误当成重生；
- 该重生的问题只做机械裁切；
- 未经用户确认批量删除图片；
- 图片移动后不更新代码引用。

页面装饰强度原则：

- Landing：最强品牌与海报感；
- Explore：旅行杂志目录和档案馆感；
- Workspace：有手帐气质，但信息清晰优先；
- Settings：最克制。

整体避免：

- 大面积黑色 UI；
- 科技感 dashboard；
- 过度卡片化 SaaS；
- 满屏贴纸；
- 装饰压过功能。

## 18. Desktop First 范围

- `v1.8` 采用 Desktop First。
- 电脑端是正式设计目标和验收目标。
- 移动端不做深度改造，只要求不因桌面端修改而严重崩溃或完全不可访问。
- 移动端正式体验进入后续独立版本。

## 19. 跨页面状态与数据衔接

四个核心页面不是互相独立的，必须统一考虑：

- Landing 的首次访问状态如何影响根路径和 Explore；
- Explore 的搜索、灵感和档案模板如何进入 Create / Plan / Workspace；
- Workspace 如何区分本地草稿、已保存计划和从档案复制而来的个人模板；
- Settings 如何影响欢迎页展示、动效、地图偏好和基础显示设置；
- 登录后如何返回原目标页面；
- Header、导航、用户状态和保存状态如何跨页统一。

后续实现时，不能先把一个页面做死，再让其他页面被动适配。

## 20. 工程与 Git 规则

- 每个大页面阶段遵循：
  修改
  → 测试
  → 页面截图和变更清单
  → 用户人工确认
  → 继续下一步
- 不要每个小 Step 都提交。
- 一个完整版本或用户明确确认的阶段完成后，再统一 `commit` 和 `push`。
- 未经用户确认：
  - 不 `commit`
  - 不 `push`
  - 不创建 PR
  - 不发布线上版本

## 21. v1.8 成功标准

`v1.8` 可以视为成功，当且仅当：

1. Landing、Explore、Workspace、Settings 形成统一桌面端产品链路；
2. Landing 只承担首次欢迎页职责，老用户默认进入 Explore；
3. Explore 形成清晰的“搜索 / 灵感 / AI 创建”三入口；
4. 旅行档案阅读器形成稳定的桌面阅读体验；
5. Workspace 的 Read/Edit 双模式具备真实可用的阅读和编辑完成度；
6. 登录、保存、打开、更新、删除、返回原页面等关键状态可信、清晰；
7. 图片与视觉资产正式接入并与页面完成度匹配；
8. 桌面端达到可给朋友展示和试用的完成度。

## 22. v1.8 验收标准

验收时至少需要确认：

- 根路径、`/welcome`、Explore、Create、Plan、Workspace、Login、Trips、Settings 的入口关系正确；
- 首次访问与老用户逻辑正确；
- Explore 三类入口和档案书架结构完整；
- 旅行档案阅读器支持打开、关闭、收起、恢复和返回关闭；
- Workspace 从 AI 创建、Explore 模板、My Trips 进入都不迷路；
- Read/Edit 模式切换、保存状态、放弃修改和地图同步清晰可用；
- 全站主要功能文案为中文；
- 正式 Explore 内容范围不出现国外目的地；
- 桌面端截图验收通过；
- 移动端无严重破版。

## 23. 已知风险

- 当前 v1.7 基线中，很多能力已经存在，但完成度不一致，容易误判为“只差换皮”。
- Explore 例文与档案字段在未审计前不能写死，否则容易让实现和真实数据脱节。
- Workspace Read/Edit 的边界如果不先审计，后续容易在已有组件上反复返工。
- 图片资产已很多，但未必都适合运行时，需要持续筛选、替换和清理。
- Landing、Explore、Workspace、Settings 同时牵涉导航和状态流，若先各做各的，后续整合成本会很高。

## 24. 后续版本方向

- AI 理解自然语言搜索意图的增强搜索；
- 用户主动公开私人计划；
- 地图高级交互能力；
- 长期用户记忆体系；
- 国际版内容；
- 移动端正式重构；
- 更完整登录体系；
- 分享、协作、历史版本与多端同步。
