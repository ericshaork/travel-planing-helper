# v1.3 分阶段实施计划

## 1. 阶段总览

`v1.3` 按顺序拆成 8 个阶段：

```text
阶段 0：规划文档与范围冻结
  -> 阶段 1：Provider 架构与 Mock POI / Route
  -> 阶段 2：Amap POI 接入
  -> 阶段 3：Amap Route 接入
  -> 阶段 4：行程节奏判断
  -> 阶段 5：天气增强
  -> 阶段 6：基础地图 / route insight 展示
  -> 阶段 7：总体验收与部署准备
```

通用边界：

- 继续复用现有 `TripPlan`
- 优先做 provider、enrichment、derived layer
- 先有 mock，再接真实 provider
- 不为了 `v1.3` 重做整站 UI
- 不做登录、收藏、数据库、支付或多人协作

## 3. 已完成阶段摘要

### 阶段 0

- 新增 `docs/PRD_v1.3.md`
- 新增 `docs/TECH_DESIGN_v1.3.md`
- 新增 `docs/V1.3_PHASE_PLAN.md`
- 更新 `README.md`
- 冻结 `v1.3` 核心目标为“真实地点与路线可信度增强”

### 阶段 1

- 新增 `PoiProvider` / `RouteProvider`
- 新增 `MockPoiProvider` / `MockRouteProvider`
- 新增 `getPoiProvider()` / `getRouteProvider()`
- 新增 `USE_MOCK_POI` / `USE_MOCK_ROUTE`
- mock 模式零 Key 可跑

### 阶段 2

- 新增 `AmapPoiProvider`
- 服务端 `fetch` 调高德 Web 服务 POI 文本搜索
- `POI_PROVIDER=amap` 且 `USE_MOCK_POI=false` 时切到真实高德
- 空结果返回 warning，不把“查不到”当致命错误

### 阶段 3

- 新增 `AmapRouteProvider`
- 支持 `driving` / `walking`
- `transit` 当前明确 warning 并降级
- Route 与 POI 共用 `AMAP_API_KEY`

### 阶段 4

- 新增 `TripEnrichment`
- 新增 `DayRouteSummary` / `PaceWarning` / `MapPoint`
- 新增 `extractPlaceNamesFromTripPlan()`
- 新增 `analyzeDayPace()`
- 新增 `enrichTripPlanWithRoutes()`

### 阶段 5

- 新增 `lib/weather/summary.ts`
- 新增 `WeatherImpact` / `WeatherDaySummary` / `TripWeatherSummary`
- 新增 `buildTripWeatherSummary()` 与 `analyzeWeatherImpact()`
- 已落地降雨、高温、低温、大风、天气不可用的保守提示规则
- `WEATHER_PROVIDER` 现支持 `mock` / `qweather`
- `USE_MOCK_WEATHER=true` 或 `WEATHER_PROVIDER=mock` 时零 Key 可跑

### 阶段 6

- 新增 `POST /api/enrich-trip`
- 新增 `buildTripResultEnrichment()`
- `/result` desktop 在每日行程旁增加轻量 route insight
- mobile 保留原分页结构，并新增 `route` 页
- enrichment 作为派生数据异步加载，失败不影响 `TripPlan` 主阅读链路
- 当前“地图”仍以点位列表和路线摘要为主，不接前端地图 SDK

### 阶段 7

- 补齐 README、阶段计划、技术设计、手动验收清单
- 复核 `.env.example` 完整性
- 复核服务端 Key 边界与前后端职责边界
- 重新执行 `lint` / `typecheck` / `build` / `test`
- 确认 `v1.3` 不改主 schema、不进入 `v1.4`

## 4. 阶段 7 收尾结论

- `v1.3` 主功能与派生层范围已完成
- `TripPlan` schema 未变
- `/api/generate-trip` 与 `/api/parse-trip` 未改协议
- `AMAP_API_KEY`、`QWEATHER_API_KEY`、`LLM_API_KEY` 继续只在服务端读取
- `/result` 客户端页通过 `/api/enrich-trip` 拉派生数据，不直接 import provider
- 当前版本可作为 `v1.3` 发布候选

## 5. 当前明确不做

- 不做完整 Desktop Workspace UI
- 不做完整 Mobile Workspace UI
- 不接前端地图 SDK
- 不做地图选点写回行程
- 不把 enrichment 结果写回主 schema
- 不进入 `v1.4`

## 6. 后续承接

- `v1.4` = Desktop Workspace UI
- `v1.5` = Mobile Workspace UI
- `v2.0` = 更完整的旅行工作台
