# V1.8.4 Auth, User Center, And Settings Audit

## 1. Audit Scope

本阶段只做现状审计，不修改业务代码，不删除文件，不 commit，不 push。

本次审计通过阅读现有页面、组件、Supabase 初始化代码、trips API、migration 与 localStorage 存储逻辑完成，重点判断：

- 当前 Auth 是否已具备正常网站产品的基础形态；
- Supabase 与 localStorage 的职责边界是否清晰；
- My Trips / Workspace 保存链路是否能承接用户系统；
- Settings 与长期偏好是否已经存在；
- 未登录用户体验与登录门槛是否符合 v1.8.4 目标。

## 2. Product Direction

v1.8.4 的目标，是把 Wanderly 从“本地可用旅行工作台”推进到“正常网站产品”阶段：

- 支持注册 / 登录 / 退出；
- 支持保存和找回旅行计划；
- 支持 My Trips 作为长期历史计划入口；
- 支持 Settings 存用户长期默认偏好；
- AI 生成时同时读取“用户长期偏好 + 当前旅行偏好”；
- Supabase 继续承担长期存储；
- localStorage 只承担未登录草稿、当前 workspace session 和 UI 状态缓存。

当前项目已经有一部分基础设施，但还没有形成完整用户系统。

## 3. Current Auth System

### 3.1 当前是否使用 Supabase Auth

是。当前登录系统基于 Supabase Auth：

- 浏览器端 client：`lib/supabase/browser.ts`
- 服务端 / access token client：`lib/supabase/server.ts`
- auth 状态与操作封装：`lib/supabase/auth-client.ts`

浏览器 client 配置为：

- `autoRefreshToken: true`
- `detectSessionInUrl: true`
- `persistSession: true`

这意味着 magic link 回跳后的 session 会自动写入 Supabase 的浏览器持久化存储，并在刷新后继续保留。

### 3.2 当前登录状态如何获取

当前登录状态由 `components/auth/useAuthStatus.ts` 统一获取：

- 初始化时调用 `client.auth.getUser()`
- 订阅 `client.auth.onAuthStateChange(...)`
- 状态模型分为 `loading / anonymous / authenticated`

页面顶部登录状态主要通过：

- `components/auth/UserMenu.tsx`
- `components/auth/AuthStatus.tsx`

来展示。

### 3.3 当前登录方式

当前项目只看到邮箱 magic link / OTP 登录，没有发现真正的“密码注册 / 密码登录”实现。

证据：

- `components/auth/LoginForm.tsx` 只调用 `signInWithMagicLink(...)`
- `lib/supabase/auth-client.ts` 内部调用 `client.auth.signInWithOtp(...)`
- `shouldCreateUser: true`

这表示：

- 现状是“输入邮箱 -> 发登录链接 -> 点击邮箱链接回跳”；
- 首次输入邮箱时，Supabase 会自动创建用户；
- 没有单独的注册流程；
- 没有密码模式；
- 没有独立的“注册页”。

### 3.4 登录页 / 注册页 / 退出

当前页面现状：

- 登录页存在：`app/login/page.tsx`
- 注册页不存在：未发现 `app/register/*`
- 独立 auth callback 页面未发现：未发现 `app/auth/*`
- 退出功能存在：`components/auth/SignOutButton.tsx`

退出调用 `client.auth.signOut()`，然后 `router.refresh()` 更新界面。

### 3.5 登录后回跳逻辑

当前登录页支持 `returnTo` 参数：

- `app/login/page.tsx`
- `components/auth/LoginForm.tsx`
- `lib/trips/save-client.ts` 里的 `buildSaveTripLoginHref(...)`

现有主要回跳使用场景：

- 未登录保存 Workspace 时，回跳 `/workspace`
- 未登录进入 `/trips` 时，回跳 `/trips`

说明回跳机制已存在，但使用范围还比较窄，尚未形成全站统一登录门槛策略。

### 3.6 为什么用户可能感觉“一直处于登录状态”

主要原因很明确：

- `persistSession: true`
- `detectSessionInUrl: true`
- 顶部 `UserMenu` 持续显示已登录状态

也就是说，只要 Supabase session 没失效、用户没主动退出，刷新页面后仍然会保持登录。

这本身不是 bug，但目前产品层没有把“为什么现在是登录状态、怎么退出、退出后会发生什么”解释得足够清楚，容易让用户感觉“总是登录着”。

### 3.7 未登录访问受限页面如何处理

当前未登录访问受限能力的处理方式是“页面内提示登录”或“操作时跳登录页”，而不是全站统一拦截。

已确认：

- `/trips` 未登录会展示登录提示，而不是直接报错
- Workspace 点击“保存计划”时，未登录会跳 `/login?returnTo=/workspace`
- trips API / detail API 未带 token 会返回 401

但还没有看到完整的“受限页面守卫体系”。

这会直接影响后续：

## 5. localStorage And Supabase Boundary

### 5.1 当前 localStorage key

当前已确认的主要 key：

- `travel-planning:parsed-trip`
- `travel-planning:trip-draft`
- `travel-planning:trip-plan-draft`
- `travel-planning:trip-request`
- `travel-planning:trip-plan`
- `travel-planning:trip-enrichment`
- `travel-planning:trip-weather-summary`
- `travel-planning:restored-saved-trip`
- `travel-planning:workspace-session`
- `wanderly:workspace-split-ratio`
- `wanderly_explore_favorites`

### 5.2 这些 key 当前各自承担什么

偏业务态：

- `travel-planning:trip-request`
  当前旅行请求
- `travel-planning:trip-plan`
  当前行程结果
- `travel-planning:trip-enrichment`
  enrichment 数据
- `travel-planning:trip-weather-summary`
  天气摘要
- `travel-planning:trip-draft`
  create / plan 草稿
- `travel-planning:trip-plan-draft`
  完整草稿态
- `travel-planning:parsed-trip`
  NLP 解析结果

偏工作台 session：

- `travel-planning:restored-saved-trip`
  当前恢复的 saved trip metadata
- `travel-planning:workspace-session`
  当前 workspace 来源 / 默认模式 / 本地 draft id

偏 UI 状态：

- `wanderly:workspace-split-ratio`
  左右分栏比例

偏 Explore 收藏：

- `wanderly_explore_favorites`
  目前仍是 localStorage 收藏，并按 `user_id` 伪分组，但不走 Supabase。

### 5.3 当前边界是否清晰

目前边界是“能跑”，但还不够清晰。

比较清楚的部分：

- 长期 trips 在 Supabase
- 当前 Workspace 工作态在 localStorage
- 未登录 blank draft 在 localStorage

不够清楚的部分：

- 已登录用户的当前 Workspace 草稿，仍高度依赖 localStorage 作为主工作态
- saved trip 是否与当前本地工作态完全同步，依赖恢复流程和 metadata
- Explore favorites 仍是 localStorage，不属于真正用户中心
- 没有明确的 local draft -> cloud draft 升级规则

### 5.4 是否存在过期状态 / 混乱风险

存在。

重点风险：

- localStorage 里保留旧 `trip-plan / trip-request / workspace-session` 时，用户可能误以为这是“云端最新状态”
- `restored-saved-trip` metadata 与实际 `trip-plan` 内容可能出现不完全一致
- `workspace-session` 的 sourceType 只在前端保存，刷新和跨设备无法作为长期真相
- favorites 使用 localStorage，会导致“登录了但换设备收藏没了”

结论：v1.8.4 需要显式定义“哪些是缓存，哪些是真相来源”。

## 8. User Defaults vs Trip-Specific Preferences

### 8.1 当前已存在什么偏好字段

当前“本次旅行偏好”是存在的，主要在 `TripRequest / TripRequestDraft` 中：

- `interests`
- `travelStyles`
- `mustVisitPlaces`
- `avoidPlaces`
- `accommodationPreference`
- `localTransportPreference`
- `schedulePreference`
- `specialRequirements`

Create / Plan / Explore 导入草稿都会落到这套 request 结构里。

### 8.2 当前是否已有“用户长期默认偏好”

没有。

没有看到：

- user defaults 表
- profile 上的 preference 字段
- settings 读取逻辑
- AI prompt 里注入 user defaults 的逻辑

### 8.3 当前 prompt 是否区分两层偏好

没有。

当前 AI 生成链路：

- `app/api/generate-trip/route.ts`
- `lib/trip/planner.ts`
- `lib/ai/generateTrip.ts`
- `lib/ai/prompts.ts`

本质上只把本次 `tripRequest` 传入模型，再加天气上下文。

没有出现：

- `userDefaults`
- `profilePreferences`
- `settingsPreferences`

这说明目前 AI 只读“这次旅行偏好”，不读“长期默认偏好”。

### 8.4 当前长期偏好与单篇偏好是否混在一起

更准确地说：当前只有单篇偏好，长期偏好层缺失。

所以 v1.8.4 不是“冲突处理没做好”，而是“第二层根本还没建起来”。

### 8.5 最小改造建议

建议最小落地方式：

1. 在 Supabase 新增用户设置表，专门存长期默认偏好。
2. 本次旅行偏好继续保留在 `TripRequest`。
3. 生成前做一层 merge：
   - 先读 user defaults
   - 再叠加本次 trip request
   - 冲突时本次 trip request 优先
4. 不自动把本次偏好回写 Settings。
5. 只在用户明确点击“保存为我的默认偏好”时，才更新长期偏好。

## 9. Guest Experience And Login Gates

### 9.1 当前未登录用户能做什么

基本符合“轻体验先上手”的方向：

- 浏览 Landing：可以
- 浏览 Explore：可以
- 进入 Create：可以
- 使用 AI 生成：可以
- 进入 Workspace：可以
- 打开空白计划：可以
- 编辑当前本地工作台：可以

### 9.2 当前未登录用户不能做什么

当前明确要求登录的能力：

- 保存计划到云端
- 进入 My Trips
- 打开历史计划（因为历史计划来自云端）

### 9.3 登录门槛当前是否合理

大方向是合理的，但还不够完整。

符合目标的地方：

- 没有强制用户一上来就登录
- 保存时再提示登录
- My Trips 作为登录能力隔离开

还需要补的地方：

- “继续长期编辑”目前虽然业务上仍可继续本地编辑，但产品上还没有清楚说明“未登录只是在本机缓存”
- Settings 还不存在，所以“持久保存设置需登录”还没有正式落地
- 登录成功后回到原操作的链路还没有覆盖所有可能入口

### 9.4 是否存在原生 alert / confirm / prompt

本次审计未发现浏览器原生 `window.alert / confirm / prompt` 还残留在 auth / trips / settings 相关主链路中。

### 9.5 建议的登录门槛原则

建议 v1.8.4 统一成：

未登录可做：

- Landing / Explore 浏览
- Create / AI 首次生成
- 查看结果
- 进入 Workspace 短期编辑

触发登录的动作：

- 保存到我的行程
- 打开我的行程
- 打开历史计划
- 保存默认偏好
- 使用需要长期身份绑定的数据功能

## 10. Route And Entry Flow Impact

### 10.1 当前主链路是否还成立

当前 v1.8.3 已完成的主链路，基本仍然成立：

- Landing -> Explore：可用
- Landing / Create -> AI 创建 -> Workspace：可用
- Landing / Create -> 空白创建 -> Workspace：可用
- Explore -> 导入到工作台 -> Workspace：可用
- My Trips -> Workspace：可用，但要求登录
- Workspace Read/Edit 切换：可用
- `/result`：兼容入口保留

### 10.2 哪些链路不能被登录打断

不应被登录打断：

- Landing 浏览
- Explore 浏览
- AI 首次生成
- 生成结果查看
- 空白工作台创建
- Explore 导入到 Workspace
- Workspace 本地阅读 / 编辑

### 10.3 哪些链路应该加登录门槛

应该明确登录门槛：

- Workspace 保存
- My Trips
- 历史计划打开
- Settings 持久保存
- 任何跨设备可找回的数据能力

### 10.4 `/result` 的影响

`/result` 当前仍应保持兼容，但在 auth / settings 阶段不应该再作为“主心智入口”强化。

用户身份、保存、长期编辑、历史计划恢复，都应继续围绕 `/workspace`。

## 11. Issues

### P0

- 缺少真正的注册 / 登录正常化形态。当前只有邮箱 magic link，没有密码注册、密码登录，也没有独立注册页，用户系统还不能算“正常网站产品”。
- 缺少真正的 Settings 页面与长期偏好数据层。当前没有 `/settings`、没有 settings 表、没有用户默认偏好读写链路。
- 长期偏好未接入 AI。当前 prompt 只读取本次 `TripRequest`，还没有“用户默认偏好 + 本次偏好”的双层输入。
- `trip_plans` 缺少稳定 `source_type` 等来源字段，无法支撑 My Trips 对 AI 生成 / 空白手搓 / Explore 导入做长期分类管理。

### P1

- localStorage 与 Supabase 的真相边界不够清楚。当前 Workspace 工作态高度依赖本地缓存，容易出现“本地旧计划”和“云端已保存计划”心智混淆。
- 登录持久化是正常的，但“为什么还在登录 / 如何明确退出 / 退出后影响什么”的产品表达不够清楚，容易让用户误判为异常状态。
- My Trips 目前只有列表 + 打开 + 删除，缺少搜索、筛选、重命名、分类标签等用户中心基本管理能力。
- Explore favorites 仍保存在 localStorage，不属于真正的用户中心数据。
- 登录回跳虽已存在，但尚未形成全站统一策略，后续在保存、设置、历史打开、导入等动作上仍需统一。

### P2

- Workspace Sidebar 中已经有“设置”入口占位，但当前只是占位项，容易提前制造用户预期。
- 登录页文案仍明显带有“阶段说明 / phase 心智”，更像开发阶段说明页，而不是正式网站登录页。
- My Trips 页面与登录页的文案、结构还偏过渡态，后续需要更统一的中文产品表达与 Wanderly 风格收口。

## 12. Recommendations For Step 2-8

### Step 2：用户系统产品规则设计

- 明确 Guest / Authenticated 两种模式下各页面和按钮权限。
- 明确“短期本地工作态”和“长期云端保存”的文案与规则。
- 明确登录后回跳策略：保存、My Trips、Settings、历史计划打开都统一处理。

### Step 3：注册 / 登录 / 退出正常化

- 保留 Supabase Auth，不切数据库。
- 在 magic link 基础上决定是否补密码模式；如果继续 magic link，也要把注册 / 登录表达做成正常网站可理解模式。
- 增加明确的注册入口与退出反馈。
- 统一 Header / 登录页 / 登录后状态展示。

### Step 4：登录门槛与保存规则

- 未登录用户允许 AI 首次生成和短期编辑。
- 保存到云端时统一提示登录。
- 登录后回到原操作继续保存。
- 明确“未登录编辑 = 本机缓存；登录保存 = 云端找回”。

### Step 5：My Trips 完善

- 在 `trip_plans` 增加来源字段与状态字段。
- My Trips 增加搜索、筛选、标签和重命名。
- 区分 AI 生成 / 空白手搓 / Explore 导入 / 已保存草稿。

### Step 6：Settings 页面骨架

- 新建 `/settings`
- 分区至少包含：
  - 账号设置
  - 长期旅行偏好
  - Workspace 偏好
  - AI 偏好
- 先保证读写和登录门槛正确，再做视觉 polish。

### Step 7：用户偏好接入 AI

- Supabase 新增长期偏好表或扩展 profile/settings 表。
- 生成前 merge：
  - user defaults
  - trip-specific request
- 冲突时单篇计划优先。
- 提供“保存为我的默认偏好”显式动作。

### Step 8：收口验收

- 验证 Landing / Explore / Create / Workspace / My Trips / Settings 全链路。
- 验证 Guest 与 Authenticated 的权限边界。
- 验证保存、恢复、退出、切设备找回是否成立。
- 验证中文文案、错误提示、登录回跳和空状态是否统一。
