# Wanderly v1.8 Explore Archive Schema For Generator

## 1. 目的

这份 schema 不是给前端直接渲染用的，而是给半自动文案生成器、后续批量扩写流程和标准化校验链路使用的。

对应机读文件：

- [data/explore/schema/explore_archive_schema_v1.json](C:/Users/10200/Desktop/travel_planing/data/explore/schema/explore_archive_schema_v1.json)

## 2. 适用范围

v1.8 正式 Explore 内容只允许：

- 中国大陆
- 香港
- 澳门
- 台湾

当前生成器不要产出国外城市内容。

## 3. 核心约束

- `externalId` 必须稳定唯一，不能用随机数。
- `slug` 必须英文小写、短横线连接、全局唯一。
- `theme` 必须从受控集合中选，不能自由发挥。
- `title`、`summary`、`story`、`featuredReason`、`recommendedFor` 必须是中文可读文案。
- `dailyItinerary` 必须结构化，不能再把模板字符串暴露给前端。
- `food` 和 `places` 必须结构化。
- `imagePrompt` 只用于补图，不进入前端展示。

## 4. 受控 theme 集合

- `citywalk`
- `food`
- `heritage`
- `night`
- `seaside`
- `jiangnan`
- `nature`
- `family`
- `romance`
- `museum`
- `shopping`
- `mountain`
- `lake`
- `slowTravel`
- `general`

## 5. 推荐字段生成规则

### `externalId`

- 推荐规则：`explore_{sourceFileStem}`
- 必须稳定、唯一、可追溯

### `slug`

- 唯一时保留原始 slug
- 冲突时追加稳定尾号
- 不允许随机短串

### `story`

- 优先复用 `summary`
- 如果 `summary` 太短，可用 `title + summary` 轻量扩展
- 不要凭空生成长篇散文

### `featuredReason`

- 建议从 `summary` 压缩成一句中文短说明
- 适合精选卡片和抽屉首屏

### `recommendedFor`

- 优先由 `tripType` 推导
- 再由 tags 补充

示例：

- `couple` -> `["情侣"]`
- `family` -> `["亲子", "家庭"]`
- `solo` -> `["独游"]`
- `student` -> `["学生", "朋友同行"]`

### `highlights`

- 建议 2-4 条
- 每条都要有：
  - `title`
  - `description`
  - `tag`

### `dailyItinerary`

- 必须输出每天结构
- 每天必须有：
  - `dayNumber`
  - `title`
  - `summary`
  - `activities`

每条 `activity` 至少有：

- `timeBlock`
- `label`
- `description`
- `poiRefs`
- `foodRefs`

## 6. 禁止内容

生成器和标准化链路都不允许输出这些展示脏值：

- `锟`
- `�`
- `{activity.description}`
- `undefined`
- `null`
- `NaN`

也不要输出：

- 国外城市
- 英文标题
- 英文摘要
- 实时票价
- 实时酒店价格
- 未核实的实时开放状态

## 7. fallback 约定

- `coverImageUrl` 允许为空字符串
- `galleryImages` 允许为空数组
- `food[].imageUrl` 允许为空字符串
- `places[].imageUrl` 允许为空字符串
- `theme` 实在无法稳定判断时回退到 `general`
- `pace` 缺失时回退到 `balanced`
- `budgetLevel` 缺失时回退到 `medium`

## 8. 对 1000 篇扩展的意义

这份 schema 的作用不是只服务当前 200 篇，而是把：

- 字段命名
- 生成边界
- 语言约束
- 主题集合
- fallback 规则

提前固定下来。后续扩到 1000 篇时，生成器只要遵守这份 schema，就能稳定进入同一条 normalize 和前端读取链路。
