# UI Audit v1.7

> 文档状态：v1.7 历史审计记录
> 本文保留 v1.7 规划阶段的界面问题判断、优先级与设计方向。实际实施过程中，为支撑目标 UI，同时补入了必要的技术、数据与页面基础能力。v1.7 最终实际完成状态请以 `docs/V1.7_RELEASE_NOTES.md` 为准；后续精修进入 v1.8。

## A. 审计摘要

### 当前 v1.6 已完成什么

当前产品已经完成一条可用的旅行规划主链路：

- `/create -> /plan -> /result` 可完整生成旅行方案
- AI、天气、POI、路线和地图能力已经接入
- `/result` 已具备 desktop workspace 形态
- 登录、保存、我的行程、历史计划打开、更新、删除已经可用
- 本地工作台状态与 Supabase 云端持久态已经打通

也就是说，产品已经从“只是一个能吐文案的旅行生成器”走到了“有真实工作台结构的 AI 旅行产品雏形”。

### 当前 v1.7 为什么要做 UI / UX 大修

问题已经不在“有没有功能”，而在“功能如何被看见、理解和信任”。

当前 v1.6 的主要短板：

- `/result` 的布局空间分配还不稳，左侧导航对内容有遮挡风险
- 地图价值没有被充分放大，存在感偏弱
- 天气、状态、图标等表达仍偏工程态
- 多个页面仍保留 phase 交付语境，像阶段性 demo，不像成熟产品
- 颜色、层级、卡片和状态组件有基础，但整体记忆点还不够鲜明

### 本轮审计范围

本轮审计覆盖：

- `/result` 工作台
- 左侧导航
- 地图区域
- route inspector
- 天气展示
- marker / active marker
- itinerary block
- top bar / 保存状态
- `/trips`
- `/login`
- `/create`
- `/plan`
- `/` 欢迎页
- icon / color / spacing / card / empty / error / loading / 文案语气一致性
- desktop 结构与 mobile 回归风险

### 本轮不做代码实现

本轮仅做：

- UI / UX 审计
- 问题清单
- 视觉方向建议
- 优先级划分
- 阶段规划

本轮不做：

- 页面代码修改
- 组件实现修改
- CSS / Tailwind 修改
- API / 数据库 / schema 修改

## A+. 用户人工补充观察

### 视觉气质偏好

用户明确偏好更浅色、更轻松的旅行手帐方向，而不是偏工具感或偏企业 SaaS 的界面。现有米黄色纸张底色可以保留，但需要补足低饱和的浅粉、浅绿、浅黄、浅蓝层次，让页面从“保守暖色”升级到“有情绪、有记忆点的浅色手帐系统”。

### 水彩笔触与手帐元素

用户希望探索水彩笔触式强调层：

- 像水彩画笔轻轻涂过的一笔
- 边缘自然
- 轻微晕染
- 低饱和
- 半透明
- 带纸感

它适合用在关键强调位置，例如：

- 按钮文字下方
- active tab
- 当前 Day
- 天气提醒
- active marker
- 欢迎页局部点缀

但不应滥用到：

- 所有按钮
- 删除按钮
- 密集列表小按钮
- 所有 chip

### 欢迎页问题需要提升优先级

用户认为当前欢迎页问题比预期更大，已经不只是“氛围不够”，而是“首屏叙事方式不对”。它现在更像说明页或广告页，文字太多，视觉重心不足，不知道先看哪里。因此欢迎页在 v1.7 的优先级应前移，不再放在最后才做 polish。

### 欢迎页主视觉正式方向

欢迎页主视觉方向已明确为：

- 真实景点图作为主视觉主体
- 水彩手帐风纹理 / 色块作为背景氛围层
- 少量旅行拼贴元素作为辅助点缀

设计原则：

- 真实景点图是主角
- 水彩背景是氛围底层
- 拼贴元素是辅助点缀
- 不做成满屏贴纸墙
- 不让拼贴压过主视觉

目标感受：

- 第一眼像旅行产品
- 第二眼有手帐气质
- 第三眼能看到拼贴细节

### 结果页双栏 + 大地图 + 可拖拽分隔线方向

用户最终明确了 `/result` 的 desktop 方向：

- 不是报告页
- 也不是地图页
- 而是“可编辑旅行工作台”

其核心第一屏应同时回答三件事：

- 这是哪趟旅行
- 当前正在看哪一天
- 这一天大概怎么走

desktop 结构方向也已明确：

- 左侧放行程工作区
- 右侧放大地图和辅助信息
- 中间使用可拖拽分隔线

### `/result` 默认比例正式定为 55 / 45

`/result` desktop 双栏默认初始比例正式定为：

- 左侧工作区 `55%`
- 右侧地图区 `45%`

phase 2 执行时不再写成待定。拖拽范围建议控制在 `40 / 60` 到 `60 / 40`，避免任一侧过窄。

### 天气展示边界

用户对 v1.7 天气模块边界做了明确收束。优先展示：

- 气温
- 是否下雨
- 是否需要带伞
- 降雨概率或雨天提示
- 台风、暴雨、高温、雷暴等极端天气预警
- 是否适合户外活动

不作为重点：

- 普通风力
- 湿度
- 气压

只有在台风、大风预警、海边、山地、高空项目等明显影响旅行决策的场景下，风力才应被提升。

### 现有无效入口和疑似 bug

用户额外指出的具体问题包括：

- `/create` 仍存在“自己慢慢挑 / 地图探索 v1.5”一类无实际功能入口
- `/result` itinerary block 的“查看详情”点击无响应
- `/result` 工作台两侧空白偏多
- `/result` 左侧导航固定策略与内容关系不自然
- 欢迎页与工作台部分文案仍像内部阶段说明
- `v1.7` 明确以 desktop 为主，mobile 仅做基础回归

## C. P0 问题

P0 定义：不修会明显影响使用或主体验。

### P0-1 左侧导航遮挡主内容

- 现象：`DesktopWorkspaceShell` 为 sidebar 只预留 `84px` 列宽，但 `WorkspaceSidebar` hover 后会扩展到 `228px`，属于覆盖式展开。
- 影响页面 / 模块：`/result`、landing desktop workspace 壳。
- 为什么影响体验：会压住左侧主区内容边缘，破坏“稳定工作台”的感受，尤其在用户需要频繁切换导航时更明显。
- 建议方向：优先修成“展开时不遮挡主内容”的结构，而不是只调透明度或阴影。

### P0-2 `/result` 三栏空间分配失衡

- 现象：当前 top bar、当前 Day、右侧 inspector 都想成为核心，导致地图和日程都不够舒展。
- 影响页面 / 模块：`/result` desktop。
- 为什么影响体验：核心页面应该先让用户读懂行程和路线，而不是先面对一组密集工具区。
- 建议方向：重新定义三栏优先级，优先保证当前 Day 与地图的阅读面。

### P0-3 地图区域过小

- 现象：地图卡高度基线仅 `280px`，在右栏中视觉占比偏小。
- 影响页面 / 模块：`/result` 右侧 inspector。
- 为什么影响体验：地图已是 v1.5 核心能力，但当前展示方式仍像“证明我接上了”，而不是“帮助你决策”。
- 建议方向：地图至少应从“小预览”升级为“当前 Day 的主要辅助视图”。

### P0-4 首屏信息层级混乱

- 现象：打开 `/result` 后，top bar 事实卡、工具按钮、Day tabs、当前 Day、右侧 inspector 同时争主位。
- 影响页面 / 模块：`/result` desktop 首屏。
- 为什么影响体验：用户第一眼不知道该先看总览、看 Day、看地图，还是先处理保存和导出。
- 建议方向：明确首屏默认任务，减少同层级高权重模块数量。

### P0-5 登录保存主链路感知偏弱

- 现象：保存能力已经可用，但保存、更新、已保存、需登录的提示主要集中在按钮附近的小文案。
- 影响页面 / 模块：`/result`、`/login`、`/trips`。
- 为什么影响体验：v1.6 已进入“云端持久化”阶段，但产品感知仍像附加能力，不像核心闭环的一部分。
- 建议方向：把保存链路提升为更明确的状态系统，而不是按钮附属说明。

## D. P1 问题

P1 定义：不影响功能，但明显影响产品感和可理解性。

### 重点问题

- 天气展示优先级不对：现在更像“天气资料卡”，不是“出行建议卡”。
- marker / active marker 工程态偏强：可用，但不够自然、不够有旅行产品气质。
- 图标系统有基础但未统一成完整语言：特别是 sidebar、状态和功能入口之间。
- 保存 / 更新 / 打开状态表达仍偏轻：行为结果不够醒目。
- `/trips` 卡片信息层级需要更强主次：尤其标题、日期、最近更新时间和动作关系。
- 登录页体验偏绕：magic link 合理，但页面叙事与预期管理不足。

### 其他 P1 问题

- `/create` 仍有过时 phase 文案，削弱成熟度。
- `workspace-panel`、soft panel、普通卡混用，组件语义边界不清。
- itinerary block 的标签、动作、详情入口较多，视觉密度偏碎。
- route inspector 内部统计、地图、点位、路线段同层，右栏读起来累。
- desktop 大修时 mobile 入口极易被边缘化，必须阶段性回归。

## E. P2 问题

P2 定义：提升精致度、情绪价值、记忆点。

### 重点问题

- 欢迎页缺少更强的图片 / 插画 / 旅行氛围。
- 空状态模块缺少统一的轻视觉承载。
- 当前动效整体克制是对的，但缺少少量有意义的进入与切换反馈。
- 手帐风已有纸张纹理和暖色基础，但还没形成真正的产品记忆点。
- 色彩层次偏保守，页面之间的情绪差异不够。

### 其他 P2 问题

- Header 品牌记忆点一般。
- 中英文混排的风格还不够统一。
- 字体层级、说明文案密度、卡片留白还可以再收细。

## G. 色彩与视觉方向

### 当前色彩为什么显得单调

当前全局 token 已有不错的方向：

- 纸张色
- 浅沙色
- 浅鼠尾草绿
- clay 暖色强调

问题不是“颜色错了”，而是“颜色之间的层次和情绪差还不够大”：

- 大多数模块都落在类似明度区间，页面容易显得平
- accent 色使用谨慎，导致重点信息和普通信息拉不开
- 地图、天气、保存状态、空状态等本应具备明显语义差异的模块，颜色上仍偏同质
- landing 虽然暖，但第一眼没有形成鲜明记忆点

### 用户明确偏好的方向

用户这轮已经把视觉偏好收得更具体：

- 浅色
- 水彩
- 旅行手帐
- 小清新
- 轻松治愈
- 纸张感
- 贴纸感
- 手写分块
- 轻卡通元素

这意味着 v1.7 不是简单“把现有色板调亮一点”，而是要建立更明确的浅色水彩手帐语言。

### 建议的底色与强调层关系

建议保留米黄纸张底色作为基础承托，再引入低饱和的：

- 浅粉
- 浅绿
- 浅黄
- 浅蓝

作为补充强调层。它们不应该平均铺满页面，而应服务于重点模块和情绪节点。

### 哪些模块最需要色彩层级

- `/result` top bar 与当前 Day 工作区
- 天气模块
- 保存 / 已保存 / 更新 / 错误状态
- 地图 marker / active marker
- `/trips` 卡片动作区
- landing 首屏氛围区
- 空状态 / 错误状态

### 哪些地方适合更“旅行手帐 / 小清新 / 轻松治愈”的风格

- landing 首屏背景与配图承载
- `/result` 当前 Day 标题区与地图联动焦点
- 天气提醒卡
- 空状态卡与轻提示卡
- marker / point detail / route reminder 的微视觉

### 减少大面积黑色块

用户明确不喜欢大面积黑色块。除文字外，页面应尽量避免大面积纯黑 CTA 或黑色功能块，因为它会把产品气质拉向工具站，而不是旅行手帐。

这不代表不能有深色文字或深色边框，而是：

- 主 CTA 不应继续主要依赖黑底
- 强调层可以转向浅色底、深色字、边框和轻水彩强调
- 重点应放在“有层次的柔和强调”，而不是“强对比的大黑块”

### 水彩笔触式强调层

用户希望探索水彩笔触作为关键强调层，而不是全局样式替代。适合的使用位置包括：

- 当前 Day
- active tab
- 天气提醒
- active marker
- landing 主 CTA 周边
- 重要提示或轻贴纸式强调

不适合的使用位置包括：

- 删除按钮
- 高频密集的小按钮
- 所有 chip
- 所有列表项

这类水彩层应服务于记忆点，而不是变成新的噪音来源。

### 视觉方向关键词建议

#### 方向 1：纸感手帐

- 关键词：纸感、便签、手帐、暖白、轻贴纸
- 优点：最贴近当前已有视觉资产，重构成本可控
- 风险：做不好会像文具模板，缺少旅行感
- 适合模块：`/result` 卡片系统、预算、天气、导出区、空状态

#### 方向 2：清新城市散步

- 关键词：清新、轻风、城市漫步、低饱和、留白
- 优点：更轻、更现代，适合路线与天气这种信息密度较高的页面
- 风险：如果过度收敛，容易丢掉“旅行手帐”的人格
- 适合模块：地图、route inspector、top bar、`/trips`

#### 方向 3：治愈旅途便签

- 关键词：治愈、柔和、旅行感、小票、轻胶带
- 优点：情绪价值强，适合 landing 与 onboarding
- 风险：如果装饰过多，会影响信息清晰度
- 适合模块：欢迎页、create、login、empty state

### 欢迎页与 explore 的视觉处理建议

欢迎页和 explore / inspiration 区域可以比工作台更拼贴、更插画化，因为它们承担的是氛围建立和灵感唤起，而不是高密度编辑任务。

建议方向：

- 欢迎页主视觉采用真实景点图
- 背景使用水彩手帐风纹理与色块
- 少量旅行拼贴作为辅助点缀
- 景点、美食、灵感卡可借鉴旅行记录、拼贴卡、手写分块和轻卡通元素

但仍需控制：

- 不做满屏贴纸墙
- 不让装饰压过 CTA 和主标题
- 不让 explore 的视觉语言污染 `/result` 主工作台

### `/result` 工作台应更克制

`/result` 可以吸收浅色水彩手帐语言，但必须明显比欢迎页更克制。原因是：

- 它承担阅读、编辑、保存、路线判断等高密度任务
- 需要清楚的一级、二级、三级信息层级
- 装饰一旦过量，会直接影响 itinerary、地图和天气的理解效率

因此，欢迎页 / explore 可以更像拼贴海报，而工作台应更像“有手帐气质的可编辑工具台”。

### 色彩建议总结

- 不建议推翻现有暖纸张基底
- 用户偏好已明确偏向浅色水彩手帐，而不是深色工具面板
- 应在现有基底上强化重点模块的语义色差
- 欢迎页主视觉应由真实景点图主导，水彩背景和拼贴元素做辅助
- 应减少大面积黑色 CTA，改用更柔和的浅色强调方式
- 水彩笔触应作为关键强调层，而不是全局默认样式
- 应优先优化天气、保存状态、active marker、空状态这几类“需要被一眼看懂”的模块
- 欢迎页 / explore 可以更拼贴、更插画化，`/result` 工作台则要更克制
- phase 0 不进入具体色板落地，但 v1.7 应明确建立“基础底色 + 功能语义色 + 情绪强调色”的分层体系

## H. 信息层级审计

### `/result` 打开后，用户第一眼应该看到什么

用户第一眼应该看到：

1. 这趟旅行的总目标和当前在看的 Day
2. 今天怎么走
3. 地图和路线是否顺
4. 是否需要处理保存或修改

### 一级信息

- 当前 Day
- 当前 Day 的路线主题
- 地图 / 路线可读结果
- 明确的天气出行提醒
- 保存状态

### 二级信息

- 全局 trip summary
- 预算摘要
- 住宿与交通建议
- 待修改清单

### 三级信息

- 开发语境说明
- 解释性长文案
- 低频操作
- 次要统计 chip

### 当前页面是否把次要信息做得太重

是。

当前存在两个典型问题：

- 工具按钮和说明型卡片比核心行程更抢眼
- 统计 chip、补充说明和 section 标题数量偏多，导致视线被不断切碎

### 建议如何调整

- 收缩首屏 CTA 数量
- 让当前 Day 与地图形成真正的“双核心”
- 弱化解释性长文案，把更多解释放进二级层或可折叠区
- 让保存状态从“附属小字”升级为“明确状态反馈”

## I. 布局审计

### desktop `/result` 三栏布局的问题

- sidebar hover 扩展与内容列宽存在冲突
- 主内容区承担了过多模块职责
- inspector 固定宽度下地图与列表都被压缩
- top bar 过厚，挤占首屏内容

### 左侧导航问题

- 结构上属于覆盖式展开
- 信息上中英混排与功能状态混合，略显杂
- “Soon” 与真实可用入口并置，容易产生噪音

### 右侧地图和 inspector 的空间问题

- 地图区太小
- 点位列表、路线段、提醒栈都是竖向堆叠，右栏拉得很长
- 右栏缺少更明确的分层节奏

### 不同宽度下可能出现的问题

- 常见 1280px 左右宽度下，主区与 inspector 都容易偏窄
- sidebar hover 对中等宽度尤其不友好
- top bar 多按钮在部分宽度下容易换行，造成视觉碎裂

### mobile 不做深重构，但需要回归检查

- 当前 mobile 已有分页浏览结构，基础不错
- 但 desktop 大修容易让内容优先级再次偏向 desktop，导致 mobile 的 route / edit / export 感知变弱
- 建议每个 phase 都做移动端截图级回归，而不是最后统一回归

## J. 组件一致性审计

### 卡片圆角

- 当前有 28px、24px、20px、18px、16px 等多档并存
- 方向不算错，但语义边界不清

### 阴影

- 有纸张感，但不同卡片使用的阴影策略差异较大
- 一级卡与二级卡的区别还可以更明确

### border

- `line`、`line-strong`、dashed、实线混用较多
- 当前更像“局部可用”，不是完整规则

### chip

- 基础样式统一，但语义色不完整
- 统计 chip 数量偏多时会放大页面噪音

### button

- 主按钮语言较稳定
- 次按钮、danger 按钮、disabled 状态还不够统一

### hover / active

- sidebar、day tab、按钮、block 的 active 规则不完全一致
- active marker 与 active block 也没有形成统一语义

### icon

- 自绘 icon 可用，但与产品气质仍有距离
- 页面间未形成完整体系

### loading / error / empty state

- 基本覆盖到位
- 但视觉与语气没有完全收成一套

### 中英文混排

- `PLAN BAR`、`CURRENT DAY`、`Desktop Workspace`、`active day sync` 风格不统一
- 有些地方偏产品，有些地方偏内部标注

### 文案语气

- 口语化总体方向是对的
- 但部分页面仍残留 phase/版本推进式表达，需要清掉

## K. v1.7 建议原则

1. 先解决遮挡和空间，再做装饰。
2. 地图要从“预览”变成“可读工具”。
3. 天气要从“气象字段”变成“旅行建议”。
4. 状态表达必须统一，尤其是保存、更新、打开、错误和空状态。
5. 保留纸张感，但减少工程感和阶段 demo 感。
6. 视觉升级不等于堆装饰，信息清晰度优先。
7. 不新增复杂功能，先把已有能力做好看、做好懂、做好用。
8. desktop 大修时每阶段都做 mobile 回归。
9. v1.7 desktop 优先，mobile 只做基础回归，不做深度重构。
10. 欢迎页要从文字说明页升级为产品海报页。
11. 欢迎页主视觉采用真实景点图 + 水彩手帐背景 + 少量旅行拼贴。
12. 结果页采用双栏工作台：左侧编辑行程，右侧放大地图，中间比例可调。
13. `/result` 默认比例定为左侧 55%、右侧 45%。
14. 保留米黄纸张底色，减少大面积黑色块，用低饱和水彩和贴纸感建立记忆点。
15. 手帐感不能牺牲信息清晰度。
