# V1.7 Phase Plan

> 文档状态：v1.7 历史阶段计划
> 本文保留 v1.7 规划阶段的阶段拆分、优先级与验收设想。实际实施过程中，为支撑目标 UI，同时补入了必要的技术、数据与页面基础能力。v1.7 最终实际完成状态请以 `docs/V1.7_RELEASE_NOTES.md` 为准；后续精修进入 v1.8。

## 总目标

在不改业务主链路、不改数据库结构、不改 TripPlan schema 的前提下，完成一次围绕现有 v1.6 完成态的 UI / UX 大修。

## 版本边界补充

- `v1.7` 以 desktop UI / UX 大修为主
- mobile 本轮只做基础回归
- 移动端深度优化顺延到 `v1.9`
- 优先修 desktop 使用和观感、欢迎页、`/result` 工作台、地图、天气和浅色手帐视觉

## Phase 0：UI 审计与设计方向冻结

### 目标

- 读取现有页面和组件实现
- 输出 UI 审计、PRD 和后续阶段计划
- 冻结 v1.7 的问题边界、优先级和设计原则

### 允许改什么

- 文档

### 禁止改什么

- 页面代码
- 组件逻辑
- CSS / Tailwind
- API
- 数据库
- schema

### 验收标准

- `docs/UI_AUDIT_v1.7.md` 完成
- `docs/PRD_v1.7.md` 完成
- `docs/V1.7_PHASE_PLAN.md` 完成
- README / roadmap / 手动验收文档同步

### 是否需要用户确认

- 需要
- phase 0 产出应先由用户人工筛选和冻结后，再进入 phase 1

## Phase 1：欢迎页与过期入口收口

### 目标

- 欢迎页从文字说明页改为产品海报式首页
- 减少文字，强化主视觉、主标题和 CTA
- 欢迎页主视觉采用真实景点图 + 水彩手帐背景 + 少量旅行拼贴
- 去除或弱化欢迎页无用左侧导航
- 清理 `/create`、`/login`、`/trips` 中 phase / 版本推进式文案
- 处理 `/create` “自己慢慢挑 / 地图探索 v1.5”这类无实际功能入口

### 允许改什么

- `/` 欢迎页布局与文案
- landing 首屏视觉结构
- `/create` 页面过期入口与说明文案
- `/login`、`/trips` 中明显 phase 语境文案
- 轻量图片 / 插画占位结构，但不接内容生态

### 禁止改什么

- 不新增内容生态
- 不新增地图探索真实能力
- 不新增复杂营销模块
- 不改登录逻辑
- 不改数据库
- 不改 AI / Weather / Map / Route 主链路

### 验收标准

- 欢迎页首屏有明确视觉重心，不再像大段文字广告
- 欢迎页主视觉符合真实景点图 + 水彩手帐背景 + 少量旅行拼贴
- 用户能一眼知道产品是“把旅行想法变成可修改行程”
- 欢迎页无效左侧导航被去除或明显弱化
- `/create` 不再展示过期或无实际功能的 v1.5 / v1.6 占位入口
- 页面文案不再像内部开发阶段说明

### 是否需要用户确认

- 需要
- 欢迎页方向变化较大，建议完成后先给用户看，再进入 result 工作台大修

## Phase 2：/result 双栏工作台与导航修复

### 目标

- 修复左侧导航遮挡
- `/result` desktop 改为左右双栏
- 左侧行程工作区，右侧大地图和辅助信息
- 默认比例左侧 `55%`、右侧 `45%`
- 中间可拖拽分隔线
- 减少两侧空白
- 检查并修复 itinerary block “查看详情”无响应

### 允许改什么

- `/result` 页面布局
- `DesktopWorkspaceShell`
- sidebar 宽度、固定策略与展开策略
- 主工作区与地图区列宽
- 中间 resizable divider
- itinerary block 详情入口的 UI 响应问题
- 工作台安全边距与首屏视觉重心

### 禁止改什么

- 不改地图数据逻辑
- 不改路线洞察数据逻辑
- 不改保存 / 更新 / 删除逻辑
- 不改 TripPlan schema
- 地图选点
- 不做拖拽 itinerary block
- 自动重排

### 验收标准

- 左侧导航不再遮挡主内容
- `/result` 默认左侧 `55%`、右侧 `45%`
- `/result` 默认地图明显大于 v1.6，不再只是右侧小预览
- 用户可以通过中间分隔线调整左侧行程区和右侧地图区占比
- 当前 Day、地图和行程积木形成清楚的主工作区
- 工作台两侧空白明显减少
- “查看详情”入口可点击并有明确反馈
- 登录、保存、打开、更新、删除能力不受影响

### 是否需要用户确认

- 需要
- 这是 v1.7 的核心结构变动，必须用户确认后再进入后续视觉细化

## Phase 3：地图 marker 与天气旅行化

### 目标

- 优化 marker / active marker 的视觉表达
- 让地图点位更自然、更有旅行产品气质
- 将天气模块从气象字段展示改为旅行建议
- 优先表达气温、是否下雨、是否带伞、是否适合户外、极端天气预警
- 弱化普通风力、湿度、气压等低优先级信息

### 允许改什么

- marker / active marker 样式
- 地图点位状态表达
- 天气卡片布局
- weather summary 展示优先级
- 天气提醒文案
- 地图空状态与天气空状态表达

### 禁止改什么

- 不改高德地图 provider
- 不改和风天气 provider
- 不改天气接口协议
- 不改 weather schema
- 不做地图选点
- 不做路线自动重排
- 不做复杂专项天气推理

### 验收标准

- active marker 更容易识别，但不再像工程调试态
- 地图点位和列表之间对应关系更自然
- 用户第一眼能判断天气是否影响当天行程
- 天气卡优先展示气温、下雨、带伞、户外适宜度和极端天气
- 普通风力、湿度、气压不再抢主要位置

### 是否需要用户确认

- 建议需要
- 天气层级和 marker 样式可给 1-2 套方案供用户选择

## Phase 4：浅色手帐视觉系统与组件一致性

### 目标

- 建立更统一的浅色手帐视觉语言
- 保留米黄色纸感底色
- 增加浅粉、浅绿、浅黄、浅蓝等低饱和强调色
- 减少大面积纯黑色 CTA 和黑色块
- 探索水彩笔触式强调层
- 统一 button / chip / card / icon / empty / error / loading 的表达
- 统一 sidebar 图标体系
- 优化保存 / 打开 / 更新 / 删除状态

### 允许改什么

- 全局视觉 token
- button / chip / card 视觉层级
- 主 CTA 视觉样式
- 状态标签
- sidebar 图标
- 保存 / 打开 / 更新 / 删除状态表达
- 空状态与错误状态
- 水彩笔触式强调层的轻量实现方案

### 禁止改什么

- 不把所有按钮都做成水彩样式
- 不引入重装饰导致信息不可读
- 不改核心交互流程
- 不改 API 请求逻辑
- 不改数据结构
- 不引入复杂插画系统

### 验收标准

- 页面整体更浅、更柔和、更像旅行手帐
- 主 CTA 不再依赖大面积黑底
- 水彩笔触只用于关键强调，不造成视觉噪音
- 按钮、chip、卡片、状态和图标更统一
- 页面之间不再像多个 phase 拼起来的临时界面

### 是否需要用户确认

- 需要
- 水彩笔触和浅色强调策略需要用户确认后再大面积使用

## Phase 5：手帐氛围、插画与整体 polish

### 目标

- 强化欢迎页和 explore / inspiration 的旅行手帐氛围
- 引入轻卡通、小贴纸、手写分块、水彩色块
- 景点、美食、灵感卡更像旅行记录或手帐拼贴
- 优化示例 prompt 和 CTA
- 做整体间距、字体层级、文案语气 polish

### 允许改什么

- landing 视觉编排
- inspiration / travel mood 卡片视觉
- 示例 prompt 展示方式
- 轻量卡通图案或贴纸感装饰
- 手写分块风格的局部表达
- 全局文案语气微调
- 背景纹理与轻量装饰

### 禁止改什么

- 不新增内容生态真实能力
- 新增复杂营销模块
- 新增登录门槛
- 不新增真实 explore 数据库
- 不让装饰影响工作台可读性

### 验收标准

- 首屏更有吸引力
- 仍然一眼看懂产品做什么
- inspiration / explore 区域更有旅行氛围
- 手帐感和水彩感更明显，但不喧宾夺主
- 页面整体视觉更有记忆点

### 是否需要用户确认

- 需要
- 视觉方向建议在这一阶段再次对齐

## Phase 6：回归验收与发布准备

### 目标

- 跑全流程回归
- 确认 UI 改造不破业务
- 做发布前收口

### 允许改什么

- 回归修复
- 文案微调
- 样式微调
- 明确的视觉 bug 修复

### 禁止改什么

- 临时扩 scope
- 新加功能
- 改主链路

### 验收标准

- `/create /plan /result /trips /login` 全流程通过
- desktop 体验优先确认通过
- 登录保存不破
- 地图天气不破
- mobile 只要求不严重破版
- `npm.cmd run lint`
- `npm.cmd run build`
- `npm.cmd run typecheck`
- `npm.cmd test`

### 是否需要用户确认

- 需要
- 建议作为 v1.7 最终验收节点
