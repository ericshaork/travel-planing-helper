# V1.8.3 Workspace Step 2 Entry And Data Map Plan

## 1. Purpose

This document is a planning artifact for Step 2 only.

- It is based on the Step 1 audit and current codebase state.
- It does not change business code.
- It does not include commit or push actions.
- Its goal is to define how Workspace entry flows, route ownership, trip identity, and map workspace responsibilities should be unified before implementation starts.

## 2. Product Correction From Step 1

Step 1 confirmed the core correction:

- Workspace is not "left trip detail page + right small map plugin".
- Workspace is "left plan workspace + right full map workspace".

That correction matters because the current right side is still organized as:

- `MapPanel`
- `WorkspaceInspector`
- `InspectorMapPreview`
- stats / warnings / point list / route legs

This means the current product shape is still route inspector first, map second.

Step 2 therefore needs to lock two product truths before UI refinement:

1. `/workspace` must become the single formal workspace route.
2. The right pane must be treated as a first-class map workspace, not a preview card inside an inspector stack.

## 3. Target Workspace Model

The target model for v1.8.3 should be:

- Left Plan Pane
  - trip title and archive context
  - Day tabs
  - day timeline
  - place cards
  - read-first trip narrative
  - future block editing entry
- Right Map Workspace Pane
  - full-height map container
  - day-aware markers
  - selected place state
  - left/right selection sync
  - route understanding surface
  - reserved toolbar space for search, distance, route mode, optimization, and layers

Recommended interaction default:

- AI-generated trip: enter `Read Mode`
- saved trip from history: enter `Read Mode`
- blank manual trip: enter `Edit Mode` or edit-ready empty state

Reason:

- Read Mode works when there is already readable itinerary content.
- Blank trip starts from creation work, not reading work.

## 4. Three Entry Flows

### 4.1 AI Generated Trip

Current state:

- Current route chain is `/create` -> `/plan` -> `/workspace`.
- `app/plan/page.tsx` already pushes to `/workspace` after successful generation.
- `components/workspace/WorkspaceRoutePage.tsx` reads trip state from `localStorage` via `loadTripPlan()` and `loadTripRequest()`.

Target state:

- AI generation remains the cleanest and primary entry into `/workspace`.
- Generated trip should arrive with explicit workspace identity metadata, not just shared storage payloads.
- Entry should carry source semantics such as `sourceType = "ai-generated"`.

Recommended default mode:

- `Read Mode`

Data that should enter Workspace:

- normalized `tripRequest`
- normalized `tripPlan`
- optional `tripEnrichment`
- optional weather summary
- workspace session identity
- source type

Likely files to change later:

- `app/plan/page.tsx`
- `components/workspace/WorkspaceRoutePage.tsx`
- `hooks/useTripWorkspaceState.ts`
- `lib/trip/storage.ts`

### 4.2 Saved Trip / My Trips / History

Current state:

- `/trips` uses `openSavedTripIntoWorkspace(trip.id)`.
- `lib/trips/open-flow.ts` restores trip data into `localStorage`.
- It then navigates to `/result`, not `/workspace`.
- The saved trip is therefore not opened through a route-owned workspace identity. It is restored into shared client storage and then shown through the legacy result entry.

Target state:

- saved trip entry should route directly into `/workspace`
- `/workspace` should know this is a saved trip session
- the session should carry `savedTripId`
- local editable state can still hydrate from local cache, but route ownership should no longer depend on `/result`

Recommended default mode:

- `Read Mode`

Data that should enter Workspace:

- `savedTripId`
- normalized `tripRequest`
- normalized `tripPlan`
- restored enrichment if available
- weather summary if available
- saved metadata such as `savedAt`

Likely files to change later:

- `lib/trips/open-flow.ts`
- `lib/trips/restore-saved-trip.ts`
- `app/trips/page.tsx`
- `components/trips/SavedTripCard.tsx`
- `components/workspace/WorkspaceRoutePage.tsx`
- `lib/trip/storage.ts`

### 4.3 Blank Manual Trip

Current state:

- there is no real blank-workspace entry today
- `app/create/page.tsx` has `self-directed`, but it is placeholder-only
- `WorkspaceRoutePage` empty state links to `/create` and `/explore`
- no blank draft is created
- no blank draft identity exists

Target state:

- user can start from zero and enter `/workspace`
- system creates a lightweight blank workspace draft before render
- blank trip uses the same workspace shell as AI and saved trips
- blank trip gets a distinct identity and source type from the start

Recommended default mode:

- `Edit Mode`

Data that should enter Workspace:

- `blankDraftId`
- minimal `tripRequest` shell or editable request draft
- minimal workspace draft structure
- empty day list or starter day scaffold
- source type such as `blank-manual`

Likely files to change later:

- `app/create/page.tsx`
- potential future `app/workspace/page.tsx` entry handling
- `components/workspace/WorkspaceRoutePage.tsx`
- `hooks/useTripWorkspaceState.ts`
- `lib/trip/storage.ts`
- `lib/trips/open-flow.ts`

## 5. Route Consolidation Strategy

### 5.1 Current state

Current workspace-related routes are:

- `/workspace`
- `/result`

Both render the same shell through `WorkspaceRoutePage`, but route responsibility is not unified:

- AI entry already uses `/workspace`
- saved trip entry still uses `/result`
- several product narratives still describe `/result` as the active destination

### 5.2 Recommended route ownership

Formal route:

- `/workspace`

Compatibility route:

- `/result`

Meaning:

- `/workspace` becomes the official workspace address for all new product flows
- `/result` remains as backward-compatible alias during migration
- `/result` should not be deleted in this phase
- `/result` should stop receiving new product responsibilities

### 5.3 Short-term migration

Short-term goal for Step 2 and Step 3:

- move all real entry actions to `/workspace`
- keep `/result` rendering the same shell for compatibility
- reduce `/result` language in UI copy and helper flows

### 5.4 Mid-term migration

Mid-term goal after Workspace stabilizes:

- keep `/result` as passive compatibility entry
- optionally redirect it later after analytics and old-link risk are understood
- remove result-page mental model from product copy and internal naming where practical

## 6. Trip Identity And Storage Model

### 6.1 Current state

Current identity is weak and mostly storage-driven.

Today the workspace primarily reads:

- `travel-planning:trip-plan`
- `travel-planning:trip-request`
- `travel-planning:trip-enrichment`
- `travel-planning:trip-weather-summary`
- `travel-planning:restored-saved-trip`

Current saved metadata only clearly carries:

- `savedTripId`
- `savedTripTitle`
- `restoredAt`
- `savedAt`

Current gaps:

- no `localDraftId`
- no `blankDraftId`
- no unified workspace session identity
- no route-level trip identity
- no strong distinction between generated, restored, and blank sessions

### 6.2 Recommended identity fields

Recommended workspace identity model:

- `sourceType`
  - `ai-generated`
  - `saved-trip`
  - `blank-manual`
- `savedTripId`
  - present when opened from saved cloud trip
- `localDraftId`
  - present when user is editing a local unsaved working draft
- `blankDraftId`
  - present when user enters from zero before any save
- `workspaceSessionId`
  - optional client session identifier for UI restore and diagnostics

Recommended rule:

- only one primary trip identity should own the session at a time
- `savedTripId` and `blankDraftId` should not compete
- `localDraftId` should represent editable local state, not cloud ownership

### 6.3 URL and storage relationship

Recommended ownership split:

- URL identifies which workspace entry context the user is opening
- localStorage holds the latest editable client payload
- Supabase holds persisted saved trips

Practical interpretation:

- URL should not try to serialize the full trip
- localStorage should not be the only source of identity truth
- Supabase should not be read on every minor interaction once workspace is open

### 6.4 Save logic recommendation

Recommended create/update behavior:

- if `savedTripId` exists: save updates current saved trip
- if only `localDraftId` exists: first explicit save creates a saved trip and binds returned `savedTripId`
- if only `blankDraftId` exists: first explicit save creates a saved trip and replaces blank-only ownership with persisted identity

### 6.5 localStorage and Supabase relationship

Recommended clarity:

- Supabase is persistence for saved trips
- localStorage is active workspace cache
- a restored saved trip may hydrate localStorage for interaction speed
- but identity should still be anchored by saved trip metadata, not inferred only from reused browser keys

## 7. Blank Draft Model

### 7.1 Why blank draft is required

Step 1 identified this as a P0 gap.

Without a blank draft model:

- the third required entry flow does not exist
- Workspace cannot act as the universal trip workbench
- manual planning cannot share the same shell as AI and history flows

### 7.2 Recommended minimal blank draft fields

Recommended minimal blank draft payload:

- `blankDraftId`
- `sourceType = "blank-manual"`
- `title`
- `destinationCity`
- `startDate`
- `endDate` or `tripDays`
- `dailyItinerary`
- `notes` or starter description
- `updatedAt`

Recommended empty defaults:

- title: `"Untitled trip"`
- destination: empty
- dates: empty
- daily itinerary: empty array or one starter day scaffold
- enrichment: absent
- weather summary: absent

### 7.3 Recommended blank entry behavior

Flow recommendation:

1. user clicks manual / start from zero
2. app creates blank draft identity
3. app stores editable starter state locally
4. app opens `/workspace`
5. workspace enters `Edit Mode`

### 7.4 Save recommendation

Blank draft save should:

- create first persistent trip record
- receive `savedTripId`
- convert workspace ownership from blank draft to saved trip
- keep local cache updated after save

## 8. Right Map Workspace Structure

### 8.1 Current state

Current right side is not a map workspace.

It is an inspector composition:

- `MapPanel`
- `WorkspaceInspector`
- `InspectorMapPreview`
- detail card
- route stats
- point list
- route legs
- warnings

Current underlying map capability does exist:

- `AmapBaseMap`
- zoom/pan from SDK
- markers
- active marker state
- marker click handling
- fit-to-markers behavior

### 8.2 Target structure

Recommended right-pane structure:

- Map Workspace Header
  - current day context
  - marker count
  - lightweight workspace actions
- Full Map Canvas
  - dominant vertical area
  - not constrained to preview height
- Selection Detail Band
  - selected place summary
  - route context
  - unresolved status or notes
- Secondary Foldable Utility Surfaces
  - warnings
  - route stats
  - place list
  - route legs

This keeps the map as the main surface and moves inspector content into supporting roles.

### 8.3 Capability assessment

Already implemented:

- real map SDK container
- marker rendering
- current-day point rendering
- active point highlight
- marker click callback
- left/right selection sync foundation

Partially present at infrastructure level:

- zoom / pan via underlying SDK
- route distance / duration data in enrichment

Missing as product surfaces:

- dedicated map toolbar
- search entry
- layer toggle
- locate entry
- route mode entry
- two-point distance / time query entry
- multi-point route optimization entry
- in-map or bottom detail workspace surface

### 8.4 Reservation strategy for v1.8.3

v1.8.3 does not need to fully implement advanced map tools, but Step 2 should reserve layout and state seams for:

- search
- distance/time query
- route mode
- layer control
- future optimization tools

Recommendation:

- reserve a top toolbar zone inside the right workspace
- reserve a bottom detail/info band
- collapse non-essential inspector stacks behind drawers or foldable sections

## 10. Risks And Guardrails

### Risks

- route migration may accidentally break saved trip opening if `/result` is removed too early
- identity refactor may create duplicate ownership between saved trips and local drafts
- blank draft support may fork schema shape if it is introduced outside the existing trip model
- map workspace refactor may over-focus on layout and miss day/place/marker sync stability
- enrichment dependency may cause map area to feel broken when enrichment is late or unavailable

### Guardrails

- do not delete `/result` during this phase
- do not create a second incompatible workspace schema for blank trips
- do not make map enrichment the only condition for rendering a usable workspace
- keep AI-generated, saved, and blank trips inside one workspace shell
- keep the left/right sync model explicit before adding advanced map tools

## 11. Recommended Step 3-6 Execution Plan

### Step 3: Workspace entry and data-flow consolidation

- formalize `/workspace` as the single official route
- move saved trip open-flow from `/result` to `/workspace`
- define unified workspace identity for AI, saved, and blank entries
- add blank draft creation path

### Step 4: Workspace main layout correction

- preserve left/right split near `55 / 45`
- keep left pane as plan-first narrative surface
- convert right pane from inspector stack to full map workspace shell
- demote stats / warnings / route-leg blocks to secondary surfaces

### Step 5: Read Mode skeleton and map workspace foundation

- refine Read Mode so it feels like archive reading plus full map reference
- complete Day / Place / Marker sync as a first-class behavior
- stabilize selected place state across left and right panes
- reserve toolbar and detail-band seams for future search, distance, and optimization

### Step 6: Minimum viable Edit Mode and closeout

- blank trips enter edit-first
- saved and AI trips remain read-first
- define minimal editable workspace behavior without requiring full AI regeneration for every action
- remove leftover result-page and inspector-only framing where possible

## Summary

Step 2 should not be treated as a cosmetic planning exercise.

It needs to settle four structural decisions before implementation:

1. `/workspace` becomes the formal route owner.
2. AI, saved, and blank manual trips all enter the same workspace shell.
3. trip identity must become explicit enough to support `savedTripId`, `localDraftId`, and `blankDraftId`.
4. the right side must be rebuilt as a full map workspace, not left as `InspectorMapPreview` inside an inspector stack.
