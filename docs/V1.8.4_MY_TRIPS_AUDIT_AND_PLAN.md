# V1.8.4 My Trips Audit And Plan

## 1. Scope

本阶段只做审计与实施方案，不修改业务代码、不改 migration、不改 API 行为、不改 Workspace 保存/恢复链路、不 commit、不 push。

审计目标：

- 明确当前 `trip_plans` 的真实字段与 RLS 状态；
- 明确 Trips API 当前已支持和未支持的能力；
- 明确 My Trips 页面与组件当前能做什么；
- 明确历史计划进入 `/workspace` 的真实链路；
- 为 Step 7B 提供一份可直接实施的边界清晰方案。

### 3.2 Missing Fields

当前 **没有** 看到以下字段：

- `status`

- `local_draft_id`
- `last_opened_at`
- `archived_at`

### 3.3 Current RLS

当前 `trip_plans` 已开启 RLS，且策略是标准 own-only：

- `trip_plans_select_own`
- `trip_plans_insert_own`

- `auth.uid() = user_id`

这意味着：

- 用户隔离已经有健康基础；
- 只要后续 Step 7B/8 继续沿用 `user_id` 过滤和 RLS，就不会破坏私有数据边界。

## 4. Current Trips API

### 4.1 Current Supported Capabilities

当前 API 已支持：

1. 列表：`GET /api/trips`
2. 创建保存：`POST /api/trips`
3. 打开单条：`GET /api/trips/[tripId]`
4. 更新整条计划内容：`PATCH /api/trips/[tripId]`
5. 删除：`DELETE /api/trips/[tripId]`

### 4.2 Current GET /api/trips

当前列表接口：

- 必须带登录 token；
- 未登录返回 401；
- 只查当前用户 `user_id`；
- 返回字段只有：
  - `id`
  - `title`
  - `destination_city`
  - `start_date`
  - `end_date`
  - `days`
  - `budget`
  - `cover_image_url`
  - `created_at`
  - `updated_at`
- 按 `updated_at desc` 排序；
- **不支持** 搜索；
- **不支持** 筛选；
- **不支持** 状态标签；
- **不支持** 来源标签；
- **不支持** 最近打开时间。

### 4.3 Current POST /api/trips

当前保存接口：

- 只保存当前完整 `tripRequest + tripPlan + enrichment/weather`；
- 由服务端使用 auth user id 写入 `user_id`；
- title 由 `tripPlan.tripTitle` 或目的地 fallback 推导；
- **不会写入** `source_type / status / trip_preferences_json / local_draft_id`；
- **没有首保存时 local draft -> cloud trip 的正式映射字段**。

### 4.4 Current GET /api/trips/[tripId]

当前详情接口：

- 必须登录；
- 只允许加载当前用户自己的 trip；
- 会返回完整 `trip_request_json / trip_plan_json / enrichment_json / weather_summary_json`；
- 这条链路已经足够支撑“从历史计划恢复到 Workspace”。

### 4.5 Current PATCH /api/trips/[tripId]

当前 PATCH：

- 语义是“用新的完整 trip 内容覆盖这条已保存计划”；
- 入参仍然是 `tripRequest + tripPlan + tripEnrichment`；
- 只返回 `tripId + updatedAt`；
- **不支持** 单独重命名；
- **不支持** 单独更新 `status`；
- **不支持** 单独更新 `source_type`；
- **不支持** 单独更新 `last_opened_at`；
- **不支持** metadata 级局部 patch。

### 4.6 Current DELETE /api/trips/[tripId]

当前删除：

- 是真实删除；
- 删除的是整条 `trip_plans` 行；
- 没有 archive / restore；
- 没有 soft delete；
- 没有 `deleted_at`。

## 6. Current Open-to-Workspace Chain

### 6.1 Actual Open Flow

当前历史计划打开链路是：

1. `/trips` 点击卡片
2. 调用 `openSavedTripIntoWorkspace(trip.id)`
3. `openSavedTrip()` 请求 `GET /api/trips/[tripId]`
4. `restoreSavedTripToStorage()` 把完整 trip 恢复进本地存储
5. 写入：
   - `trip-request`
   - `trip-plan`
   - `trip-enrichment`
   - `trip-weather-summary`
   - `restored-saved-trip`
   - `workspace-session`
6. `workspace-session.sourceType = "saved_trip"`
7. `workspaceModeDefault = "read"`
8. 最后导航到 `/workspace`

### 6.2 What This Means

当前历史计划打开 **不是** 路由参数驱动，也不是服务端会话驱动，而是：

- “先从 API 拉完整 trip”
- “再恢复进 localStorage”
- “再让 `/workspace` 读取本地状态”

这条链路的特点：

- 快
- 与现有 Workspace 架构兼容
- 但 cloud trip 和 local draft 的边界仍然偏弱

### 6.3 Local Draft Impact

打开历史计划会直接把：

- `trip-request`
- `trip-plan`
- enrichment/weather

写回当前本地存储。

也就是说：

- 它会覆盖当前本地工作态内容；
- 当前并没有“先比较冲突再选择合并”的机制；
- 这也是为什么 Step 8 更适合承接“保存 / 恢复 / 草稿合并”。

### 6.4 /result Compatibility

当前 `/result` 没被删除。

它仍然复用：

- `WorkspaceRoutePage mode="result"`

而 `WorkspaceRoutePage` 本质上还是从 localStorage 读取：

- `loadTripPlan()`
- `loadTripRequest()`

因此：

- `/workspace` 和 `/result` 现在仍共享同一套本地恢复事实来源；
- 任何对历史计划恢复链路的重构，都不能误伤 `/result` 兼容入口。

### 6.5 One Notable Mismatch

`openSavedTripIntoWorkspace()` 代码里当前已经导航到 `/workspace`，但 `tests/trips/open-flow.test.ts` 的断言仍写着 `/result`。

这说明：

- 历史测试/文档里还残留旧心智；
- Step 7B 如果要补 My Trips 增强，最好顺手清理这类“实现与测试叙述不一致”的地方；
- 但本阶段不改行为。

## 7. Main Gaps

当前 My Trips / `trip_plans` 的核心缺口有五类：

### 7.1 Missing Classification Fields

没有：

- `source_type`
- `status`
- `trip_preferences_json`
- `local_draft_id`
- `last_opened_at`

这直接限制了：

- 搜索/筛选
- 分类标签
- 最近打开排序
- local draft 与 saved trip 的对应关系
- 单篇计划偏好与长期默认偏好的边界

### 7.2 PATCH Semantics Too Heavy

当前 `PATCH /api/trips/[tripId]` 是“整条计划覆盖式更新”，而不是 metadata patch。

因此当前很难优雅支持：

- 重命名
- 归档/恢复
- 只更新 `last_opened_at`
- 只更新来源/状态

### 7.3 List API Too Thin

`GET /api/trips` 现在只适合“简单列表展示”，不适合：

- 搜索标题/目的地
- 按来源筛选
- 按状态筛选
- 按最近打开排序

### 7.4 UI Is Actionable But Not Manageable

当前 `/trips` 可以打开和删除，但还不能算“管理中心”。

它更像：

- 历史列表入口

而不是：

- 我的行程管理页

### 7.5 Local-vs-Cloud Truth Is Still Blurry

当前 saved trip 打开后，会把完整内容恢复到 localStorage；
但 `trip_plans` 里又没有：

- `local_draft_id`
- `status`
- `last_opened_at`

这使得后续若要做：

- “这条本地草稿是不是来自某条已保存计划”
- “这次保存是新建还是覆盖”
- “最近打开”

都会很吃力。

## 8. Step 7B Recommended Plan

Step 7B 推荐只做 **My Trips / trip_plans 管理增强本身**，不要抢 Step 8 的“保存/恢复/草稿合并”工作。

### 8.1 Recommended Migration

建议 **需要新增 migration**。

推荐在 `trip_plans` 上新增：

- `source_type text`
- `status text`
- `trip_preferences_json jsonb`
- `local_draft_id text`
- `last_opened_at timestamptz`

当前暂不建议在 Step 7B 就加：

- `archived_at`
- `deleted_at`

如果 Step 7B 只做轻量 archive，可先用 `status = 'archived'` 表示，不急着上软删除体系。

### 8.2 Recommended Field Meanings

#### `source_type`

建议值：

- `ai_generated`
- `blank_manual`
- `explore_imported`

用途：

- My Trips 来源标签
- 筛选
- 后续行为统计

#### `status`

建议值：

- `draft`
- `saved`
- `archived`

用途：

- 区分普通已保存计划与归档计划
- 支撑筛选和 archive/restore

#### `trip_preferences_json`

用途：

- 保存单篇计划偏好
- 和 `user_settings` 的长期偏好分层
- 为后续 AI merge 做准备

Step 7B 不要求真正接入 AI，只要求字段底座明确。

#### `local_draft_id`

用途：

- 建立本地 Workspace 草稿和云端 trip 的关联
- 为 Step 8 的保存/恢复/覆盖判断做准备

#### `last_opened_at`

用途：

- My Trips 最近打开排序
- 列表展示“最近打开”
- 用户中心更像正常产品

### 8.3 Recommended API Enhancements

Step 7B 建议增强：

1. `GET /api/trips`
   - 支持 `query`
   - 支持 `status`
   - 支持 `sourceType`
   - 支持按 `last_opened_at` / `updated_at` 排序规则

2. `PATCH /api/trips/[tripId]`
   - 支持 metadata 级 patch
   - 支持只改 `title`
   - 支持只改 `status`
   - 支持只改 `last_opened_at`
   - 支持只改 `trip_preferences_json`

3. `GET /api/trips/[tripId]`
   - 维持现状即可

4. `DELETE /api/trips/[tripId]`
   - Step 7B 可以先保留真实删除
   - archive/restore 用 `PATCH status`

### 8.4 Recommended Page Enhancements

Step 7B 的 My Trips 页面建议补：

- 搜索框
- 筛选栏
  - 全部
  - 已保存
  - 草稿
  - 已归档
  - AI 生成
  - 空白手搓
  - Explore 导入
- 来源标签
- 状态标签
- 重命名入口
- 最近打开时间展示
- 删除确认继续保留
- 空状态继续保留
- 错误状态继续保留

### 8.5 Recommended Open Flow Change In Step 7B

Step 7B **只建议轻改**：

- 打开历史计划成功后，顺手更新一次 `last_opened_at`

但不要在 Step 7B：

- 改写恢复模型
- 引入复杂 query-based workspace identity
- 改 `/workspace` 的事实来源
- 改 `/result` 兼容逻辑

这些都更适合留给 Step 8。

## 9. Step 7B Boundary

Step 7B 推荐只做：

- `trip_plans` 字段补齐
- Trips API metadata 能力增强
- `/trips` 搜索/筛选/重命名/标签增强
- 打开历史计划时更新 `last_opened_at`

Step 7B 不建议做：

- 草稿合并
- 冲突提示
- local draft 与 cloud trip 双向同步策略
- 打开历史计划前的选择器/差异对比
- AI 偏好或长期偏好接入生成链路
- 软删除体系
- 多版本历史

## 10. Step 8 Should Handle

以下内容建议明确留给 Step 8：

- local draft 与 saved trip 的正式映射策略
- 打开历史计划是否覆盖当前本地草稿的冲突处理
- 保存时新建/覆盖/另存为逻辑
- `local_draft_id` 真正接入 Workspace 保存链路
- saved trip / current draft / blank draft 三者关系梳理
- `/workspace` 与 `/result` 的最终兼容收口

## 11. Risks

Step 7B 最大风险有四个：

1. 把 metadata 增强和 Workspace 恢复链路重构绑在一起，导致范围失控。
2. 在没有 `local_draft_id` 保存逻辑配套完成前，误以为字段补齐就等于草稿问题解决。
3. 改动列表接口或 PATCH 语义时，误伤当前已可用的“保存 -> 打开 -> 删除”基础链路。
4. 为了做“正常用户中心”而过早引入软删除、多版本、复杂筛选，超出 v1.8.4 范围。

## 12. Recommendation Summary

结论：

- 当前 My Trips 能用，但仍是基础版；
- `trip_plans` 还缺少用户中心所需的关键 metadata 字段；
- Trips API 目前适合“整条计划保存/打开/删除”，不适合“细粒度管理”；
- `/trips -> /workspace` 的恢复链路已经成立，但应被视为 Step 8 的重点保护对象；
- Step 7B 应聚焦 “字段 + API metadata + 列表管理增强”，不要提前重构 Workspace 恢复真相。
