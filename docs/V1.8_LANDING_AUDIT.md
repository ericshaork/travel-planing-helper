# Wanderly v1.8 Landing 只读审计

> 文档状态：v1.8.1 Step 5 审计输出
> 审计时间：2026-07-11
> 审计范围：只读检查 Landing 当前代码、相关导航与资源，不修改业务代码
> 审计基线来源：`AGENTS.md`、`docs/PRD_v1.8.md`、`docs/V1.8_PHASE_PLAN.md`、`docs/V1.8_NAVIGATION_AND_STATE.md`、`docs/V1.8_DESIGN_SYSTEM.md`、`docs/V1.8_ASSET_MANIFEST.md`

## 1. 文档用途

本文用于回答三件事：

1. 当前 `Landing` 在代码里实际上是什么状态；
2. 当前实现与 `v1.8` Landing 基线之间差了什么；
3. 后续 Step 6 之前，技术、UI、图片分别该补什么输入。

本文只输出审计结论，不直接修改：

- 路由；
- 页面代码；
- 样式；
- 图片；
- 其他文档。

## 2. 审计基线

本次审计严格按以下产品基线判断：

- Landing 只承载首次品牌欢迎、产品解释、进入 Explore 或 Create 的分流；
- Landing 不是老用户日常首页，不是 Workspace 入口总枢纽，不是完整营销页；
- 主 CTA 应为 Explore，次 CTA 应为 Create；
- `/` 的目标行为应由首次引导状态和登录状态共同决定；
- `/welcome` 属于 v1.8 目标，当前代码里还不存在；
- 打开、刷新、关闭 Landing 不应自动完成 onboarding；
- 深层链接不应被强制拉回 Landing；
- v1.8 正式 Landing 内容范围应聚焦中国大陆 / 香港 / 澳门 / 台湾；
- 功能性文案应以中文为主，英文只能做少量装饰；
- 设计验收以桌面端 `1280 / 1440 / 1920` 为主，移动端只做不严重破版检查。

## 3. 当前代码结构

与 Landing 直接相关的当前结构如下：

- 根路由 `app/page.tsx`：当前直接返回 `LandingPage`；
- 主体页面 `components/landing/LandingPage.tsx`：当前只有首屏海报和 3 个 CTA；
- 全局头部 `components/layout/Header.tsx`：Landing 使用 `minimal` 头部；
- 用户状态入口 `components/auth/UserMenu.tsx`：根据登录状态切换为登录按钮或邮箱 + 退出；
- 全局布局 `app/layout.tsx`：仅提供 `html/body` 包裹；
- 全局样式 `styles/globals.css`：已存在暖纸张色系、面板语义和 reduced motion 降级；
- Landing 图片目录 `public/images/landing/*`：资源量大，但当前首页实际只显式使用 `hero-main.png`。

当前结构结论：

- Landing 已有单独组件；
- 但 Landing 目前仍是“固定页面”，不是“根路由状态分流层”；
- 页面内容体量明显不足以承载 v1.8 目标中的正式欢迎体验。

## 4. 当前路由与访问逻辑

### 4.1 当前代码状态

- `/`：当前固定渲染 Landing，没有 onboarding 判断；
- `/welcome`：当前不存在；
- `/explore`：当前已存在，且更接近日常首页；
- `/create`：当前已存在；
- `/workspace`：当前已存在；
- `/result`：当前已存在，且与 `/workspace` 共享 `WorkspaceRoutePage`；
- `/trips`：打开历史计划后当前仍跳到 `/result`；
- `/login`：已支持 `returnTo` 回跳。

### 4.2 与 v1.8 目标的差异

- 当前 `/` 还不是“新用户看 Landing、老用户进 Explore”的状态入口；
- 当前没有 `/welcome` 独立访问地址；
- 当前没有任何 onboarding 完成标记；
- 当前仍允许用户从 Landing 直接点入 `/workspace`；
- 当前历史计划仍走 `/result`，未迁到 `/workspace`。

## 5. 页面流程审计

### 5.1 当前 Landing 用户流程

当前首屏只提供 3 个入口：

- `/explore`
- `/create`
- `/workspace`

这意味着当前 Landing 实际上更像：

- 入口分发页；
- 临时导航页；
- 老用户和新用户混用的落地页。

它还不是 v1.8 目标中的“首次品牌欢迎页”。

### 5.2 当前首次访问状态

本次代码审计未发现与以下语义对应的已实现逻辑：

- onboarding complete；
- first visit；
- welcome seen；
- root redirect by onboarding。

当前只发现：

- `localStorage` 主要用于 Trip 草稿、TripPlan、已恢复 saved trip 元数据；
- 未发现 Landing 首次引导状态专用 key；
- 未发现 cookie 方案；
- 未发现 `middleware` 或根路由分流逻辑。

结论：

- v1.8 Landing 的核心状态机制当前尚未实现；
- 当前 `/` 行为完全不符合 Step 5 审计基线。

### 5.3 当前跨页面链路

- Landing -> Explore：存在；
- Landing -> Create：存在；
- Landing -> Workspace：存在，但不符合 v1.8 Landing 职责；
- Explore inspiration -> Create：存在，且通过 `startExploreCreateFlow()` 写入本地草稿；
- Plan -> Workspace：存在；
- Trips -> Result：存在，未迁移；
- Login -> returnTo：存在。

## 6. 当前 UI 结构

### 6.1 首屏结构

当前 `LandingPage` 的可见内容非常少：

- 一张全屏 Hero 大图；
- 一个顶部最简 Header；
- 底部居中的 3 个圆角按钮。

不可见但仍留在组件里的内容还有：

- 示例 prompt 区；
- “WHY THIS HOME EXISTS” 模块；
- Footer。

这些内容被放在 `hidden` 容器里，说明当前 Landing 仍停留在半成品状态。

### 6.2 Header 与用户状态

当前 Landing 使用 `Header minimal`：

- 左上是横版 Logo；
- 右上是 `UserMenu`；
- 默认没有为 Landing 明确配置 `Explore / Create` 主次导航；
- 已登录状态会显示邮箱和退出登录；
- 未登录状态会显示“登录”按钮。

优点：

- 登录态信息已经接入；
- 顶部品牌露出已经存在。

问题：

- 当前已登录用户来到 Landing 时，没有“进入 Explore / 返回我的旅行”的明确轻入口；
- 当前最简头部更多是视觉壳，不是完整的 Landing 导航层。

### 6.3 文案状态

Landing 首屏 CTA 为中文，但隐藏区仍有明显英文：

- `WHY THIS HOME EXISTS`

跨页面检查还可见大量工程/阶段英文沿用到与 Landing 相邻链路：

- `TRAVEL ARCHIVE`
- `ALREADY IN`
- `MAGIC LINK`
- `LOGIN REQUIRED`
- `RESULT MISSING`
- `WORKSPACE EMPTY`

结论：

- Landing 自身不是“全英文”，但它所在链路还远未达到 v1.8 中文产品语言标准。

## 7. 电脑端宽度检查

### 7.1 本次可验证范围

本次运行层验证结果：

- `npm run dev` 可正常启动；
- Next dev 在本地输出 `Ready in 1199ms`；
- 说明当前页面至少可启动到本地开发态。

### 7.2 本次无法获得的结果

当前会话没有可用的真实截图 / 浏览器可视化验收工具，因此本次未拿到：

- `1280` 实拍结果；
- `1440` 实拍结果；
- `1920` 实拍结果。

因此与宽度直接相关的以下结论，只能基于代码推断：

- 当前页面是全屏图 + 底部按钮布局，结构上较容易在大屏显得过空；
- `max-w-4xl` 的 CTA 容器在 `1440 / 1920` 下大概率缺少足够的内容锚点；
- Header 使用固定 `left-[48px] right-[48px] top-[32px]`，在更宽屏幕上可能显得品牌区与首屏主体脱节；
- 当前无多段内容节奏，桌面端“海报欢迎页”的完成度不足。

结论状态：

- 需要后续人工截图验收。

## 8. 图片资产审计

### 8.1 当前 Landing 实际使用

Landing 当前首页显式使用的主图仅有：

- `/images/landing/hero/hero-main.png`

Explore 模块还借用了部分 Landing 装饰资源：

- `/images/landing/decoration/routes/02-airplane-route.png`
- `/images/landing/decoration/routes/12-route-notebook.png`
- `/images/landing/decoration/stamps/09-next-stop-ticket.png`

### 8.2 Landing 目录现状

`public/images/landing/` 当前包含：

- `hero/`
- `background/`
- `decoration/cities/`
- `decoration/florals/`
- `decoration/routes/`
- `decoration/stamps/`
- `decoration/tape/`

大部分文件体积在 `1MB+` 左右。

### 8.3 与 v1.8 正式范围的冲突

Landing 目录内可见大量国际语义素材：

- Amsterdam
- Barcelona
- Florence
- India
- Italy
- London
- Paris
- Rome
- Santorini
- Switzerland
- Kyoto

这与 v1.8 正式 Landing 的中国范围基线冲突。

结论：

- 当前 Landing 资源池很大，但正式可用池尚未冻结；
- 国际图片不应直接进入 v1.8 正式 Landing；
- 当前正式首页只用了一张主图，资源使用率低，但资源范围风险高。

## 9. 主视觉方向审计

### 9.1 当前方向

当前 Landing 主视觉是：

- 单张全屏实景图；
- 深色叠加遮罩；
- 底部浮层式按钮。

它已经具备“海报感”，但尚未具备：

- 明确产品解释；
- 明确品牌欢迎层次；
- 明确 Explore 主 CTA / Create 次 CTA 节奏；
- 已登录用户回看欢迎页的轻返回层。

### 9.2 当前偏差

当前主视觉偏差主要有三类：

- 过于依赖单图，产品叙事太少；
- 目前像“视觉封面 + 入口按钮”，不像“完整欢迎体验”；
- 资产池中混有不符合正式中国范围的城市与邮票意象。

## 10. 动效审计

当前与 Landing 直接相关的动效状态：

- 已有 hover 位移；
- 已有 button transition；
- 已有 backdrop blur；
- 全局已支持 `prefers-reduced-motion: reduce`。

结论：

- 动效基础不是问题；
- 当前更大的问题不是“缺动画”，而是 Landing 内容结构过薄，动效没有明确服务对象；
- v1.8 后续应控制为轻量入场和 CTA 反馈，不需要额外炫技。

## 11. 性能审计

### 11.1 当前正向点

- Landing 当前首屏实际显式图片只有一张 Hero；
- `next/image` 已启用；
- `priority` 已设置；
- `sizes="100vw"` 已设置；
- CSS 背景纹理主要用纯 CSS，不依赖额外图片。

### 11.2 当前主要风险

- `hero-main.png` 本身体积约 `2.40MB`；
- Landing 目录内大量候选图仍为大体积 PNG；
- 当前资产格式几乎全是 PNG，缺少更轻量导出策略；
- 后续如果直接把更多装饰图堆入首页，首屏压力会迅速放大。

结论：

- 当前首屏尚未爆炸，但正式精修前必须把 Landing 图像接入策略和压缩策略一起定掉；
- Step 5 不建议直接继续加图而不先做资产冻结。

## 12. 可访问性审计

当前可确认的可访问性状态：

- Hero 图片有 `alt`；
- CTA 使用 `Link`，可键盘聚焦；
- Header Logo 可聚焦返回首页；
- 全局有 `focus-visible` 样式；
- reduced motion 降级已存在。

当前问题：

- Landing 主要信息几乎都压在图片和按钮之上，首屏语义层太薄；
- 当前没有清晰的中文标题 / 副标题 / 功能解释结构；
- 已登录与未登录状态差异主要靠右上角小区域体现，不够显著；
- 宽屏下信息密度太低，可能导致理解效率差。

## 13. 跨页面衔接

Landing 与周边页面的衔接状态如下：

- 与 Explore：入口已通，但 Landing 没有正式声明其为主入口；
- 与 Create：入口已通，且视觉上与 Explore 同等级竞争；
- 与 Workspace：当前错误暴露为第三主入口；
- 与 Login：右上角用户菜单已通，`returnTo` 也已通；
- 与 Trips：当前 Landing 没有“已登录用户返回我的旅行”的明确轻入口；
- 与 `/welcome`：当前完全未实现；
- 与 `/result` / `/workspace` 迁移目标：当前 Landing 侧没有体现 v1.8 新规则。

结论：

- Landing 不是孤立问题；
- 它当前和 Explore / Create / Login / Trips 的链路已经连上，但“连法不对”。

## 17. 建议保留内容

当前 Landing 中建议保留的部分有：

- 全屏海报式欢迎方向；
- `Header minimal + UserMenu` 的基本结构；
- 以 Explore、Create 作为主要入口来源的方向；
- `next/image` 加载方式；
- 暖纸张色系、克制色板和 reduced motion 全局基础；
- 让已登录状态也能留在欢迎页而不被强退的设计方向。

## 18. 不应推翻内容

本次审计认为以下内容不应被推翻：

- Explore 应是日常首页；
- Create 应继续作为 AI 创建流程入口；
- Landing 不需要复制完整 Create 表单；
- 登录能力应继续走当前 Magic Link 主链路；
- 当前全站暖纸张 / 手账化基础样式可继续复用；
- Landing 继续保留“品牌海报感”是合理的，只是当前完成度不够。

## 19. 技术补全输入

Landing 技术侧后续最需要补的不是样式，而是状态与路由：

1. 根路由 `/` 的首次访问分流规则；
2. `/welcome` 独立路由；
3. onboarding 完成时机；
4. 已登录用户访问欢迎页时的轻返回逻辑；
5. 与 Login `returnTo` 的一致行为；
6. 深层链接不被强制拦截的实现边界。

## 20. UI 精修输入

Landing UI 侧后续应重点解决：

1. 首屏中文标题与一句话产品解释；
2. Explore 主 CTA、Create 次 CTA 的明显层级；
3. 已登录用户在欢迎页的轻入口；
4. 桌面端宽度下的内容节奏；
5. 把当前隐藏区重构为正式欢迎信息，而不是保留半成品块。

## 21. 图片处理输入

## 审计结论摘要

当前 Landing 不是坏页面，但它还不是 v1.8 需要的正式欢迎页。

现状更接近：

- 一张海报图；
- 一个临时入口页；
- 一个尚未接上 onboarding 和 `/welcome` 的根路由壳。

进入后续阶段前，最重要的不是立刻美化，而是先把以下三件事锁定：

1. `/` 与 `/welcome` 的状态逻辑；
2. Explore 主 CTA / Create 次 CTA / 去掉 Workspace 主入口；
3. Landing 正式中国范围图片池与桌面端视觉节奏。
