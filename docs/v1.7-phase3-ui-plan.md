# v1.7 Phase 3 UI Plan

## Goal

Define the smallest safe order for UI asset binding after the Phase 3.0 audit.

This plan is for:

- component-to-image binding
- low-risk runtime image adoption
- dependency-aware sequencing

This plan is not for:

- visual redesign
- CSS overhaul
- animation
- image generation

## Guiding Principle

Bind assets where:

1. the component structure is already stable
2. runtime assets already exist
3. missing assets will not force rework one step later

## Priority 1: Explore

### Why first

Explore is the first product entry for the new Travel Discovery Hall direction, and its structure is already stable:

- `CitySearchSection`
- `InspirationDeck`
- `FeaturedSection`
- `ExploreFeed`

### Phase 3.1 recommended scope

1. Bind `InspirationCard` image slots for:
   - `location`
   - `food`
   - `season`
2. Add fallback handling for missing `companion` assets without changing layout
3. Prepare `ExploreCard` cover binding contract:
   - archive cover if present
   - city image fallback if available
   - neutral fallback if no match exists
4. Keep `FeaturedSection` structurally ready, but do not fully bind it until featured assets exist

### Ready inputs

- `public/images/explore/inspiration/location/*`
- `public/images/explore/inspiration/season/*`
- `public/images/explore/food/*`
- `public/images/explore/cities/*`

### Current blockers

- `public/images/explore/inspiration/companion/*`
- `public/images/explore/featured/*`
- `public/images/explore/archive/*`
- `public/images/explore/fallback/*`

### Phase 3.1 result target

Explore should become the first page where users can visibly feel the new image system without needing a full visual redesign.

## Priority 2: Archive Viewer

### Why second

Archive Viewer is the core long-term detail experience, but it depends on a more complete visual shell than Explore.

### Phase 3.1 recommended scope

1. Bind `ArchiveCover` first
2. Add a shared archive background/template layer for:
   - desktop drawer
   - mobile page container
3. Keep section-level components shared:
   - `ArchiveIntro`
   - `ArchiveItinerary`
   - `ArchiveFood`
   - `ArchivePOI`
   - `ArchiveMap`
4. Defer decorative overlays until base template and paper assets are stable

### Required assets before real binding

- `archive/template/*`
- `archive/paper/*`
- `archive/frame/*`
- `explore/archive/*`

### Can wait until Phase 3.2+

- `archive/bookmark/*`
- `archive/decoration/*`
- stamp/tape/paper-label overlays

### Phase 3.1 result target

Archive should move from plain detail sections to a technically image-backed viewer shell, even if it still looks visually restrained.

## Priority 3: Workspace

### Why third

Workspace already functions well, and premature image binding there risks mixing product polish work with heavier editing surfaces.

### Phase 3.1 recommended scope

1. Bind low-risk empty / feedback states first
2. Add sidebar icon asset mapping
3. Add save-status decorative asset hooks only after `ui/state` exists
4. Leave itinerary card textures, map overlays, and weather illustrations for later passes unless they solve a real clarity issue

### Best first candidates

- `WorkspaceSidebar`
- `WorkspaceRoutePage` empty / unavailable states
- `SaveStatus`

### Assets already available

- `ui/empty/*`
- `ui/feedback/*`
- `icons/outline/*`
- `workspace/map/*`
- `workspace/itinerary/*`

### Still blocked

- `ui/state/*`
- `icons/hover/*`

## Suggested Phase 3.1 Execution Order

1. Explore `InspirationCard`
2. Explore archive list card fallback strategy
3. Archive cover shell
4. Archive page / drawer shared template binding
5. Workspace empty / feedback states
6. Workspace sidebar icon binding
7. Workspace save-state accents

## What Not To Do In Phase 3.1

- Do not bind `_source` assets
- Do not fill missing sets with inconsistent temporary images
- Do not start Archive decoration before template and paper are ready
- Do not spread image binding across all Workspace subcomponents at once
- Do not let Explore and Archive use different fallback rules for the same destination content

## Minimal Success Standard

Phase 3.1 should be considered successful if:

1. Explore has its first real image-backed entry surfaces
2. Archive has a reusable viewer-shell binding contract
3. Workspace gains only low-risk, high-signal visual assets
4. no UI logic has to be reworked again when the full visual phase begins
