### 1.4 Archive unavailable risk points

- `ArchiveViewer` first requests `/api/explore/${id}` from live API.
- If API fails, it only falls back to `getMockExploreDetail(id)`.
- `getMockExploreDetail()` only knows mock `id` or mock `slug`.
- Therefore `ARCHIVE UNAVAILABLE` can happen when:
  - a live list item exists in DB but its detail request fails and the slug/id is not present in mock data
  - a future entry passes a non-slug non-UUID identifier
  - an unpublished or missing slug is linked into UI
  - the detail API returns 404 for an item that was visible in list data earlier
- Secondary risk:
  - naming is inconsistent: prop names alternate between `archiveId` and `slug`
  - current system works because the API is flexible, but the naming mismatch makes future drawer wiring easier to break

### 2.2 Runtime flow

1. User clicks an archive entry.
2. Router navigates to `/explore/${slug}`.
3. `app/explore/[id]/page.tsx` receives `params.id`.
4. `ArchivePageShell` renders full-page paper shell.
5. `ArchiveViewer` calls `fetchExploreDetail(id)`.
6. `/api/explore/[id]`:
   - treats the param as slug by default
   - if the value looks like UUID, tries `getById()` first, then `getBySlug()`
7. If API fails, client fallback tries `getMockExploreDetail(id)`.
8. If both fail, page shows `ARCHIVE UNAVAILABLE`.

### 2.5 Mock / placeholder / structurally weak sections

- `ArchiveMap`
  - explicit placeholder copy
  - no true map data, no coordinates, no route polyline
  - current purpose is only “future handoff”
- `ArchiveFood`
  - image is category/theme fallback only
  - not archive-specific real food images
- `ArchiveCover`
  - left large visual is archive template art, not archive content image
  - actual city/archive image is only the smaller right photo
- `ArchiveHeader`
  - technical labels like `ARCHIVE DRAWER`, `TRAVEL ARCHIVE`, short id code
  - useful for internal shell testing, not ideal for Chinese consumer reading

### 2.6 Why the current detail page feels messy

This is not one single problem. It is the combination of:

- Layout mismatch
  - current page is built as a full stacked paper page, not a constrained right-side reader
- Template-first instead of content-first
  - the first large visual area is decorative template art, not archive cover content
- Data split across DB, mock, and `rawContent`
  - some sections depend on normalized fields
  - some depend on `rawContent.recommendedFor`
  - some editorial fields are missing from DB mapping even though type definitions allow them
- Too many small sections
  - tags, recommended-for, highlights, map handoff, food, POI all render independently, making the reading rhythm fragmented
- English/internal residue
  - labels like `TRAVEL ARCHIVE`, `ROUTE RELATION`, `Map handoff`, `FOOD NOTES`, `PLACE RECORDS`, `Close`
- Image asymmetry
  - top block has decorative assets, but archive-specific imagery is limited
  - food/place cards do not have real per-record images
- Duplicate generation layers
  - there is an old wrapper layer under `components/explore/*`
  - active implementation lives under `components/explore/archive/*`
  - that duplication increases mental overhead during refactor

## 5. Archive Unavailable Risk Summary

Primary risk points:

- list-to-detail mismatch between live DB content and mock fallback content
- ambiguous naming: `archiveId` props frequently carry a `slug`
- current client fallback only knows mock ids/slugs, not arbitrary external ids
- future drawer work may accidentally pass `id`, `slug`, or `externalId` inconsistently

Recommended normalization rule before drawer implementation:

- UI open action should always pass one stable `archiveSlug`
- detail fetch layer may still accept slug or UUID, but UI contract should stop being ambiguous

## 7. Recommended Implementation Sequence

### Step A: Normalize data and image fields

- Define one UI-open identifier contract: use `slug`
- Add a detail-view model or selector so drawer/page both consume the same normalized archive shape
- Decide where `recommendedFor` lives:
  - top-level normalized field preferred
  - fallback from `rawContent.recommendedFor`
- Decide which editorial fields remain first-class:
  - `archiveIntro`
  - `featuredReason`
  - `theme`
  - `pace`
- Confirm cover fallback chain in one place only

### Step B: Change Explore home entry behavior to open drawer

- Keep card/link semantics
- Intercept in `ExploreHome`
- Replace current `router.push()` open flow with local selected-archive state
- Keep direct route fallback available for no-JS or cross-page entry

### Step C: Implement 60%-65% right drawer shell

- Reuse `ArchiveDrawerShell`
- Widen to target desktop width
- Add overlay and close behavior
- Decide mobile behavior separately
  - likely keep full page or bottom sheet later

### Step D: Refactor current reader content into drawer-first template

- Keep `ArchiveViewer` data loading
- Replace current paper-stack content order with the new simpler drawer template
- Reduce decorative sections
- Make cover image content-led

### Step E: Wire “用此行程创建”

- Reuse `buildTripPlanDraftFromExplore()`
- Keep current Create flow entry
- Add optional future quick actions:
  - budget lighter
  - slower pace

### Step F: Fill images and fallback strategy

- Ensure all visible archives resolve a top cover
- Add secondary fallback for non-China or future archives
- Delay food/place record images to later

### Step G: Keep `/explore/[id]` as compatibility route

Recommended near-term strategy:

- keep `/explore/[id]` alive for direct links, SEO, saved routes, and no-JS fallback
- make drawer the primary Explore-home experience
- later decide between:
  - full compatible detail page
  - page rendering the same drawer content in standalone mode
  - or redirect rules after product direction is stable
