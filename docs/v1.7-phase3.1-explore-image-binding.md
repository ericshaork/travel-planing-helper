# v1.7 Phase 3.1 Explore Image Binding

## Scope

This phase adds the first runtime image binding layer to Explore without changing:

- database
- API contracts
- Explore data schema
- Workspace logic
- animation system

The work is intentionally limited to:

- `InspirationCard`
- `FeaturedSection`
- `ExploreFeed`
- `ExploreCard`
- lightweight Archive asset reception in `ArchiveCover`

## 1. Resolver Layer

New file:

- `lib/explore/image-resolver.ts`

Purpose:

- keep image mapping and fallback rules out of UI components
- centralize city image path generation
- centralize featured/feed/archive image candidates
- provide a small archive asset bundle for future visual phases

## 3. Archive Reception

This phase does not redesign Archive Viewer, but `ArchiveCover` now receives a minimal asset bundle:

- template:
  - `/images/archive/template/archive-template-main.png`
- paper:
  - `/images/archive/paper/archive-paper-light.png`
- frame:
  - `/images/archive/frame/archive-photo-frame-01.png`
- cover candidates:
  - city-specific Explore covers
  - archive list city cover
  - archive list default cover
  - city fallback

## 4. Runtime Loading Rule

Shared component:

- `components/explore/ResolvedImage.tsx`

Behavior:

- uses `next/image`
- accepts ordered `sources`
- starts with the first candidate
- falls through to the next candidate on load error
- prevents components from carrying repeated fallback `if` chains

## 5. Finalized Contract Notes

The food inspiration contract is now finalized to semantic runtime filenames:

- `spicy-card.png`
- `seafood-card.png`
- `tea-card.png`
- `dessert-card.png`

Fallback handling remains:

- `/images/explore/fallback/explore-fallback-food.png`

Archive note:

- a dedicated `archive/cover/*` asset is not required by the current `ArchiveCover` implementation
- the current archive opening surface is composed from existing template, paper, frame, and cover-candidate assets

## 6. Result

After this phase:

- Explore feed cards resolve city-aware or archive fallback covers
