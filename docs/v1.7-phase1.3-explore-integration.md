Important normalization rules:

- `city` object maps to `city / cityCode / region`
- `budget` maps to `budgetLevel / budgetNote`
- `highlights` are derived from tags, day titles, POI names, and food names
- `dayNumber` is normalized by array order instead of trusting source `day`

## Import Flow

Importer script:

- [scripts/import-explore.ts](/C:/Users/10200/Desktop/travel_planing/scripts/import-explore.ts)

Flow:

1. read every JSON file from the target directory
2. parse raw JSON
3. normalize with `buildExploreTripImport`
4. validate against `ExploreTripContentInsert`
5. optionally upsert into Supabase by `external_id`
6. print summary and failed file list

Supported options:

- `--dry-run`
- `--dir`
- `--batch-id`
- `--status`
- `--published-at`
- `--report`

Dry-run verification result:

- total: `200`
- success: `200`
- failed: `0`

Sample import verification:

- sample size: `5`
- success: `5`
- failed: `0`
- database record count after sample import: `5`
- `GET /api/explore` returned list data successfully
- `GET /api/explore/[id]` slug lookup returned detail data successfully

Full import verification:

- source files: `200`
- importer success count: `200`
- importer failed count: `0`
- final database record count: `193`

Reason for `193` instead of `200`:

- the source folder contains `193` unique `external_id` values
- `7` pairs of JSON files share duplicate `external_id`
- importer upsert behavior intentionally uses `external_id` as the conflict key

Duplicate `external_id` pairs found:

- `exp_beijing_student_3d_food_citywalk_v1`
- `exp_chengdu_student_3d_food_citywalk_v1`
- `exp_dali_student_3d_food_citywalk_v1`
- `exp_guangzhou_student_3d_food_citywalk_v1`
- `exp_guilin_student_3d_food_citywalk_v1`
- `exp_hangzhou_student_3d_food_citywalk_v1`
- `exp_wuhan_student_3d_food_citywalk_v1`

Important behavior:

- default import status is `reviewing`
- for Explore cold-start display, published seed data should be imported with
  `--status published`

2. `ExploreGrid` requests `GET /api/explore`
3. API reads from repository
4. repository reads published rows from Supabase
5. `/explore/[id]` mounts `ExploreDetail`
6. `ExploreDetail` requests `GET /api/explore/[id]`
7. API returns normalized detail payload

## Risks

- imported content with status `reviewing` will not appear in the public Explore
  list until re-imported or updated to `published`
- pipeline currently does not provide `theme` or `cover_image`, so those fields
  remain optional
- source day numbering is not fully trustworthy, so importer normalization must
  remain in place
- duplicate `external_id` values reduce final row count because importer upserts
  by source identity rather than inserting duplicates
- this phase adds page skeletons only; filtering UI, pagination, and visual
  polish remain for later phases
