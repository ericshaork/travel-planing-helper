# v1.7 Explore JSON Analysis

## Scope

This document records the real structure of Explore seed content generated by
`travel-content-pipeline/output/explore_plans_production/`.

The goal is to align the main site data model, importer, database schema, and
API with the actual JSON shape before building the Explore integration loop.

## Sample Files Reviewed

- `beijing_couple_2d_001.json`
- `beijing_family_3d_003.json`
- `beijing_student_3d_002.json`
- `beijing_solo_2d_006.json`

Additional batch checks were also run across all 200 files to verify parse
success and spot structural anomalies.

## Top-Level Fields Found

All reviewed samples shared this top-level shape:

```json
{
  "id": "string",
  "slug": "string",
  "title": "string",
  "summary": "string",
  "city": {
    "code": "string",
    "name": "string",
    "region_name": "string"
  },
  "trip_type": "string",
  "days": 2,
  "budget": {
    "level": "low | medium | high",
    "note": "string"
  },
  "pace": "slow | balanced | dense",
  "tags": ["string"],
  "image_prompt": "string",
  "daily_itinerary": [],
  "pois": [],
  "food": []
}
```

## Nested Field Structure

### `city`

```json
{
  "code": "beijing",
  "name": "Beijing",
  "region_name": "Beijing"
}
```

Notes:

- `city` is an object, not a plain string.
- `code` is suitable for filtering and URL-safe matching.
- `name` is suitable for display.
- `region_name` is suitable for secondary metadata.

### `budget`

```json
{
  "level": "medium",
  "note": "..."
}
```

Notes:

- `budget.level` is compact and filter-friendly if needed later.
- `budget.note` is narrative text and should stay flexible.

### `daily_itinerary[]`

```json
{
  "day": 1,
  "title": "string",
  "summary": "string",
  "activities": [
    {
      "time_block": "morning | afternoon | evening | ...",
      "description": "string",
      "poi_refs": ["poi-id"],
      "food_refs": ["food-id"]
    }
  ]
}
```

### `pois[]`

```json
{
  "id": "string",
  "name": "string",
  "district": "string",
  "type": "string",
  "reason": "string",
  "recommended_duration_minutes": 90
}
```

### `food[]`

```json
{
  "id": "string",
  "name": "string",
  "district": "string",
  "category": "string",
  "reason": "string"
}
```

## Missing or Nonexistent Fields

The current Explore pipeline output does not provide these fields directly:

- `theme`
- `highlights`
- `cover_image`

Implications:

- `theme` should remain optional in the main-site Explore model unless a later
  pipeline version guarantees it.
- `highlights` should be derived during import from existing content instead of
  assumed to exist in source JSON.
- `cover_image` should remain nullable for now and must not block import.

## What Should Become Database Columns

Recommended relational columns:

- `id` (DB UUID primary key)
- `external_id`
- `slug`
- `title`
- `summary`
- `city`
- `city_code`
- `region`
- `trip_type`
- `days`
- `tags`
- `theme`
- `pace`
- `budget_level`
- `budget_note`
- `status`
- `review_status`
- `cover_image_url`
- `image_prompt`
- `published_at`
- `created_at`
- `updated_at`
- source metadata fields such as `source_pipeline`, `source_batch_id`,
  `source_content_key`, `source_file_path`

## What Should Stay JSON

Recommended JSONB fields:

- `highlights_json`
- `itinerary_days_json`
- `poi_highlights_json`
- `food_highlights_json`
- `raw_content_json`

Reasoning:

- These structures are nested and likely to evolve.
- Explore detail pages need full content fidelity.
- Keeping nested content in JSONB avoids premature relational over-modeling.

## Dataset Quality Notes

Batch verification findings:

- All 200 files parsed successfully with Node.js.
- Some files contain repeated or inconsistent `daily_itinerary[].day` values
  even when array length matches the declared `days`.

Examples:

- `beijing_student_3d_002.json` -> `[1, 1, 1]`
- `dunhuang_student_3d_183.json` -> `[1, 1, 1]`
- `lijiang_family_2d_175.json` -> `[1, 1]`
- `shanghai_family_3d_018.json` -> `[1, 1, 3]`
- `shenzhen_student_2d_035.json` -> `[1, 1]`

Importer implication:

- The importer should normalize `dayNumber` by array order when source day
  values are duplicated or invalid.

## Summary

The Explore pipeline output is already structurally usable for main-site
integration, but the current main-site Explore schema assumptions are outdated.
The main adjustments required are:

1. Treat `city` as structured metadata instead of a string.
2. Add support for `pace`, `budget`, and source `externalId`.
3. Derive `highlights` instead of expecting it in source JSON.
4. Preserve `daily_itinerary`, `pois`, and `food` as JSONB content.
5. Normalize day numbering during import to protect Explore detail rendering.
