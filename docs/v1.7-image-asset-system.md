# v1.7 Image Asset System

## 1. Phase Goal

本阶段只设计正式图片资产体系，不执行任何迁移，不修改代码，不改 UI，不接入图片。

当前仓库现状：

- 当前分支未扫描到已提交的运行时图片目录。
- 当前分支未扫描到已提交的正式图片文件（`png/jpg/webp/svg/gif`）。
- 当前代码中也没有稳定的图片资源引用链路。

这说明现在最需要的不是继续堆素材，而是先冻结一套和网站架构一致的图片系统。

## 2. Design Principle

新的图片体系不再按“生成了什么类型的图”组织，而是按“网站哪个模块在用”组织。

组织原则：

1. 先按产品模块分层。
2. 页面直接使用的成品图，与原始大图、sheet、参考图分开。
3. 目录命名稳定，不跟随单次设计稿波动。
4. 一个资源只服务一个主要场景；跨场景复用时保留来源说明。
5. 网站运行时只引用正式目录，不引用 `_source/`。

## 5. Naming Rules

推荐命名规则：

- 全小写
- 用 `kebab-case`
- 避免中文名、空格、版本尾缀混乱
- 避免 `final-final-2` 之类命名

推荐格式：

```text
[module]-[subject]-[variant]-[size].[ext]
```

示例：

- `landing-hero-city-night-wide.webp`
- `explore-location-beach-card.webp`
- `archive-cover-chengdu-food-walk.webp`
- `workspace-weather-rain-umbrella.png`
- `brand-logo-horizontal-dark.svg`

## 6. Runtime Rule

运行时只允许引用：

- `images/brand`
- `images/landing`
- `images/explore`
- `images/archive`
- `images/workspace`
- `images/icons`
- `images/stickers`
- `images/ui`

运行时禁止引用：

- `images/_source`

## 7. Current Conclusion

当前分支的关键问题不是“图片太多”，而是“正式资产体系尚未落地”：

- 没有正式 runtime 图片目录。
- 没有页面到资产的稳定映射。
- 没有成品图与源素材分层。
- 没有围绕 Landing / Explore / Archive / Workspace 的模块化管理。

因此，v1.7 Phase 2.2.4.0 的正确结果应当是：

1. 先冻结目录架构。
2. 再按页面需求补资源。
3. 之后才做迁移和代码接图。
