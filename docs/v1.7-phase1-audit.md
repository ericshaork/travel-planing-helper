## 1. 审计目标

本阶段只做架构与技术审计，不修改代码，不调整 UI，不做视觉升级。

当前审计重点：

- 盘点当前项目目录结构
- 盘点现有页面与缺失页面
- 判断页面职责是否合理

- 给出 Explore 技术接入方案
- 给出组件拆分建议
- 给出 `v1.7 Phase 1` 的实施顺序与风险点

#### `/explore`

## 3. `/result` 是否承担过多 workspace 功能

结论：**是，且已经明显过载。**

[app/result/page.tsx](/C:/Users/10200/Desktop/travel_planing/app/result/page.tsx) 当前约 `1621` 行，承担的内容包括：

- 旅行结果状态恢复
- `TripPlan` 与 `TripRequest` 的 localStorage 读取
- enrichment 请求与错误兜底
- mobile / desktop 双视图
- Workspace sidebar 行为
- 当前 Day 切换
- 地图点位联动
- itinerary block 选中联动
- pending changes 状态
- quick modification
- regenerate 流程
- export 入口
- save current trip 按钮接入

这带来的问题：

1. `/result` 已经成为事实上的 workspace 页面，但没有正式升格
2. 页面体量过大，后续 UI Upgrade 风险高
3. state、layout、交互、数据加载耦合过深
4. future workspace 和 result 的职责边界会越来越模糊

建议结论：

- `v1.7 Phase 1` 必须先把 `/result` 当作“待拆的临时工作台”
- 后续不应继续把新能力直接堆到这一个页面里

## 6. Explore 技术接入方案

### 6.1 方案 A：JSON 直接读取

做法：

- 直接把 pipeline 输出 JSON 放进主站
- `/explore` 直接读取 JSON 渲染

优点：

- 起步快
- 适合本地验证结构
- 不用立刻设计数据库

缺点：

- 不适合作为正式线上内容源
- 200 份内容以后维护成本迅速上升
- 不适合筛选、分页、上下架、推荐
- 更新内容需要重新部署
- 内容审核流不好接

### 6.2 方案 B：Database 存储

做法：

- pipeline 输出 JSON
- 经审核后导入数据库
- 主站 `/explore` 从数据库读内容

优点：

- 适合公共内容资产库
- 适合 200+ 内容规模
- 支持审核、发布、排序、推荐、筛选
- 更适合未来内容运营

缺点：

- Phase 1 需要补数据库表和导入流程设计

### 6.3 推荐

推荐：**Database 存储作为正式方案**。

原因：

1. 你已经明确产品方向升级为 Explore 内容平台，不是静态攻略页。
2. 你已经明确 pipeline JSON 未来要走“审核 -> 数据库 -> Explore 页面”。
3. 当前约 200 份内容规模，已经不适合把 JSON 当正式前台数据源。

### 6.4 推荐落地方式

建议两段式：

#### 第一段

- JSON 只作为 pipeline 输出和导入中间产物
- 用来做结构校验和导入前检查

#### 第二段

- 主站正式读取数据库
- Explore 页面不直接依赖 JSON 文件

## 11. 风险点

### 风险 1：继续把新能力堆进 `/result`

后果：

- result 页进一步膨胀
- workspace 永远无法真正独立

### 风险 2：把 Explore 直接做成 JSON 页面

后果：

- 短期快，长期难维护
- 很快卡在审核、排序、推荐、分页上

### 风险 3：把 Explore 内容直接塞到 `trip_plans`

后果：

- 用户私有数据与公共内容资产混淆
- 权限与运营逻辑会变脏

### 风险 4：先做 UI，再做结构

后果：

- 后续 UI Upgrade 会被架构耦合反复拖慢
- 很容易一边改外观，一边被迫返工数据与状态层

## 12. 审计结论

当前项目已经具备：

- Landing
- Create
- Result
- Trips
- AI 生成主链路
- 天气与地图基础能力
- 初步的 workspace 组件

但还没有具备：

- 真正独立的 Workspace 核心入口
- Explore 内容平台的页面与数据层
- 收藏体系
- 内容资产模型

所以 `v1.7 Phase 1` 的正确方向不是先做视觉，而是先补技术基础：

1. 把 `/result` 从“过载结果页”解耦成更清晰的工作台结构
2. 明确 `/workspace` 的正式入口定位
3. 为 Explore 设计独立数据模型和数据库接入方案
4. 把 JSON 降级为导入中间产物，而不是正式线上数据源
