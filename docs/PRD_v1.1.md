# PRD v1.1：点击式旅行积木编辑

## 1. 文档定位

本文是 MVP 1.1 的增量产品需求，建立在已完成的 MVP 1.0 之上。MVP 1.0 的完整需求仍以 `docs/PRD.md` 为准，本文不覆盖或重述 1.0 的全部能力。

MVP 1.1 只围绕三件事展开：

1. 让用户清楚知道 `/plan` 还缺什么；
2. 让手机端用户快速找到某一天和修改入口；
3. 让用户不用写复杂提示词，也能通过点击表达修改意图。

## 2. 产品定位

### 2.1 一句话定位

面向自由行新手的可编辑 AI 旅行计划：AI 先生成一版，用户再通过点几下，把方案改成更适合自己的版本。

### 2.2 v1.1 的核心差异

v1.1 不是继续堆“生成更多内容”，也不是把结果页改成聊天窗口。它把完整行程变成可理解、可操作的旅行积木：

- 点击式积木编辑为主；
- 轻量自然语言修改为辅；
- 用户不用说得很复杂；
- 每次点击都先生成一条可检查的修改意图；
- 用户确认后才重新生成完整方案。

### 2.3 产品承诺

- 个性化：用户能明确表达不喜欢、想替换、必须保留和想增加的内容；
- 省时间：不用重新描述整趟旅行，也不用学习提示词；
- 适合自己：用户能把“AI 给的版本”快速变成“我的版本”；
- 可控：系统不会因一次误触立即重排，修改意图始终先让用户确认。

## 3. 目标用户

主要用户仍是自由行新手，特别是：

- 已经拿到 AI 生成的行程，但觉得其中一两个安排不合适；
- 知道自己“不想要这个”，却不知道怎样写完整修改要求；
- 想保留某个必去地点，又担心重新生成后丢失；
- 在手机上查看多日行程，难以快速跳到目标日期；
- 在 `/plan` 漏填字段后，不知道为什么无法继续。

本版本不以专业行程规划师、多人团队编辑者或需要精确分钟级排程的用户为主要对象。

## 4. 核心用户问题

### 4.1 缺失信息不可见

用户最后提交失败时，只知道“还不能生成”，却不能快速判断缺什么、缺项在哪一页。

### 4.2 结果页在手机上过长

多日行程、天气、预算、景点和建议连续展开，用户需要长距离下滑，难以回到某一天或修改入口。

### 4.3 修改门槛仍然偏高

现有 `RegenerateBox` 要求用户自己组织自然语言。很多用户只能判断“这个不喜欢”，却不擅长说清保留项、替换范围和新增偏好。

### 4.4 重新生成缺少可控感

用户担心重新生成会把满意的部分一起改掉，也不容易确认系统到底理解了什么。

## 5. v1.1 核心价值

| 优先级 | 价值 | 衡量方式 |
|---|---|---|
| P0 | 点几下就能表达修改意图 | 用户无需手写完整句子即可形成可提交要求 |
| P0 | 快速发现并定位缺失字段 | 提交失败后一次点击到达对应字段 |
| P0 | 手机端快速找到目标 Day | 无需从页面顶部盲目下滑 |
| P1 | 修改前可检查、可补充 | 任何快捷操作都不自动提交 |
| P1 | 重新生成仍然安全、完整 | 复用现有 API，返回完整且通过 Schema 的 `TripPlan` |
| P1 | 修改结果可解释 | `appliedChanges` 清楚说明这次改了什么 |

## 6. 功能范围

### 6.1 `/plan` 缺失信息提示增强（P0）

必须包含：

1. 顶部缺失信息摘要；
2. 每个缺失项使用用户能理解的字段名称和短提示；
3. 点击摘要中的缺失项，切到对应步骤并聚焦对应字段；
4. 字段附近显示字段级错误；
5. 缺失或错误字段有明显但不过度刺眼的高亮；
6. 最终提交失败时自动滚到第一个错误；
7. 用户修改字段后，相关错误及时清除或重新校验；
8. 日期组合错误、天数与日期不一致等非“空值”问题同样能定位；
9. 摘要使用 `role="alert"` 或等价可访问反馈，字段通过 `aria-invalid` 和错误说明关联。

顶部摘要不应变成长错误日志。推荐结构：

```text
还差 3 项就能排路线
[出发城市] [预算] [出行风格]
```

### 6.2 行程柜展示（P0）

结果页将现有三段式每日行程重新表达为：

- 一天一个 `Day Cabinet`；
- 每天固定三个 `Time Slot`：上午、下午、晚上；
- 每个 `ItineraryItem` 是一个 `Itinerary Block`；
- `Day Cabinet` 继续展示日期、主题、路线顺序、安排理由和当天提醒；
- `Itinerary Block` 继续展示地点、类型、推荐原因、停留时间、交通、天气、备选和攻略；
- 空时间段允许显示轻量空状态，不强行制造安排。

桌面端可默认展开多天；手机端默认先显示多日摘要，并只展开一个 Day，避免所有详情同时铺开。

### 6.3 手机端结果页导航（P0）

必须包含：

- 顶部 Day 导航；
- 当前 Day 明确选中；
- 点击 Day 后展开并定位到对应 `Day Cabinet`；
- 顶部“修改”入口，定位到 `RegenerateBox`；
- 多日摘要至少包含 Day、主题和简短路线概览；
- 展开状态不改变完整 `TripPlan`；
- 键盘和读屏用户可以识别展开状态；
- 页面刷新后无需持久化展开状态，可回到确定性的默认 Day。

“默认多日摘要”只适用于小屏幕展示策略，不删除内容，不改变复制和导出结果。

### 6.4 积木操作（P0）

每个 `Itinerary Block` 提供四个操作：

| 操作 | 用户意图 | 建议生成的修改句 |
|---|---|---|
| 不要这个 | 移除当前地点或活动，并避开再次安排 | “不要第 2 天下午的鼓浪屿，并避免把它安排到其他时段。” |
| 换一个 | 替换当前积木，其他约束尽量保留 | “把第 2 天下午的鼓浪屿换成同区域、更轻松的替代安排。” |
| 一定保留 | 锁定当前地点或活动 | “一定保留第 2 天下午的鼓浪屿，重排时不要删除。” |
| 加类似 | 增加与当前积木相似的体验 | “在不让行程过满的前提下，增加一个和第 2 天下午鼓浪屿类似的体验。” |

交互规则：

1. 点击后自动生成或更新 `modificationRequest`；
2. 自动把修改意图带到 `RegenerateBox`；
3. 不自动发起网络请求；
4. 用户能继续修改文字；
5. 用户必须点击确认按钮后才提交；
6. 提交继续使用当前 `TripRequest`、完整 `previousPlan` 和最终 `modificationRequest`；
7. 成功后用完整新 `TripPlan` 替换当前方案；
8. 失败时保留旧方案和未提交/可重试的修改意图；
9. 同一积木的互斥操作应合并或替换，避免同时出现“不要”和“保留”；
10. 操作文案必须包含 Day、时间段和地点，降低同名地点或多次出现造成的歧义。

### 6.5 快捷修改（P1）

提供以下快捷操作：

- 轻松一点；
- 少走路；
- 预算低一点；
- 加美食 / 夜市；
- 不早起。

规则：

- 快捷操作只生成 `modificationRequest` 或轻量 constraints；
- 不自动提交；
- 可与积木操作合并；
- 再次点击可以取消或避免重复；
- 修改句要具体，不使用空泛词；
- “预算低一点”只要求在原预算内降低开销，不生成或承诺实时价格；
- “少走路”只能作为偏好约束，v1.1 没有真实距离计算能力。

### 6.6 轻量聊天理解优化（P1）

只包含：

- 优化 `RegenerateBox` 占位提示；
- 优化积木操作和快捷操作生成的修改句；
- 展示待应用意图预览；
- 优化成功后的 `appliedChanges` 展示；
- 用户仍可在生成句后补充一句自然语言；
- 发生冲突时提示用户确认或以最后明确选择为准。

不包含：

- 多轮消息列表；
- 对话角色气泡；
- 会话历史；
- 跨页面长期记忆；
- Agent、工具调用编排或完整聊天系统。

### 6.7 完整重新生成（P0）

v1.1 继续复用：

```text
POST /api/generate-trip
tripRequest + previousPlan + modificationRequest
→ 完整新 TripPlan + appliedChanges
```

客户端的积木操作不是最终数据修改。只有服务端返回并通过现有 Schema 校验的完整 `TripPlan` 才能成为新的当前方案。

### 6.8 复制与导出回归（P0）

- 复制和 Markdown 导出继续读取当前完整 `TripPlan`；
- 展开/收起状态、Day 导航、constraints 和修改草稿不进入导出；
- 重新生成成功后，复制和导出使用新方案；
- 重新生成失败时，复制和导出仍使用旧方案。

## 7. v1.1 明确不做

- 拖拽；
- 调整格子高低；
- 新增或删除时间隔板；
- 自定义清晨、午餐、傍晚、深夜等 Time Slot；
- 精确时间轴；
- 地图 API；
- 真实距离和通勤时间计算；
- 实时开放时间；
- 真实票务、酒店价格、库存、预订、支付；
- 登录、收藏、数据库、跨设备同步；
- 真正局部 Patch；
- 直接在客户端修改 `TripPlan` 作为最终结果；
- 完整聊天系统、多轮消息和聊天历史；
- 历史版本管理或回滚；
- 多人协作；
- 为 2.0 提前重写架构。

## 8. 主要页面变化

### 8.1 首页 `/`

无核心功能变化。只做必要回归，不新增入口矩阵或营销模块。

### 8.2 分步补充页 `/plan`

新增：

- 页面顶部 `MissingFieldsSummary`；
- 缺失项点击定位；
- 字段级错误和高亮；
- 最终提交失败滚动与聚焦；
- 清楚区分缺失、格式错误和字段冲突。

保留：

- 三步填写结构；
- 现有草稿恢复；
- 现有标准化与生成流程。

### 8.3 结果页 `/result`

新增或调整：

1. 旅行总览；
2. 天气提醒；
3. 预算摘要；
4. 顶部 Day 导航与“修改”入口；
5. 手机端多日摘要；
6. `Day Cabinet / Time Slot / Itinerary Block`；
7. 积木操作；
8. 快捷修改与意图预览；
9. `RegenerateBox` 确认提交；
10. 住宿、交通、注意事项、复制和导出继续保留。

景点攻略可以继续作为补充区展示，但不能和 `Itinerary Block` 产生两套互相冲突的编辑入口。v1.1 默认只在每日行程积木上提供操作。

## 9. 核心用户流程

### 9.1 缺失信息修复

```text
用户进入 /plan
→ 顶部看到缺失摘要
→ 点击“预算”
→ 页面切到第 2 步并聚焦预算字段
→ 填写后错误消失
→ 提交时若仍有错误，自动定位第一个错误
→ 信息完整后正常生成
```

### 9.2 积木修改

```text
用户在 Day 2 下午看到某个地点
→ 点击“不要这个”
→ RegenerateBox 自动出现具体修改句
→ 页面定位或提示修改区已有 1 条意图
→ 用户检查、补充或取消
→ 点击“按这些要求重排”
→ POST /api/generate-trip
→ 返回完整新 TripPlan
→ 页面、复制和导出全部使用新方案
```

### 9.3 快捷修改

```text
用户点击“少走路”和“不早起”
→ 两条意图进入预览
→ 不自动请求
→ 用户确认
→ 完整重新生成
→ appliedChanges 说明已减少步行、推迟早间安排
```

## 10. 用户故事

### `/plan`

- 作为漏填信息的用户，我希望一眼看到还缺几项，而不是到最后自己回头找；
- 作为手机用户，我希望点击“预算”就跳到预算输入，而不是手动翻步骤；
- 作为遇到日期冲突的用户，我希望错误显示在日期附近，并告诉我怎么改。

### 结果浏览

- 作为查看 5 天行程的手机用户，我希望先看到每天摘要，再只展开我关心的一天；
- 作为行程使用者，我希望随时能跳到 Day 3，而不是不断滚动；
- 作为想改方案的用户，我希望顶部就能找到“修改”入口。

### 积木编辑

- 作为不喜欢某个景点的用户，我希望点“不想要这个”就能形成明确修改要求；
- 作为担心必去地点被删的用户，我希望点“一定保留”锁定意图；
- 作为不知道替代地点的用户，我希望点“换一个”让系统给出合适替代；
- 作为喜欢某类体验的用户，我希望点“加类似”，同时不要把行程塞得太满；
- 作为谨慎的用户，我希望所有点击先让我确认，不要一碰就消耗一次生成。

### 轻量修改

- 作为不知道怎么写提示词的用户，我希望点“轻松一点”就得到可读的修改句；
- 作为有额外想法的用户，我希望能在自动生成的句子后继续补充；
- 作为重排后的用户，我希望看见这次实际应用了哪些变化。

## 11. 验收标准

### 11.1 `/plan` 缺失提示

- [ ] 缺失核心字段时，页面顶部显示缺失数量和字段列表；
- [ ] 点击任一缺失项可切换到正确步骤；
- [ ] 对应字段被滚动到可见区域并获得焦点；
- [ ] 字段显示 `aria-invalid`、错误说明和高亮；
- [ ] 提交失败自动定位第一个错误；
- [ ] 修复后相关错误消失；
- [ ] 日期、天数、预算和列表字段的非空值错误也能字段级展示；
- [ ] 已有草稿恢复和生成流程不回归。

### 11.2 Day Cabinet

- [ ] 每个 `DailyItinerary` 映射为一个 Day Cabinet；
- [ ] 上午、下午、晚上固定三段且内容映射正确；
- [ ] 每个 `ItineraryItem` 只映射为一个 Itinerary Block；
- [ ] 路线、理由、天气、交通、备选与提醒仍可阅读；
- [ ] 桌面和手机均无横向溢出；
- [ ] 手机端默认显示多日摘要，只展开一个 Day；
- [ ] 顶部 Day 导航可切换和定位；
- [ ] 顶部“修改”入口可定位到修改区。

### 11.3 积木与快捷修改

- [ ] 每个积木都有四个规定操作；
- [ ] 操作生成的意图包含 Day、时间段、地点和动作；
- [ ] 点击后不发网络请求；
- [ ] 意图自动填入或同步到 `RegenerateBox`；
- [ ] 用户能编辑、取消和确认；
- [ ] 互斥操作不会同时生效；
- [ ] 五个快捷修改可生成明确意图且不自动提交；
- [ ] 用户确认后仍调用 `/api/generate-trip`；
- [ ] 请求包含 `tripRequest`、完整 `previousPlan`、`modificationRequest`；
- [ ] 成功返回完整 `TripPlan`，不是局部 Patch；
- [ ] `appliedChanges` 清楚展示；
- [ ] 失败时保留旧方案。

### 11.4 回归

- [ ] 旅行总览、天气、预算、住宿、交通和注意事项仍显示；
- [ ] 天气失败仍能降级；
- [ ] 复制和 Markdown 导出与当前完整方案一致；
- [ ] localStorage 损坏时仍有安全空状态；
- [ ] Mock AI / Mock Weather 零 Key 流程可运行；
- [ ] lint、typecheck、build、test 全部通过。

## 12. 朋友二轮测试重点

邀请 3—5 位自由行新手，不先解释操作，至少覆盖 360px、390px、430px 手机宽度和一个桌面宽度。

重点观察：

1. 用户漏填信息后，能否在 10 秒内说出还缺什么；
2. 用户是否会主动点击顶部缺失项；
3. 定位后是否知道该填哪个字段；
4. 手机端是否能在不连续长滚动的情况下找到 Day 3；
5. 用户是否理解“每天一个柜子、上午下午晚上三个格子”；
6. 用户想删一个地点时，是先点按钮还是先找聊天框；
7. 四个积木操作是否容易区分；
8. 点击后，用户是否注意到“还没有提交”；
9. 自动生成的修改句是否符合用户真实意图；
10. 用户是否会补充或改写自动生成的句子；
11. 用户是否敢点“一定保留”，并相信重新生成不会删掉它；
12. 快捷修改中哪几个最常用，哪几个含义模糊；
13. 重排后 `appliedChanges` 是否能建立信任；
14. 修改失败时，用户是否理解旧方案仍被保留；
15. 相比直接输入聊天要求，点击式编辑是否更省时间；
16. 用户是否强烈提出保存、分享或发给同行人；该反馈用于判断是否提前 v1.4；
17. 用户是否强烈要求自定义时间格；该反馈用于判断是否进入 v1.2；
18. 用户是否把“少走路”误解成真实距离计算；若有，需调整文案边界。

建议记录三个核心指标：

- 首次成功修改完成率；
- 从发现不满意到提交修改的用时；
- 无需手工重写自动修改句的比例。

## 13. v1.1 完成定义

只有同时满足以下条件，v1.1 才算完成：

1. `/plan` 缺失项可见、可点、可定位；
