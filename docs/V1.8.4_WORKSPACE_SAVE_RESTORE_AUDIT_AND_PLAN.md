本阶段只做 Workspace 保存 / 恢复 / 本地草稿链路审计与方案输出，不修改业务代码，不修改 schema，不修改 Trips API 行为，不修改 `localStorage` 结构，不 commit，不 push。

本次审计目标：

- 盘清当前有哪些入口会把用户送进 `/workspace`
- 盘清当前本地 `localStorage` / draft / session metadata 的真实结构

- 盘清游客与登录用户在 Workspace 保存上的边界
- 输出 Step 8B 的最小安全落地方案

## 3. Current Workspace Entry Sources

当前会进入 `/workspace` 的入口主要有五类。

### 3.1 AI 生成入口

路径：

1. `/create` 进入自然语言输入
2. `/plan` 补全 `TripRequest`
3. 调 `/api/generate-trip`
4. 成功后写本地 `trip-plan` 和 `workspace-session`
5. 跳 `/workspace`

当前特征：

- 使用 `saveTripRequest()` 把完整 `TripRequest` 写进本地
- 使用 `saveTripPlan()` 把完整 `TripPlan` 写进本地
- 使用 `setWorkspaceSessionMetadata()` 记录：
  - `sourceType = "ai_generated"`
  - `workspaceModeDefault = "read"`
- 调生成前会 `markCurrentTripAsUnsaved()` 和 `clearWorkspaceSessionMetadata()`
- 不带云端 `tripId`
- 不区分游客 / 登录用户，游客也能完整进入 Workspace

### 3.2 空白手搓入口

路径：

1. `/create`
2. 点击“从空白计划开始”
3. `createBlankWorkspaceDraft()`
4. 跳 `/workspace`

当前特征：

- 直接在本地创建一份空白 `TripPlan`
- 清掉：
  - `trip-request`
  - `trip-draft`
  - `trip-plan-draft`
  - `parsed-trip`
  - enrichment / weather summary
- 调用 `markCurrentTripAsUnsaved()`
- 写入 `workspace-session`：
  - `sourceType = "blank_manual"`
  - `workspaceModeDefault = "edit"`
  - `localDraftId`
  - `blankDraftId`
  - `createdAt`
- 仍然不带云端 `tripId`

### 3.3 Explore 导入入口

当前有两条 Explore 相关路径：

1. `startExploreCreateFlow()`
   - 只保存 `trip-plan-draft`
   - 跳 `/create?entry=explore`
   - 后续再进 `/plan`

2. `startExploreWorkspaceFlow()`
   - 从 Explore 草稿构造成一个可打开的 `TripPlan`
   - `saveTripPlanDraft()`
   - `saveTripPlan()`
   - `clearTripRequest()`
   - `markCurrentTripAsUnsaved()`
   - `setWorkspaceSessionMetadata({ sourceType: "ai_generated", workspaceModeDefault: "read" })`
   - 跳 `/workspace`

注意：

- Explore 进入 Workspace 当前没有写入专门的 `explore_import` session source
- `TripPlanDraft.sourceType` 在 draft 结构里有，但 Workspace session 目前没有沿用它
- 这意味着“Explore 导入”在 Workspace 里会被看成一种 AI 来源近似态，而不是独立来源

### 3.4 My Trips 历史计划打开入口

路径：

1. `/trips`
2. `openSavedTripIntoWorkspace(tripId)`
3. 可选先 `PATCH last_opened_at`
4. `GET /api/trips/[tripId]`
5. `restoreSavedTripToStorage()`
6. 跳 `/workspace`

恢复到本地时会写：

- `trip-request`
- `trip-plan`
- `trip-enrichment`
- `trip-weather-summary`
- `restored-saved-trip`
- `workspace-session`

其中：

- `restored-saved-trip.savedTripId = cloud trip id`
- `workspace-session.sourceType = "saved_trip"`
- `workspaceModeDefault = "read"`

这是当前唯一一条会把“云端 trip 身份”写回本地 metadata 的入口。

### 3.5 /result 兼容入口

`/result` 当前并没有独立的数据源，而是和 `/workspace` 共用 `WorkspaceRoutePage`：

- `/workspace` -> `WorkspaceRoutePage mode="workspace"`
- `/result` -> `WorkspaceRoutePage mode="result"`

但 `WorkspaceRoutePage` 实际读取的仍然是：

- `loadTripPlan()`
- `loadTripRequest()`

也就是说：

- `/result` 只是一个兼容路由壳
- 它依赖的还是同一套本地 `localStorage` 事实来源
- Step 8B 不能误伤这套共享恢复逻辑

## 4. Current localStorage / Draft Structure

### 4.1 Current Keys

`lib/trip/storage.ts` 当前定义了这些 key：

- `travel-planning:parsed-trip`
- `travel-planning:trip-draft`
- `travel-planning:trip-plan-draft`
- `travel-planning:trip-request`
- `travel-planning:trip-plan`
- `travel-planning:trip-enrichment`
- `travel-planning:trip-weather-summary`
- `travel-planning:restored-saved-trip`
- `travel-planning:workspace-session`

### 4.2 trip-draft

`TRIP_DRAFT_STORAGE_KEY` 存的是 `TripRequestDraft`，只包含“计划前置需求草稿”，例如：

- 出发地 / 目的地
- 日期 / 天数
- 预算
- interests / travelStyles
- mustVisitPlaces / avoidPlaces
- accommodation / schedule / transport preference
- specialRequirements

特点：

- 不含 `TripPlan`
- 不含 `title`
- 不含云端 `tripId`
- 不含 `source_type`
- 不含 `local_draft_id`
- 本质上是 `/plan` 阶段表单草稿

### 4.3 trip-plan-draft

`TRIP_PLAN_DRAFT_STORAGE_KEY` 存的是 `TripPlanDraft`：

- `tripTitle`
- `sourceType`
- `sourceExploreId?`
- `sourceExploreSlug?`
- `tripRequestDraft`
- `tripPlanSeed`

特点：

- 它确实带 `sourceType`
- 但它是 Explore / 计划前置阶段用的 draft，不是 Workspace 持续编辑的事实主源
- 不带云端 `tripId`
- 不带 `local_draft_id`
- 不带 `savedTripId`

### 4.4 trip-request / trip-plan

`TRIP_REQUEST_STORAGE_KEY` 与 `TRIP_PLAN_STORAGE_KEY` 才是 `/workspace` 与 `/result` 真正读取的主数据：

- `trip-request` 存完整 `TripRequest`
- `trip-plan` 存完整 `TripPlan`

特点：

- `trip-plan` 是 Workspace 编辑过程持续覆盖写入的主键值
- 不带云端 `tripId`
- 不带 `source_type`
- 不带 `status`
- 不带 `trip_preferences_json`
- 不带 `local_draft_id`
- 不带 `updatedAt / savedAt`

### 4.5 restored-saved-trip

`RESTORED_SAVED_TRIP_STORAGE_KEY` 存的是 `SavedTripMetadata`：

- `savedTripId`
- `savedTripTitle`
- `restoredAt?`
- `savedAt?`

这是当前“本地正在绑定哪条云端 trip”的唯一正式标识。

特点：

- 不在 `trip-plan` 里
- 不在 `workspace-session` 里
- 单独存在
- 被 `persistCurrentTrip()` 用来判断“新建还是覆盖”

### 4.6 workspace-session

`WORKSPACE_SESSION_STORAGE_KEY` 存的是 `WorkspaceSessionMetadata`：

- `sourceType`
  - `ai_generated`
  - `saved_trip`
  - `blank_manual`
- `workspaceModeDefault`
  - `read`
  - `edit`
- `localDraftId?`
- `blankDraftId?`
- `createdAt?`
- `updatedAt`

特点：

- 用来描述“这次 Workspace 会话从哪来、默认用什么模式打开”
- 当前只有空白计划入口会真正写入 `localDraftId`
- 历史计划恢复不会写 `localDraftId`
- AI 生成入口不会写 `localDraftId`
- Explore 入口当前也不会写 `localDraftId`

### 4.7 Current Draft Reality Summary

当前本地草稿结构可以概括为：

- `trip-draft`：前置表单草稿
- `trip-plan-draft`：Explore / 计划前置草稿
- `trip-plan`：Workspace 当前编辑中的事实主计划
- `restored-saved-trip`：云端绑定 metadata
- `workspace-session`：当前会话来源与模式

当前并不存在：

- 多草稿体系
- `trip-plan` 内嵌 `cloudTripId`
- `trip-plan` 内嵌 `local_draft_id`
- 通用型“本地 draft id <-> 云端 trip id”正式映射

## 5. Current Save Flow

### 5.1 Save Entry

Workspace 顶栏的保存入口来自：

- `TripWorkspace`
- `SaveStatus`
- `SaveCurrentTripButton`

`SaveCurrentTripButton` 是当前唯一正式“保存到云端”触发器。

### 5.2 Visitor Save Behavior

游客点击保存时：

1. `SaveCurrentTripButton.handleSaveClick()`
2. 若 `authState.status !== "authenticated"`
3. 直接 `router.push(buildSaveTripLoginHref("/workspace"))`

特点：

- 会跳 `/login?returnTo=/workspace`
- 不会清空本地 `trip-plan`
- 不会清空 `trip-request`
- 不会清空 `trip-plan-draft`
- 理论上登录回来后，本地草稿仍然在

### 5.3 Authenticated Save Behavior

登录用户点击保存时：

1. `SaveCurrentTripButton` 组装：
   - `tripRequest`
   - `tripPlan`
   - `tripEnrichment`
2. 调 `persistCurrentTrip()`
3. `persistCurrentTrip()` 读取 `getSavedTripMetadata()`
4. 若已有 `savedTripId`：
   - 调 `PATCH /api/trips/[tripId]`
   - 覆盖原 trip 内容
5. 若没有 `savedTripId`：
   - 调 `POST /api/trips`
   - 新建一条 trip
6. 保存成功后 `markCurrentTripAsSaved()`
7. `SaveCurrentTripButton` 再从本地读回 metadata 更新 UI

### 5.4 New vs Update Decision

当前“新建还是覆盖”的判断完全依赖：

- `RESTORED_SAVED_TRIP_STORAGE_KEY.savedTripId`

并不依赖：

- `trip-plan` 自带字段
- `workspace-session.localDraftId`
- `trip_plans.local_draft_id`

这意味着：

- 只要本地 `savedTripId` 在，就会走覆盖
- 只要本地 `savedTripId` 不在，就会走新建
- 当前没有更细粒度的“本地草稿 identity”

### 5.5 Current Save Payload

当前 `POST /api/trips` 与整条 `PATCH /api/trips/[tripId]` 的 payload 仍然是：

- `tripRequest`
- `tripPlan`
- `tripEnrichment?`

服务端当前会写：

- `title`
- `destination_city`
- `start_date`
- `end_date`
- `days`
- `budget`
- `trip_request_json`
- `trip_plan_json`
- `enrichment_json`
- `weather_summary_json`

其中 `POST` 默认还会写：

- `source_type = "ai_generated"`
- `status = "saved"`
- `trip_preferences_json = {}`
- `local_draft_id = null`
- `last_opened_at = null`

但这里有一个重要现状：

- 不论当前 Workspace 实际来自 AI、空白手搓还是 Explore，`POST` 默认都写成 `ai_generated`
- 也不会把 `workspace-session.localDraftId` 写进 `local_draft_id`

### 5.6 Save Success Aftermath

保存成功后：

- 会写回 `restored-saved-trip`
- 会保留 `savedTripId`
- 不会清空 `trip-plan`
- 不会清空本地草稿
- 不会跳转 My Trips
- 不会改 `workspace-session.sourceType`

### 5.7 Save Failure

保存失败时：

- `SaveCurrentTripButton` 只更新按钮状态和中文错误消息
- 本地 `trip-plan` 保持不变
- 本地 `savedTripId` 不会被清空

## 6. Current Restore Flow

### 6.1 History Open Flow

当前从 My Trips 打开历史计划的链路是：

1. `/trips`
2. 可选先 `PATCH last_opened_at`
3. `GET /api/trips/[tripId]`
4. `restoreSavedTripToStorage()`
5. 跳 `/workspace`

### 6.2 What restoreSavedTripToStorage() Actually Writes

恢复时会：

- `saveTripRequest(input.trip.trip_request_json)`
- `saveTripPlan(input.trip.trip_plan_json)`
- 清掉 `trip-draft`
- 写 `trip-enrichment`
- 写 `trip-weather-summary`
- `markCurrentTripAsSaved({ savedTripId, savedTripTitle, restoredAt })`
- `setWorkspaceSessionMetadata({ sourceType: "saved_trip", workspaceModeDefault: "read" })`

### 6.3 What It Does Not Do

当前恢复时不会：

- 保留当前本地 `trip-plan`
- 比较当前本地未保存草稿与即将打开的云端 trip
- 写入 `workspace-session.localDraftId`
- 写入 `trip_plans.local_draft_id`
- 合并已有本地 draft

### 6.4 Draft Overwrite Risk

也就是说，当前用户若已经在 Workspace 里有一份尚未保存到云端的本地改动，再从 `/trips` 打开别的历史计划，会直接覆盖当前本地工作态。

当前没有：

- 冲突提示
- 放弃 / 覆盖二次确认
- 暂存当前草稿
- 草稿对比

这是 Step 8B / 后续阶段必须小心保护的风险点。

## 7. Current Visitor vs Authenticated Boundary

### 7.1 Visitors Can Still Use Workspace

游客当前仍可：

- 进入 `/workspace`
- 进入 `/result`
- 从 AI 生成进入 Workspace
- 从空白手搓进入 Workspace
- 从 Explore 导入进入 Workspace
- 在 Workspace 本地编辑

### 7.2 Visitors Cannot Save To Cloud

游客点击保存时：

- 不会调用 `/api/trips`
- 会跳 `/login?returnTo=/workspace`

### 7.3 Login Return Risk

登录回到 `/workspace` 之后，本地草稿理论上仍在，因为当前登录流程没有主动清空这些 key：

- `trip-plan`
- `trip-request`
- `trip-plan-draft`
- `workspace-session`

因此当前产品事实上已经具备：

- “游客先改本地草稿”
- “保存时被引导登录”
- “登录后回到 Workspace 再保存”

但它还缺少正式产品化保证：

- 没有显式文案说明“草稿会保留”
- 没有 local draft identity
- 没有登录前后保存成功的一致性标识

### 7.4 Logged-in Save To trip_plans

登录用户当前可以正常保存到 `trip_plans`，但保存绑定逻辑仍然偏脆弱：

- 绑定完全依赖 `savedTripId` metadata
- 没有本地 draft id 映射
- 对空白计划 / Explore 导入 / AI 生成没有差异化 source 写入

## 8. Main Risks

当前 Workspace 保存 / 恢复底座的主要风险有六个。

### 8.1 Cloud Identity Is Stored Outside trip-plan

当前 cloud trip identity 存在 `restored-saved-trip`，不在 `trip-plan` 或 `workspace-session` 主体里。

这会导致：

- 保存是否覆盖的判断依赖单独 metadata
- 如果 metadata 丢了，就会把本应更新的计划另存成新 trip

### 8.2 localDraftId Only Exists For Blank Entry

虽然 `workspace-session` 结构已经支持 `localDraftId`，但当前只有空白计划入口会真正写它。

AI 生成、Explore 导入、历史计划恢复都没有统一使用这套标识。

### 8.3 source_type Is Not Yet Trustworthy For Save

云端 `trip_plans.source_type` 虽然已经存在，但当前 `POST /api/trips` 默认一律写 `ai_generated`。

因此：

- 空白手搓保存后会被误记成 AI 生成
- Explore 导入保存后也会被误记成 AI 生成

### 8.4 History Open Directly Overwrites Local Workspace

当前从 `/trips` 打开历史计划时，会直接覆盖本地：

- `trip-plan`
- `trip-request`
- enrichment / weather

没有草稿保护。

### 8.5 No Formal local draft <-> cloud trip Mapping

当前 `trip_plans.local_draft_id` 已存在，但 Workspace 还没有接它。

所以还做不到：

- 识别“这份本地草稿已经对应哪条云端 trip”
- 更稳定地决定“再次保存是更新还是新建”

### 8.6 trip_preferences_json Is Still Empty

当前保存时总是写 `{}`，尚未接入单篇计划偏好。

这一步不应该急着接 AI merge，但 Step 8B 需要明确是否先把字段打通，还是继续留空。

## 9. Step 8B Recommended Plan

Step 8B 推荐只做“最小安全落地”，不要一口气做多草稿、冲突弹窗、复杂合并。

### 9.1 Recommended Goal

优先完成：

- 登录用户可以把当前 Workspace 保存到 `trip_plans`
- 游客保存时跳登录，且不丢本地草稿
- 保存成功后本地正式记录 cloud tripId
- 同一份计划再次保存时更新原 trip，不重复新建
- 把基础 `source_type / status / local_draft_id` 写对

### 9.2 localDraftId Recommendation

建议 Step 8B 正式接入 `localDraftId`。

做法建议：

- 统一让所有进入 Workspace 的入口都拥有 `workspace-session.localDraftId`
- AI 生成入口进入 `/workspace` 前补一个 `localDraftId`
- Explore 导入入口进入 `/workspace` 前补一个 `localDraftId`
- 历史计划恢复入口暂时可以不新建新 localDraftId，或保留当前不写，避免改动过大

更稳的第一版建议：

- 先只保证“新建 / AI / Explore 三种本地编辑来源”有 `localDraftId`
- 历史计划恢复保持现状，减少误伤

### 9.3 cloudTripId Recommendation

建议 Step 8B 不强行修改 `trip-plan` 结构去内嵌 `cloudTripId`。

更稳的做法：

- 继续沿用现有 `restored-saved-trip.savedTripId`
- 同时把 `workspace-session.localDraftId` 与云端 `trip_plans.local_draft_id` 对齐

这样可以避免：

- 改 `TripPlan` schema
- 改 Workspace 当前所有读写点

### 9.4 New vs Update Rule

Step 8B 推荐规则：

1. 若本地已有 `savedTripId`
   - 优先更新该 trip
2. 若没有 `savedTripId`
   - 新建一条 trip
3. 新建成功后
   - 写回 `savedTripId`
   - 若当前有 `workspace-session.localDraftId`，同步写入云端 `local_draft_id`

这仍然沿用当前用户心智，改动最小。

### 9.5 Guest Save After Login

Step 8B 推荐明确保留当前行为：

- 游客点击保存 -> `/login?returnTo=/workspace`
- 不清空任何本地 `trip-plan` / `trip-request`
- 登录回来后继续在同一份本地 Workspace 上保存

可以补的只是：

- 按钮文案 / 状态文案更明确
- 保存成功后把本地 `savedTripId` 固化

### 9.6 Open Saved Trip While Draft Exists

这件事建议不要在 Step 8B 做复杂弹窗与差异比对。

更稳的建议：

- Step 8B 先不改当前覆盖行为
- 若要做，最多只做轻提示文案
- 真正的冲突确认 / 合并策略留到后续

原因：

- 这部分最容易误伤 v1.8.3 的 Workspace 恢复主链路

### 9.7 source_type Recommendation

Step 8B 应该把保存时的 `source_type` 写对：

- AI 生成 -> `ai_generated`
- 空白手搓 -> `blank_manual`
- Explore 导入 -> `explore_import`

来源判断可先基于 `workspace-session.sourceType` 做映射。

当前需要注意：

- Explore 入口现在写的是 `workspace-session.sourceType = "ai_generated"`
- Step 8B 若要真正写对 `explore_import`，需要先修这个 session 来源标记
- 这是业务代码改动，但属于 Step 8B 应做的小范围修正，不是 Step 8A 要动的内容

### 9.8 status Recommendation

Step 8B 推荐先只使用两种基础值：

- 首次或正常保存 -> `saved`
- 后续从 My Trips 归档动作再写 `archived`

不建议 Step 8B 同时引入复杂 `draft` 云端语义。

原因：

- 当前本地 Workspace 草稿已经由 `localStorage` 承担
- 若马上引入“云端 draft 自动同步”，范围会骤然扩大

### 9.9 trip_preferences_json Recommendation

建议 Step 8B 继续保持轻量：

- 可以先继续写 `{}`，不急着接
- 或者只在代码结构上预留传参位置，不真正接 UI

推荐更稳方案：

- 留到 Step 9 或更后面，与单篇偏好来源一起正式接入

### 9.10 local_draft_id Mapping Recommendation

Step 8B 推荐只做单向映射：

- 本地 `workspace-session.localDraftId`
- 保存时写入云端 `trip_plans.local_draft_id`

先不要做：

- 通过 `local_draft_id` 自动回查云端 trip
- 多草稿列表
- 草稿合并

### 9.11 What Must Stay For Step 9

以下内容建议明确留给 Step 9：

- 用户长期偏好并入 AI 生成上下文
- `trip_preferences_json` 真正参与 AI 合并
- `userDefaults` 与 `tripSpecificPreferences` 优先级整合
- 保存时自动带长期偏好 / 单篇偏好的逻辑闭环

### 9.12 What To Avoid In Step 8B

Step 8B 暂时不要做：

- 多草稿管理
- 草稿冲突弹窗
- 历史计划与当前草稿差异对比
- 自动合并本地与云端版本
- 新增 migration
- 改 `TripPlan` schema
- 改 `/result` 恢复模式

## 10. Recommended Step 8B Scope

推荐 Step 8B 具体范围：

1. 保存当前 Workspace 到 `trip_plans`
   - 继续用现有 `POST /api/trips`
   - 继续用现有整条 `PATCH /api/trips/[tripId]`

2. 游客保存跳登录
   - 保持 `returnTo=/workspace`
   - 不清空本地草稿

3. 所有主要 Workspace 入口补齐基础 session source / localDraftId
   - AI 生成
   - 空白手搓
   - Explore 导入

4. 保存成功后本地正式记录
   - `savedTripId`
   - `savedAt`

5. 再次保存同一计划时覆盖已有 trip
   - 继续按 `savedTripId` 走更新

6. 保存时把 `source_type / status / local_draft_id` 写基础正确值

## 11. Step 9 Should Handle

建议留到 Step 9 或更后续处理的内容：

- AI 长期偏好接入
- `trip_preferences_json` 真正落地
- 单篇偏好和长期偏好 merge
- 打开历史计划前的草稿冲突提示
- 更复杂的草稿保护策略
- local draft 与 cloud trip 的双向恢复与回查

## 12. Summary

当前 Workspace 保存 / 恢复体系已经能工作，但“能工作”和“关系清楚、可持续增强”还不是一回事。

当前最重要的真实现状是：

- Workspace 事实主源仍然是本地 `trip-plan`
- 云端绑定主要靠单独的 `savedTripId` metadata
- 历史计划恢复会直接覆盖本地工作态
- `local_draft_id` 虽有 schema，但 Workspace 还没真正接入
- `source_type` 保存时还没有跟 Workspace 真实来源打通

因此，Step 8B 最稳的方向不是重构 Workspace，而是：

- 在不改现有恢复主链路的前提下
- 把本地 session identity、云端保存 identity、基础 metadata 写对
- 先让“保存 / 再次保存 / 登录后保存”这条线稳定下来
