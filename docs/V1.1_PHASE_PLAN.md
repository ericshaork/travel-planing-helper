# MVP v1.1 分阶段实施计划

## 1. 使用方式

v1.1 拆成 8 个可独立验收的阶段，按顺序推进。

共通规则：

- 主线始终是点击式积木编辑
- 轻量自然语言只做辅助，不做完整聊天系统
- 所有修改最终都继续复用 `/api/generate-trip`
- 当前完整 `TripPlan` 仍然是展示、复制、导出和重新生成的唯一真实来源
- 不提前做拖拽、时间轴、地图 API、数据库、局部 Patch

## 2. 阶段依赖

```text
阶段 0 文档规划与范围冻结
  -> 阶段 1 /plan 缺失信息提示增强
  -> 阶段 2 行程柜数据映射与基础组件
  -> 阶段 3 结果页改为 Day Cabinet / Time Slot / Itinerary Block
  -> 阶段 4 顶部 Day 导航与修改入口前置
  -> 阶段 5 积木操作按钮
  -> 阶段 6 快捷修改与轻量理解辅助
  -> 阶段 7 总验收、文档更新、部署准备
```

1. 轻量 UI polish

   - 不改变现有交互链路和协议

2. 文档同步
   - README 更新到真实的 MVP 1.1 状态
   - MANUAL_ACCEPTANCE 扩成 v1.1 全量验收清单
   - 在本文件记录阶段完成情况和后续待办

## 13. v1.2 移动端优先待办

这些问题已经记录，但不属于 v1.1 本轮实现：

- 进一步减少手机端结果页“一屏铺太多模块”的压力
- 研究更强的移动端信息分层，而不是继续平铺所有模块
- 评估 Day → Slot → Block 更深一层的移动端渐进展开
- 评估修改区是否适合做更轻的抽屉式入口
- 评估总览 / 预算 / 交通 / 注意事项的移动端重排顺序

结论：v1.1 已经做到可用、可测、可部署；更激进的移动端信息架构优化留到 v1.2。
