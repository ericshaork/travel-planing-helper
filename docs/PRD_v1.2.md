# PRD v1.2：Mobile First + Cabinet Identity Upgrade + Pending Changes

## 1. 文档定位

本文是 MVP v1.2 的增量产品需求，建立在已上线的 MVP 1.0 与 v1.1 之上。

- MVP 1.0 的完整生成闭环继续以 `docs/PRD.md` 为准；
- v1.1 的点击式行程编辑继续以 `docs/PRD_v1.1.md` 为准；
- 本文只定义 v1.2 的移动端体验、行程柜识别和修改篮；
- 本文不覆盖旧文档，也不授权提前实现 v1.3、v1.4 或 2.0。

当前阶段为阶段 0：规划文档与范围冻结。本阶段不修改业务代码和页面。

## 2. 产品定位

### 2.1 一句话定位

v1.2 是一次“让已有点击式编辑真正适合手机”的体验升级：

> Mobile First + Cabinet Identity Upgrade + Pending Changes

### 2.2 三条主线

1. **Mobile First**
   - `/plan` 在手机端更像短问卷，而不是缩窄后的桌面长表单；
   - `/result` 在手机端按分页式 / 分屏式浏览，不再把所有模块同级铺开成长页面；
   - 桌面端保留 v1.1 已经可接受的展开式阅读体验。
2. **Cabinet Identity Upgrade**
   - Day 更像柜门或抽屉；
   - 上午、下午、晚上更像固定格层；
   - 行程活动更像可操作积木；
   - 纸张与手账感只负责气质，柜子结构负责识别。
3. **Pending Changes**
   - 用户可以连续收集多个修改意图；
   - 修改意图先进入“待修改清单 / 修改篮”；
   - 用户检查后一次性写入现有 `RegenerateBox`；
   - 仍由用户手动确认，重新生成完整 `TripPlan`。

### 2.3 产品承诺

- 手机上更快：一屏看见更多关键内容，减少无目的长滚动；
- 结构更清楚：先看到 Day，再进入 Slot、Block 和 Detail；
- 修改更可控：连续点多个动作不会互相覆盖；
- 旧能力不回归：桌面阅读、复制、导出、天气降级和完整重排保持可用；
- 为 2.0 留接口，但不提前做 2.0。

### 2.4 Mobile Flow First

v1.2 新增一条移动端总原则：

- 从首页输入、`/plan` 补信息到 `/result` 浏览结果，移动端都优先采用“一屏一任务”；
- 移动端优先使用步骤切换、分页浏览、分屏浏览或 Tab 切换，而不是把桌面端长页面直接压缩到手机上；
- 桌面端可以继续保持完整展开式布局；
- 移动端和桌面端允许使用不同布局，但必须复用同一套数据和业务逻辑；
- 任何移动端体验优化，都不应通过新增一套 schema、接口协议或重复业务状态来实现。

### 2.5 竞品吸收边界

v1.2 会参考 Wanderlog、Layla、Roadtrippers、TripIt、Mindtrip 的方向，但只吸收当前版本真正需要的部分：

- 吸收 Mobile Flow First；
- 吸收柜子 / 抽屉式视觉；
- 吸收积木块编辑；
- 吸收 Pending Changes 修改篮；
- 保留打字修改作为辅助方式。

v1.2 明确不吸收：

- 地图工作台；
- 地图选点加入行程；
- 图片 / 短视频；
- 酒店、机票、门票等商业闭环；
- 真实保存与分享；
- Explore 发现页。

换句话说，v1.2 吸收的是“交互骨架”，不吸收“地图层、商业层和留存层”。

## 3. 为什么现在做

v1.1 已验证“点击动作 → 修改句 → 用户确认 → 完整重排”的主链路，但上线观察暴露了四个明确问题：

1. 手机端仍像桌面网页的缩小版，页面长、卡片大、信息密度低；
2. `/plan` 日期控件的中文提示不自然，底部翻页也不够像移动问卷；
3. `ItineraryBlock` 的“详情”像小文字链接，触控入口不明显；
4. 每次积木或快捷操作都会覆盖 `RegenerateBox`，无法先收集多个意图。

如果不处理这些问题，v1.1 的编辑能力虽然“存在”，但在主要使用设备上仍不顺手；直接进入真实 API 或保存分享只会把尚未稳定的编辑体验继续放大。

竞品分析也进一步支持这个判断：当前最该吸收的是移动端分层、柜子识别和编辑表达，而不是提前进入地图、商业按钮、媒体展示或保存分享。

## 4. 已有能力与复用判断

| 已有能力 | 当前实现 | v1.2 处理方式 |
|---|---|---|
| 完整旅行方案 | `TripPlan` | 继续作为展示、复制和导出的唯一真实来源 |
| 行程柜映射 | `mapTripPlanToCabinets` / `DayCabinetView` | 继续复用，不新增 AI 输出 Schema |
| 日、时段、积木 | `DayCabinet` / `TimeSlotSection` / `ItineraryBlock` | 调整布局和视觉，不复制数据模型 |
| 移动端 Day 摘要 | `DayCabinetSummary` | 作为分层浏览基础继续升级 |
| 顶部导航 | `ResultDayNav` | 继续作为 Day 与模块导航入口 |
| 修改文案 | `buildBlockModificationRequest` / `buildQuickModificationRequest` | 修改篮直接复用确定性文案 |
| 最终修改输入 | 受控 `RegenerateBox` | 继续作为唯一提交入口 |
| 重新生成 | `POST /api/generate-trip` | 协议不变，仍返回完整 `TripPlan` |
| 本地恢复 | `localStorage` 中的草稿、需求和方案 | 保持现状；修改篮不新增持久化 |

v1.2 不重建上述能力，只在它们之间增加更适合移动端的展示状态和前端修改篮状态。

## 5. 目标用户与核心场景

主要用户仍是自由行新手，尤其是：

- 用手机完成信息补充和查看行程的人；
- 只想先看每天重点，不想一次读完全部攻略的人；
- 对多个活动都有修改想法，希望一次说完再重排的人；
- 能判断“保留 / 替换 / 删除 / 加类似”，但不擅长写复杂提示词的人。

核心场景：

1. 用户在 390px 宽手机上分步补齐需求；
2. 用户先浏览旅行总览和 Day 摘要，再进入某一天；
3. 用户展开某个积木的详情；
4. 用户连续选择 A 保留、B 替换、C 加类似；
5. 用户检查修改篮，删除误选项；
6. 用户把清单写入修改框，补充一句话后手动重排。

## 6. v1.2 目标

### 6.1 P0 目标

- `/plan` 手机端采用清楚的问卷式分页体验；
- `/plan` 日期提示对中文用户自然；
- `/result` 手机端形成“总览页 → Day 页 → 预算页 → 更多页 → 修改页”的分页式浏览，并在单日页内保持“Day → Slot → Block → Detail”层级；
- Day Cabinet 形成稳定、可识别的旅行柜视觉；
- 多个积木动作和快捷修改可进入 Pending Changes；
- Pending Changes 可查看、删除单项、清空、写入修改框；
- 写入修改框后不自动提交；
- 桌面端 v1.1 主布局与主流程不回归。

### 6.2 P1 目标

- 手机端修改篮有稳定、易找到的入口；
- 折叠状态、选中状态和按钮文案具备基本可访问性；
- 360px、390px、430px 下无横向溢出；
- 为 2.0 的“导航 + 行程柜 + 修改篮”三块关系建立清楚骨架。

### 6.3 成功判断

朋友测试中应重点验证：

- 用户能否不经讲解完成手机端 `/plan`；
- 用户能否快速说出“现在在哪一步、下一步做什么”；
- 用户能否在 10 秒内找到指定 Day；
- 用户是否理解 Day、Slot、Block 的层级；
- 用户是否能连续加入三条不同修改且不丢失；
- 用户是否理解“写入修改框”不等于“已经重新生成”；
- 桌面用户是否仍能连续阅读完整方案。

## 7. 功能范围

### 7.1 `/plan` 手机端问卷式体验

手机端必须：

- 每次聚焦一个步骤，保留现有三步业务分组；
- 清楚显示当前步骤和总步骤；
- 主要内容优先，当前旅行草稿改为紧凑、可折叠摘要；
- 底部“上一步 / 下一步”具备至少 44px 的稳定触控区域；
- 缺失摘要仍可点击并定位到具体字段；
- 提交失败仍滚动到第一个错误；
- 日期字段显示“请选择日期”或中文友好的等价提示；
- 日期辅助文案说明格式为“年/月/日”，不依赖不自然的浏览器占位文字。

桌面端尽量保持 v1.1 的双栏与现有步骤布局，只修复共用的日期文案和字段可访问性。

### 7.2 `/result` 手机端分页式 / 分屏式浏览

手机端不再默认渲染“总览 + 所有 Day + 天气 + 预算 + 攻略 + 住宿 + 交通 + 修改区”的一整条长页面，而是切换为结果页内部的分页式浏览。

建议页面结构：

1. `overview`：旅行总览页，展示总览卡与 Day 入口；
2. `day-1 / day-2 / day-N`：单日行程页，每次只展示一个 Day；
3. `budget`：预算页，只展示预算相关内容；
4. `more`：更多信息页，承载天气、景点攻略、住宿、交通、注意事项；
5. `edit`：修改页，承载快捷修改、修改框、复制与 Markdown 导出。

要求：

- 顶部导航在手机端切换 `activeMobilePage`，而不是滚动到长页面锚点；
- 点击“总览”只显示总览页；
- 点击某个 Day 只显示对应 Day 页；
- 点击“预算”只显示预算页；
- 点击“更多”只显示更多信息页；
- 点击“修改”只显示修改页；
- “更多”页内部可以继续使用折叠模块，但折叠只存在于该页内部；
- BlockActions、快捷修改、RegenerateBox、复制和 Markdown 导出继续复用现有数据与提交流程；
- 点击 BlockActions 后，手机端应直接进入修改页并预填修改句；
- 复制与 Markdown 导出仍基于完整 `TripPlan`，不受当前分页状态影响；
- 桌面端继续保持 v1.1 的完整展开式阅读与滚动导航。

### 7.3 Cabinet Identity 视觉升级

视觉层级必须体现：

```text
旅行方案
└─ Day 柜门 / 抽屉
   ├─ 上午格层
   │  └─ 活动积木
   ├─ 下午格层
   │  └─ 活动积木
   └─ 晚上格层
      └─ 活动积木
```

具体要求：

- Day Summary 更像柜门或抽屉入口；
- Day Cabinet 更像固定三层柜，而不是普通大卡片；
- Slot 的边界、标签和留白状态更明确；
- ItineraryBlock 更像可选择和可操作的内容单元；
- “详情”改为明显的“查看详情 / 收起详情”按钮；
- 详情按钮触控面积足够，但不显著增加默认卡片高度；
- 继续使用米白、浅沙、浅灰绿和墨色；
- 不使用大面积蓝紫渐变、玻璃拟态、科技光效或 Emoji 功能图标；
- 不靠堆叠装饰模拟“手账”，纸感应服务于内容层级。

### 7.4 Pending Changes 修改篮

修改篮收集两类意图：

- 积木操作：不要这个、换一个、一定保留、加类似；
- 快捷修改：轻松一点、少走路、预算低一点、加美食 / 夜市、不早起。

必须支持：

- 连续加入多个修改意图；
- 查看每条意图的可读摘要；
- 删除单项；
- 清空全部；
- 对完全重复的意图去重；
- 对同一积木互相冲突的操作给出确定性处理；
- 显示当前待修改数量；
- 点击“写入修改框”后合并成一个 `modificationRequest`；
- 写入现有 `RegenerateBox`；
- 不自动请求 API；
- 用户仍可继续手动编辑修改句；
- 最终提交仍由 `RegenerateBox` 完成。

冲突原则：

- 同一积木的“不要 / 换一个 / 一定保留”属于互斥处理，后一次明确选择替换前一次；
- “加类似”属于附加意图，可与“保留”并存；
- 不同积木的动作互不覆盖；
- 快捷修改重复点击不重复加入。

### 7.5 修改篮写入与提交

```text
用户点击多个动作
→ 意图进入 Pending Changes
→ 用户查看 / 删除 / 清空
→ 点击“写入修改框”
→ 合并为 modificationRequest
→ RegenerateBox 显示可编辑文本
→ 用户手动确认
→ POST /api/generate-trip
→ 返回完整新 TripPlan
```

规则：

- 修改框为空时直接写入；
- 修改框已有用户文本时，保留原文本并追加清单内容；
- 超出既有长度上限时不静默截断，提示用户删减；
- 成功写入后清空修改篮，修改框成为待提交内容来源；
- 重新生成失败时保留旧 `TripPlan` 和修改框文字；
- 重新生成成功后清空与旧方案相关的临时修改状态。

## 8. 移动端与桌面端的关系

v1.2 明确允许移动端和桌面端使用不同布局，但不能形成两套业务逻辑：

| 维度 | 移动端 | 桌面端 |
|---|---|---|
| `/plan` | 问卷式单步聚焦、草稿折叠 | 保持 v1.1 双栏为主 |
| `/result` | 分页式 / 分屏式浏览；一页一类任务 | 保持展开式阅读为主 |
| Day Cabinet | 摘要进入单日柜 | 多日连续展示 |
| 修改篮 | 底部入口或抽屉式面板 | 固定区域或侧栏式面板 |
| 数据来源 | 当前 `TripPlan` | 当前 `TripPlan` |
| 最终提交 | `RegenerateBox` | `RegenerateBox` |

共享内容、共享状态规则、共享可访问语义；只允许布局与默认展开策略不同。

## 9. v1.2 明确不做

- 不接高德地图 API；
- 不接真实 POI；
- 不做地图基础展示或路线地图；
- 不做地图拖动、缩放、点位详情；
- 不做地图选点加入行程；
- 不做路线真实性或真实通勤计算；
- 不做景点实时开放时间；
- 不做图片 / 短视频展示；
- 不接真实票务、酒店库存、预订或支付；
- 不做保存、分享或历史版本；
- 不做已有计划导入入口；
- 不做登录、账户或数据库；
- 不做跨设备同步；
- 不做拖拽；
- 不做格子高低调整；
- 不做隔板增删；
- 不做自定义 Time Slot；
- 不做精确时间轴；
- 不做完整聊天系统或多轮消息历史；
- 不做真正局部 Patch；
- 不修改 `/api/generate-trip` 协议；
- 不修改 `TripPlan` Schema；
- 不实现真正的 constraints 持久化；
- 不把 Pending Changes 写入 `localStorage`；
- 不为未来版本重写 Provider 或项目架构。

## 10. 关键用户流程

### 10.1 手机端补充信息

```text
进入 /plan
→ 看见第 1 / 3 步
→ 填写路线与日期
→ 点击下一步
→ 填写预算与兴趣
→ 查看折叠旅行草稿
→ 最后一步确认
→ 若有错误，回到并聚焦首个字段
→ 生成行程
```

### 10.2 手机端浏览

```text
进入 /result
→ 默认进入总览页
→ 点击 Day 2
→ 只看 Day 2 页的上午 / 下午 / 晚上
→ 需要时切到预算页
→ 再切到更多页查看天气、攻略、住宿、交通和注意事项
→ 最后切到修改页做快捷修改、编辑修改框或导出
```

### 10.3 收集并提交多个修改

```text
Day 1 上午：一定保留 A
→ 修改篮 1 条
Day 2 下午：换一个 B
→ 修改篮 2 条
Day 3 晚上：加类似 C
→ 修改篮 3 条
→ 删除误选项或继续添加
→ 写入修改框
→ 补充一句“不要早起”
→ 手动点击重排
→ 完整新 TripPlan 替换旧方案
```

## 12. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 移动端和桌面端逻辑分叉 | 共享数据映射、事件和状态规则，只分布局 |
| 折叠过多导致内容难找 | 总览与 Day 始终优先，模块入口显示摘要与状态 |
| 修改篮与修改框形成两个来源 | 写入后清空修改篮，最终只提交修改框文本 |
| 旧方案引用失效 | 完整重排成功后清空 Pending Changes |
| 多意图过长 | 写入前校验既有 1000 字上限，不截断 |
| 柜子视觉过度装饰 | 先做层级、密度和触控，再做纸感细节 |
| v1.2 偷偷长成工作台 | 每阶段按“不做什么”验收 |

## 13. 通往 2.0 的位置

```text
当前 v1.1
点击动作 + Day Cabinet + 单条修改框
        ↓
v1.2
移动端分层 + 柜子识别 + 修改篮
        ↓
v1.3 / v1.4
真实性数据 + 保存分享
        ↓
2.0
导航 + 行程柜 / 时间轴 + 地图 / 地点详情 + 修改篮
```

v1.2 只建立编辑环境的骨架，不引入地图、拖拽和持久化。相对 12 个月理想状态，它解决“如何看、如何选、如何暂存修改”，仍把“真实验证、保存和直接操纵行程”留给后续版本。
