# PRD v0.1 - AI 自由行出行规划网站

## 1. 产品概述

本产品是一个面向自由行新手的 AI 出行规划网站。用户通过一句话输入和分步补充信息，提供出发城市、目的地、出行日期/天数、预算、兴趣偏好、出行风格、必去/不想去地点等信息。系统结合天气 API 和大模型能力，为用户生成一份个性化、结构清晰、可执行的城市自由行方案。

MVP v0.1 的重点不是做票务、酒店、支付或完整旅行交易平台，而是解决自由行新手最头疼的问题：不想花大量时间查攻略，但又希望得到一份适合自己、路线清楚、能直接参考的旅行方案。

## 2. 产品定位

### 2.1 产品一句话

面向自由行新手的 AI 出行规划网站，帮助用户快速生成个性化、结合天气提醒、路线顺序清楚的城市自由行方案。

### 2.2 目标用户

第一版目标用户是自由行新手，尤其是以下用户：

- 想自己出行，不想跟团；
- 不熟悉目的地，不知道该怎么安排景点；
- 不想花很多时间查攻略、地图、天气和住宿区域；
- 愿意为了节省时间和减少踩坑风险付出一点成本；
- 希望系统直接给出清楚、具体、可执行的建议。

### 2.3 核心价值

| 优先级 | 核心价值 | 说明 |
|---|---|---|
| P0 | 更个性化 | 根据用户兴趣、预算、风格、必去/不想去地点生成方案 |
| P1 | 更实时 | 接入天气 API，提供天气提醒和行程调整建议 |
| P1 | 更可靠 | 减少泛泛攻略，提供更清楚的每日行程和路线顺序 |
| P2 | 更省时间 | 减少用户查攻略、查天气、查路线的时间 |
| P2 | 更易执行 | 输出按天、时间段、景点、路线和注意事项拆分 |

## 3. 用户痛点

### 3.1 信息分散

自由行新手通常需要同时查看小红书、马蜂窝、携程、飞猪、去哪儿、高德地图、百度地图、天气 App、大众点评、12306 等多个平台，信息分散且切换成本高。

### 3.2 攻略不够个性化

网上攻略通常是别人的旅行体验，不一定适合当前用户的预算、兴趣、节奏和出行偏好。

### 3.3 路线顺序难判断

用户很难判断哪些景点应该放在同一天、先去哪后去哪、一天安排几个景点合适、景点之间如何移动。

### 3.4 天气影响容易被忽略

用户可能规划了大量户外景点，但没有提前考虑下雨、高温、台风、大风、寒潮等因素，导致实际体验变差。

### 3.5 普通 AI 攻略可执行性不足

普通 AI 生成的旅行攻略可能存在安排过满、交通时间不合理、没有说明景点之间怎么走、没有考虑预算和天气等问题。

## 4. MVP 范围

### 4.1 第一版必须做

| 模块 | 功能 | 优先级 |
|---|---|---|
| 输入 | 首页一句话描述旅行需求 | P0 |
| 输入 | 分步补充缺失信息 | P0 |

| 输入 | 必去地点、不想去地点 | P0 |
| 天气 | 根据目的地和日期获取天气信息 | P0 |
| AI 生成 | 生成每日行程安排 | P0 |
| AI 生成 | 生成景点攻略 | P0 |
| AI 生成 | 生成景点路线顺序 | P0 |
| 输出 | 卡片式 Day by Day 结果页 | P0 |
| 输出 | 天气提醒 | P1 |
| 输出 | 预算摘要 | P1 |

| 导出 | 支持复制完整方案 | P1 |
| 导出 | 支持导出 Markdown | P2 |

### 4.2 第一版不做

| 功能 | 不做原因 |
|---|---|
| 用户登录 | 增加账号、权限和数据库复杂度 |
| 收藏行程 | 依赖用户系统，后续版本再做 |
| 真实机票/酒店实时价格 | API 权限、费用和准确性要求较高 |
| 直接购买/预订 | 涉及平台合作、支付和责任问题 |
| 地图实时路线规划 | 第二版再接地图 API |
| 多人协作编辑 | 非 MVP 核心 |
| 聊天式连续修改 | 第一版只做补充要求后重新生成 |
| 精准餐厅实时状态 | 数据源复杂，第一版先做类型和区域建议 |
| App 端 | 第一版只做 Web 网站 |

## 5. 用户流程

### 5.1 主流程

1. 用户进入首页。
2. 用户用一句话描述旅行需求。
3. 系统解析用户输入，识别出发城市、目的地、日期/天数、预算、兴趣、风格等信息。
4. 系统展示已识别信息。
5. 系统通过分步问题补全缺失信息。
6. 用户确认需求。
7. 系统调用天气 API 获取目的地天气。
8. 系统调用 AI 生成旅行方案。
9. 前端展示结果页。
10. 用户可以复制或导出方案。
11. 如果不满意，用户可以补充修改要求后重新生成。

### 5.2 示例流程

用户输入：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

系统识别：

```text
出发城市：深圳
目的地：厦门
出行天数：3 天
预算：2500 元
兴趣：海边、美食、拍照
风格：轻松
```

系统追问：

```text
有没有必去的地方？
有没有不想去的地方？
住宿更希望交通方便，还是靠近景点？
能不能接受早起？
```

用户补充：

```text
必去鼓浪屿和沙坡尾，不想去寺庙，不想早起，住宿希望交通方便。
```

系统输出：旅行总览、天气提醒、住宿区域建议、每日行程、景点攻略、景点之间路线顺序、预算估算、交通/住宿平台建议。

## 6. 输入需求

### 6.1 输入方式

MVP 采用混合型输入：一句话输入 + 分步提问补充。

不采用纯长表单，也不做完整聊天式产品。

### 6.2 首页自然语言输入

首页提供一个大输入框，提示用户用一句话描述旅行计划。

示例占位文案：

```text
例如：我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边和美食，不想太累。
```

### 6.3 系统解析字段

| 字段 | 是否必填 | 示例 |
|---|---|---|
| 出发城市 | 是 | 深圳 |
| 目的地城市 | 是 | 厦门 |
| 出行日期 | 日期和天数至少一个 | 2026-07-10 |
| 出行天数 | 日期和天数至少一个 | 3 天 |
| 预算 | 是 | 2500 元 |
| 兴趣偏好 | 是 | 海边、美食、拍照 |
| 出行风格 | 是 | 轻松 |
| 必去地点 | 否 | 鼓浪屿 |
| 不想去地点 | 否 | 寺庙 |
| 住宿偏好 | 否 | 交通方便 |
| 交通偏好 | 否 | 少换乘 |
| 作息偏好 | 否 | 不想早起 |
| 特殊要求 | 否 | 不吃辣、不想爬山 |

### 6.4 分步补充问题

必问问题：

1. 你从哪个城市出发？
2. 你想去哪个城市？
3. 准备玩几天，或者具体哪几天？
4. 大概预算是多少？
5. 你更喜欢哪些类型的体验？
6. 你希望行程轻松一点，还是高效率一点？

可选问题：

1. 有没有必去地点？
2. 有没有不想去的地方？
3. 住宿更偏向交通方便、安静、景点附近，还是性价比？
4. 能不能接受早起？
5. 是否希望少走路、少换乘？

### 6.5 兴趣标签

第一版提供以下兴趣标签：

- 美食
- 自然风景
- 海边
- 历史文化
- 博物馆
- 拍照
- 购物
- 夜生活
- 小众路线
- 城市漫步
- 当地体验

### 6.6 出行风格标签

第一版提供以下出行风格标签：

- 轻松
- 高效率
- 深度游
- 低预算
- 舒适优先
- 不想早起
- 少走路

## 7. 输出需求

### 7.1 结果页结构

结果页采用卡片式 Day by Day 行程 + 预算摘要 + 天气提醒。

页面结构：

1. 顶部区域
   - 旅行标题
   - 总体摘要
   - 天气提醒
   - 预算摘要
   - 适合风格说明
2. 主体区域
   - Day 1 行程卡片
   - Day 2 行程卡片
   - Day 3 行程卡片
3. 补充区域
   - 住宿区域建议
   - 往返交通建议
   - 平台查询建议
   - 注意事项
   - 复制/导出按钮
   - 修改要求输入框

### 7.2 每日行程卡片

每一天按照时间段展示：

- 上午
- 下午
- 晚上
- 当日路线顺序
- 景点之间交通建议
- 景点攻略
- 当日注意事项

### 7.3 每个景点展示字段

| 字段 | 说明 |
|---|---|
| 景点名称 | 景点或区域名称 |
| 推荐原因 | 为什么适合用户 |
| 建议停留时间 | 例如 2-3 小时 |
| 攻略建议 | 预约、拍照、避坑、排队等建议 |
| 适合兴趣 | 与用户兴趣标签匹配 |
| 交通建议 | 从上一个地点到该地点的建议方式 |
| 天气影响 | 雨天、高温、大风等影响 |
| 替代方案 | 天气差或时间不足时的替代选择 |

### 7.4 景点路线顺序

每天必须给出明确路线顺序，例如：

```text
酒店 → 景点 A → 景点 B → 餐饮区域 → 酒店
```

同时说明这样安排的原因，例如减少跨城区移动、适合轻松路线、适合拍照时间等。

### 7.5 景点攻略要求

景点攻略应提供实际建议，而不是百科介绍。建议包含：

- 适合什么时候去；
- 建议停留多久；
- 是否需要提前预约；
- 是否适合拍照；
- 是否适合雨天；
- 是否容易拥挤；
- 附近有什么顺路点；
- 有哪些坑需要避开。

### 7.6 天气提醒要求

天气提醒应包含：

- 总体天气概览；
- 对户外景点的影响；
- 对行程顺序的调整建议；
- 需要携带的物品；
- 极端天气风险提示。

### 7.7 预算摘要要求

预算摘要第一版只做估算，不展示实时价格。

预算分类包括：

- 往返交通；
- 住宿；
- 餐饮；
- 门票/船票；
- 市内交通；
- 弹性支出。

必须提示：预算为估算值，不代表实时票价或酒店价格。

### 7.8 住宿区域建议

第一版不推荐具体酒店，只推荐住宿区域。

每个推荐区域应包含：

- 区域名称；
- 推荐原因；
- 适合用户类型；
- 交通便利性；
- 可能缺点。

### 7.9 往返交通建议

第一版不接实时票价，只提供交通方式比较和平台建议。

应包含：

- 高铁/火车/飞机/船等可选方式；
- 各方式优缺点；
- 适合当前用户的建议；
- 推荐用户去 12306、携程、飞猪、去哪儿等平台自行核实价格和时间。

## 8. 简单修改与重新生成

第一版支持用户补充新的修改要求后，系统基于原始需求重新生成完整方案。

示例：

```text
我不想去鼓浪屿了，想加一个夜市，而且整体轻松一点。
```

系统应保留原始需求，并加入新限制重新生成。

第一版暂不支持局部编辑、拖拽调整、多人协作和连续聊天记忆式修改，但需要在技术设计中预留后续扩展空间。

## 10. 异常情况

### 10.1 输入信息不足

如果用户只输入“我想去厦门玩”，系统应追问出发城市、天数、预算和兴趣偏好。

### 10.2 目的地无法识别

提示用户输入更明确的城市名，例如“厦门”“成都”“东京”。

### 10.3 日期缺失

如果没有具体日期，可以询问用户是否有具体日期。如果没有，则基于天数生成通用方案，但天气提醒只做弱提示。

### 10.4 天气 API 失败

如果天气 API 失败，不应中断整个行程生成。系统应提示实时天气获取失败，并继续生成基础行程。

### 10.5 预算过低

如果预算明显偏低，系统应提醒用户当前预算可能偏紧，并建议减少住宿标准、缩短行程或选择更低成本交通方式。

### 10.6 用户要求冲突

如果用户要求冲突，例如不想早起但想 3 天玩 15 个景点，系统应提示冲突并给出取舍建议。

## 11. 验收标准

### 11.1 功能验收

MVP v0.1 完成后应满足：

1. 用户可以输入一句旅行需求。
2. 系统可以解析并展示核心字段。
3. 用户可以补充必去/不想去地点等信息。
4. 系统可以调用天气 API。
5. 系统可以生成 Day by Day 行程。
6. 行程包含景点顺序、景点攻略、天气提醒和预算估算。
7. 用户可以复制生成结果。
8. 用户可以补充要求后重新生成。

### 11.2 体验验收

1. 用户不需要一次填写复杂长表单。
2. 结果页结构清晰，可以直接阅读。
3. 行程不要明显过满。
4. 用户能看出方案符合自己的兴趣和预算。
5. 用户能看出天气信息对行程有影响。
6. 给自由行新手使用时，用户认为比自己从零查攻略更省时间。

### 11.3 成功标准

邀请 3-5 位自由行新手试用。如果他们认为该工具比自己从零查攻略更省时间，并愿意基于生成方案继续修改或实际参考，则 MVP 成功。
