# V1.8 Explore Data & Asset Audit

## Scope

This document audits the current Explore archive data, archive field coverage,
runtime image assets, and the gap between the current implementation and the
v1.8 Explore target.

This step is audit-only:

- no business logic change
- no data mutation
- no image migration
- no drawer implementation
- no commit / no push

## 1. Current Data Source Locations

### 1.1 Explore list / detail runtime chain

Current code paths:

- Explore list API: `app/api/explore/route.ts`
- Explore detail API: `app/api/explore/[id]/route.ts`
- Repository: `lib/explore/repository.ts`
- Row mapper: `lib/explore/rows.ts`
- Client fetch layer: `lib/explore/client.ts`
- Core types: `lib/explore/types.ts`
- Zod schema: `lib/explore/schema.ts`

Runtime behavior today is split into two tracks:

1. Search / feed / detail requests use the real Explore API and Supabase table
   `explore_trip_contents`.
2. The current `/explore` homepage featured wall does not use the DB. It uses
   local mock data from `lib/explore/mock-archives.ts`.

### 1.2 Current homepage archive source

The current `/explore` homepage uses:

- `components/explore/ExploreHome.tsx`
- `components/explore/FeaturedSection.tsx`
- `lib/explore/mock-archives.ts`

Important finding:

- `FeaturedSection` imports `getMockExploreList()` directly.
- So the visible archive wall on the current homepage is mock-driven, not
  database-driven.

### 1.3 Approx. 200 AI-generated archive location

The original AI-generated archive JSON files are not present in this repo.

What the repo confirms:

- historical source folder: `../travel-content-pipeline/output/explore_plans_production/`
- importer script: `scripts/import-explore.ts`
- importer default path points to that external sibling folder
- historical import count from docs: `200` JSON files
- unique imported `external_id` count after upsert: `193`

Evidence in repo:

- `scripts/import-explore.ts`
- `docs/v1.7-phase1.3-explore-integration.md`
- `docs/v1.7-explore-json-analysis.md`

Conclusion:

- The approx. 200 AI-generated archives live in the external pipeline output
  folder, not in a checked-in `mock` or `data` directory in this repo.
- The repo keeps the importer, schema, and audit docs, but not the 200 JSON
  source files themselves.

### 1.4 Archive viewer / detail / unavailable logic

## 5. Image Mapping Rule Draft

Recommended future mapping order:

1. `coverImageUrl` exact archive-level match
2. `cityCode` / city exact match
3. `region` match
4. `theme` / taxonomy match
5. generic China-travel fallback

Suggested rule detail:

### Rule 1: archive-specific cover

If archive record has:

- `coverImageUrl`

then use it first.

### Rule 2: city exact match

Try:

- `/images/explore/cities/${cityCode}-city-card.png`
- `/images/explore/cities/${cityCode}-city-card-alt.png`

### Rule 3: region fallback

If no city image exists, map by broader region or destination family, for
example:

- coastal south
- northwest
- southwest old-town
- Jiangnan / lake-town

This requires a small explicit mapping table, not string guessing.

### Rule 4: theme fallback

If city / region both miss, fallback by theme:

- 海边 -> beach / island / coastal fallback
- 古城 -> city / old-town / heritage-like fallback
- 美食 -> food category fallback
- 山水 -> mountain / lake / waterfall / forest fallback
- 都市漫步 -> city-card / city inspiration fallback

### Rule 5: generic China fallback

If everything above misses, use:

- `public/images/explore/fallback/explore-fallback-city.png`

Recommended implementation note:

- keep foreign assets in the pool only for future international mode
- do not let current China scope fall through to foreign decorative images

- story gracefully falls back from `archiveIntro` to `summary`

- map is treated as a handoff / placeholder section, not a real live map

Current fields are not yet enough for a polished editorial drawer if you expect:

- fully reliable story text for all DB archives
- archive-specific visual covers for all entries
- explicit route field separate from itinerary
