## 2. 桌面端完整流程

建议宽度：1280px 或常用桌面窗口。

- [ ] 打开 `/`，首页正常显示，无白屏和控制台报错；
- [ ] 输入完整旅行需求并提交；
- [ ] Network 中出现本站 `POST /api/parse-trip`，返回 200；
- [ ] 成功进入 `/plan`，已识别字段与原始输入一致；
- [ ] 修改一个已识别字段，刷新后草稿仍能恢复；
- [ ] 补齐必填字段与可选偏好；
- [ ] 点击生成后出现清楚的生成状态；
- [ ] Network 中出现本站 `POST /api/generate-trip`，返回 200；
- [ ] 成功进入 `/result`；
- [ ] 完成复制、Markdown 下载和简单重新生成；
- [ ] 重新生成后页面显示完整新方案，不是局部 Patch。

推荐完整输入：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

## 3. 手机端完整流程

依次使用浏览器响应式模式检查 360px、390px、430px 宽度，并至少用一台真实手机完成一次主流程。

- [ ] 首页输入框容易点击、输入和滚动；
- [ ] 示例、兴趣标签和风格标签可换行，无横向溢出；
- [ ] 首页主按钮触控区域足够，提交后可进入 `/plan`；
- [ ] `/plan` 分步表单好点、好填，按钮不会挤在一起；
- [ ] 日期、预算、标签和长说明正常换行；
- [ ] `/result` 无横向滚动；
- [ ] 天气和预算在手机端按单列或清楚层级展示；
- [ ] 预算项、交通项、长地点名和长说明不会溢出卡片；
- [ ] 每日行程的上午、下午、晚上清楚分段，不挤成一团；
- [ ] 景点攻略、住宿建议和交通建议可连续阅读；
- [ ] “复制完整方案”按钮明显且好点击；
- [ ] Markdown 下载保持次要操作，失败时有友好提示；
- [ ] RegenerateBox 的 textarea 容易输入，提交按钮好点击；
- [ ] 重新生成 loading、错误和 appliedChanges 在小屏幕正常换行；
- [ ] 返回首页、返回 `/plan` 的入口可见。

## 4. 首页完整输入测试

输入：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

- [ ] 识别出发城市“深圳”；
- [ ] 识别目的地“厦门”；
- [ ] 识别 3 天；
- [ ] 识别预算 2500；
- [ ] 识别海边、美食、拍照；
- [ ] 识别轻松风格或对应偏好；
- [ ] 不猜测用户没有明确提供的信息；
- [ ] 提交成功后写入 `travel-planning:parsed-trip`。

## 5. 首页缺失输入测试

输入：

```text
我想去厦门玩。
```

- [ ] `/api/parse-trip` 返回结构化草稿；
- [ ] 缺少的信息以 missingFields 和 follow-up questions 表达；
- [ ] `/plan` 用人话提示出发城市、天数/日期、预算、兴趣和风格等缺失项；
- [ ] 页面不报错、不白屏、不无限 loading；
- [ ] 必填信息不足时不能生成。

同时测试：

- [ ] 空输入不会请求 API，并提示先写一句；
- [ ] 超过长度限制的输入不能提交或返回友好错误；
- [ ] 解析 API 失败时不显示堆栈、第三方响应或浏览器原始错误。

## 6. `/plan` 补充字段测试

- [ ] 已识别字段完整显示且可以修改；
- [ ] 页面按步骤补充，不是长表单；
- [ ] 只有 `days` 时允许生成通用方案；
- [ ] 只有完整日期区间时自动计算天数；
- [ ] 单边日期给出明确补充提示；
- [ ] 日期区间与 days 不一致时阻止生成；
- [ ] 预算为 0、负数或天数超过限制时阻止生成；
- [ ] 可填写必去地点、不想去地点、住宿、交通、作息和特殊要求；
- [ ] 刷新页面后从 `travel-planning:trip-draft` 恢复；
- [ ] 标准化后写入 `travel-planning:trip-request`；
- [ ] localStorage 没有草稿时显示友好空状态和返回首页入口。

## 7. `/result` 展示测试

展示顺序必须为：

1. 旅行总览；
2. 天气提醒；
3. 预算摘要；
4. 每日行程；
5. 景点攻略；
6. 住宿区域建议；
7. 交通建议；
8. 注意事项与操作区。

逐项检查：

- [ ] `dailyItinerary` 天数与行程天数一致；
- [ ] 每日路线顺序和安排理由可读；
- [ ] 上午、下午、晚上分段清楚；
- [ ] 预算明确标注为估算；
- [ ] 天气不可用时明确说明未接入或暂时不可用；
- [ ] 住宿不声称有实时库存或价格；
- [ ] 交通不声称有实时票价或余票；
- [ ] 页面展示结构化卡片，不直接倾倒模型原始长文本；
- [ ] 刷新后从 `travel-planning:trip-plan` 恢复；
- [ ] 返回首页和返回 `/plan` 的入口正常。

## 8. 复制完整方案测试

- [ ] 点击“复制完整方案”后显示成功提示；
- [ ] 粘贴到备忘录或文本编辑器，内容为完整 Markdown；
- [ ] 包含标题、总览、天气、预算、每日行程、住宿、交通和注意事项；
- [ ] 中文没有乱码；
- [ ] 复制内容与当前页面的新方案一致；
- [ ] 拒绝剪贴板权限或模拟失败时显示友好提示，不显示原始错误。

生产环境需通过 HTTPS 再验一次复制。

## 9. Markdown 下载测试

- [ ] 点击“下载 Markdown”后得到 `.md` 文件；
- [ ] 文件名不包含 Windows 非法字符；
- [ ] 文件以 UTF-8 打开，中文无乱码；
- [ ] 内容与复制结果一致；
- [ ] 内容直接来自当前 `TripPlan`，不是从 DOM 拼接；
- [ ] 手机浏览器下载失败时有友好提示，并可改用复制。

## 10. 简单重新生成测试

在 `/result` 输入：

```text
第二天太满了，少安排两个点，加一个夜市，别太早起。
```

- [ ] 空修改要求不会提交；
- [ ] 超长修改要求有明确提示；
- [ ] 请求仍发往本站 `POST /api/generate-trip`；
- [ ] 请求体包含 `tripRequest`、当前 `previousPlan` 和 `modificationRequest`；
- [ ] 成功响应返回完整新 `TripPlan`；
- [ ] 页面立即更新为新方案；
- [ ] `travel-planning:trip-plan` 被新方案覆盖；
- [ ] `appliedChanges` 存在时显示在重新生成区域附近；
- [ ] 新方案复制和下载使用更新后的 `TripPlan`；
- [ ] 刷新 `/result` 后仍显示新方案。

## 11. 重新生成失败保留旧方案

可临时断开网络、停止本地服务或让测试桩返回错误，再提交修改要求。

- [ ] 显示“先保留原方案”一类友好提示；
- [ ] 页面仍展示旧方案；
- [ ] `travel-planning:trip-plan` 没有被清空或写入半成品；
- [ ] 复制和下载仍使用旧方案；
- [ ] “再试一次”可用；
- [ ] 错误中没有堆栈、Key 或第三方原始响应。

## 12. localStorage 损坏测试

在浏览器 DevTools 的 Application / Storage 中，分别把下列 key 改为无效 JSON 或不符合 Schema 的内容，然后刷新对应页面：

- `travel-planning:parsed-trip`
- `travel-planning:trip-draft`
- `travel-planning:trip-request`
- `travel-planning:trip-plan`

预期：

- [ ] 页面不白屏、不抛未处理异常；
- [ ] 损坏的 key 会被清理；
- [ ] `/plan` 显示无草稿状态；
- [ ] `/result` 显示无结果状态；
- [ ] 用户可以返回首页重新开始。

## 13. 无结果空状态测试

清除四个 `travel-planning:*` key 后直接访问：

- [ ] `/plan` 显示友好提示和“回首页重新写”入口；
- [ ] `/result` 显示“还没有行程结果”；
- [ ] `/result` 提供返回 `/plan` 和返回首页入口；
- [ ] 两个页面都不会白屏、报错或无限 loading。

## 14. Network 检查

打开 DevTools Network，清空记录后完成一次完整流程。

- [ ] 业务 API 请求只有本站 `/api/parse-trip` 和 `/api/generate-trip`；
- [ ] 重新生成复用 `/api/generate-trip`，没有 `/api/regenerate-trip`；
- [ ] 浏览器没有直接请求 OpenAI、SiliconFlow、DeepSeek、通义或其他模型平台；
- [ ] 浏览器没有直接请求 QWeather 或其他天气平台；
- [ ] 除页面、静态资源和本站 API 外，没有意外第三方请求；
- [ ] API 响应中没有 API Key、模型 Base URL、模型名或第三方原始响应。

## 15. 安全检查

- [ ] 客户端组件没有导入 `lib/ai/client.ts`、Provider、Planner 或任何 `server-only` 文件；
- [ ] 客户端组件没有导入 `lib/weather/client.ts` 或 WeatherProvider；
- [ ] `.env.local` 被 `.gitignore` 忽略；
- [ ] `node_modules`、`.next` 和构建输出被忽略；
- [ ] `.env.example` 不含真实 Key、真实 Base URL或固定模型名；
- [ ] 服务端错误不向用户返回堆栈、Key 或第三方响应细节；
- [ ] 用户输入和 modificationRequest 有长度限制；
- [ ] AI 输出必须通过 Zod 校验，修复最多一次；
- [ ] 最终实时天气相关内容只来自 WeatherProvider；
- [ ] Mock 模式不需要任何真实 API Key。

## 17. 朋友测试问题清单

邀请 3—5 位自由行新手，不先讲解功能，让他们独立完成一次规划。记录：

1. 首页第一眼能看懂网站做什么吗？
2. 知道输入框里该写什么吗？
3. 哪一步停顿最久？为什么？
4. `/plan` 的问题是否像补信息，而不是填复杂表格？
5. 生成等待是否让人知道系统正在做什么？
6. 结果页最先看哪一块？是否找得到每日路线？
7. 天气、预算和实时信息的可信边界是否清楚？
8. 行程是否太满？有没有明显绕路或不合偏好的安排？
9. 手机上有没有难点、难滚、误触或横向滚动？
10. 是否能成功复制方案？
11. 是否能看懂并使用“想改哪儿？直接说。”？
12. 相比从零查攻略，是否明显省时间？
13. 哪句话或哪个组件最像模板站、最不自然？
14. 哪个信息缺失会阻止他们真的参考这份行程？
15. 他们愿不愿意基于这份方案继续修改或实际出行？
