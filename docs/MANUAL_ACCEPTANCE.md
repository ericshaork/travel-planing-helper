# MVP v1.2 手动验收清单

本清单用于 v1.2 的总验收、部署前检查，以及朋友二轮测试前的最后核对。

当前范围覆盖：

- Mobile Flow First
- `/plan` 手机问卷式补信息
- `/result` 手机分页式浏览
- Cabinet Identity 视觉升级
- Pending Changes 修改篮
- 轻工作台雏形

不覆盖：

- v1.2.5 真实 AI 接入
- v1.3 高德 POI / 路线 / 地图 / 天气

## 2. 开始前命令检查

先确认以下命令全部通过：

```powershell
npm.cmd run lint
npm.cmd run typecheck
npm.cmd run build
npm.cmd test
```

默认推荐先在 Mock 模式跑完一轮：

```env
USE_MOCK_AI=true
USE_MOCK_WEATHER=true
```

## 3. 主流程冒烟

- [ ] 打开 `/` 正常，无白屏、无明显报错
- [ ] 输入完整旅行需求并提交
- [ ] Network 中出现 `POST /api/parse-trip`
- [ ] 成功进入 `/plan`
- [ ] 补齐信息后点击生成
- [ ] Network 中出现 `POST /api/generate-trip`
- [ ] 成功进入 `/result`
- [ ] 能继续复制、导出、修改、重新生成

推荐输入：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

## 4. A. 首页 mobile

至少在 360px / 390px / 430px 检查：

- [ ] 打开首页后，不需要长滑就能看到输入框
- [ ] 打开首页后，不需要长滑就能看到“先排一版”
- [ ] 示例和偏好不会默认撑成长页面
- [ ] 点击真实示例后可回填输入框
- [ ] 兴趣 / 风格折叠区可展开收起
- [ ] 提交后能正常进入 `/plan`

## 5. B. `/plan` mobile

- [ ] 三步问卷式结构清楚
- [ ] 第 1 步只聚焦路线和时间
- [ ] 第 2 步只聚焦预算和兴趣
- [ ] 第 3 步只聚焦风格和确认
- [ ] 每一步更像一屏任务，不像长表单缩小版
- [ ] 上一步 / 下一步 / 生成按钮容易点击
- [ ] 当前旅行草稿在 mobile 端是紧凑摘要或折叠卡
- [ ] 日期提示中文友好
- [ ] 日期辅助文案清楚说明“年/月/日”
- [ ] 缺失字段摘要正常显示
- [ ] 点击缺失项能定位到对应字段
- [ ] 提交失败仍会滚到首个错误
- [ ] 补完字段后可正常生成

## 6. C. `/result` mobile

- [ ] `activeMobilePage` 为分页式浏览
- [ ] 总览页不展示全部 Day
- [ ] 总览页能看见清楚的下一步入口
- [ ] Day 1 页只显示 Day 1
- [ ] Day 2 页只显示 Day 2
- [ ] Day N 页只显示对应 Day
- [ ] 预算页只显示预算相关内容
- [ ] `more` 页承载天气 / 攻略 / 住宿 / 交通 / 注意事项
- [ ] `edit` 页是“修改工作台”
- [ ] mobile 端不再是一条把所有模块铺开的长页面

## 7. D. Cabinet

- [ ] Day Summary 更像小柜门 / 抽屉入口
- [ ] DayCabinet 更像单日三层柜
- [ ] 上午 / 下午 / 晚上格层边界清楚
- [ ] ItineraryBlock 更像可操作积木
- [ ] “查看详情 / 收起详情”按钮明显、可点
- [ ] 长地点名或长说明不会明显溢出

## 8. E. Pending Changes

- [ ] 点击“不要这个”会加入修改篮
- [ ] 点击“换一个”会加入修改篮
- [ ] 点击“一定保留”会加入修改篮
- [ ] 点击“加类似”会加入修改篮

- [ ] Day 页能看到“已选 N 项待修改”
- [ ] 修改页可查看待修改清单
- [ ] 修改页可删除单项
- [ ] 修改页可清空全部
- [ ] 修改页可写入修改框
- [ ] 写入修改框后不自动提交
- [ ] RegenerateBox 仍是唯一提交入口

## 9. F. QuickActions

## 13. J. 已知限制

## 14. 重新生成与边界

- [ ] 空修改句不会提交
- [ ] 超长修改句会提示用户删减
- [ ] 正常提交时仍只请求 `POST /api/generate-trip`
- [ ] 请求体仍使用 `tripRequest + previousPlan + modificationRequest`
- [ ] 成功后会更新整份 `TripPlan`
- [ ] 失败时保留旧方案
- [ ] 不新增 `/api/patch-trip`
- [ ] 不新增 `/api/chat`
- [ ] 浏览器没有直接请求模型平台
- [ ] 浏览器没有直接请求天气平台

## 15. localStorage 与空状态

- [ ] 刷新 `/plan` 后，草稿可恢复
- [ ] 刷新 `/result` 后，最近一次结果可恢复
- [ ] 清空 `travel-planning:*` 后访问 `/plan`，有友好空状态
- [ ] 清空 `travel-planning:*` 后访问 `/result`，有友好空状态
- [ ] localStorage 损坏时不白屏

## 16. 朋友二轮测试重点

邀请 3～5 位自由行新手，不先解释功能，让他们自己操作。

重点记录：

1. 他们打开首页后，是否能马上理解“先写一句，再补缺信息”
2. 他们在 `/plan` 是否知道自己现在在哪一步
3. 他们是否能在手机上快速切到某一天
4. 他们是否理解“待修改清单”和“修改框提交”的区别
5. 他们是否会在 Day 页连续选多个积木，而不是每次都跳去重排
6. 他们是否能理解 edit 页就是修改工作台
7. 他们是否强烈要求真实地图 / 保存分享 / 已有计划导入
