# MVP 1.1 手动验收清单

本清单用于阶段 7 的总验收、部署前检查，以及朋友二轮测试前的最后核对。

本轮验收范围只覆盖 MVP 1.1，不提前验 v1.2。

- 验收人：
- Commit / 构建版本：

- 设备或分辨率：
- 测试地址：
- AI 模式：Mock / Real
- 天气模式：Mock / Real

## 2. 开始前命令检查

先确认以下命令全部通过：

```powershell
npm.cmd run lint
npm.cmd run typecheck
npm.cmd run build
npm.cmd test
```

默认推荐先在 Mock 模式跑完一轮：

```env
USE_MOCK_AI=true
USE_MOCK_WEATHER=true
```

## 3. 主流程冒烟

- [ ] 打开 `/` 正常，无白屏、无明显报错
- [ ] 输入完整旅行需求并提交
- [ ] Network 中出现 `POST /api/parse-trip`
- [ ] 成功进入 `/plan`
- [ ] 补齐信息后点击生成
- [ ] Network 中出现 `POST /api/generate-trip`
- [ ] 成功进入 `/result`
- [ ] 能继续复制、导出、修改、重新生成

推荐输入：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

## 4. `/plan` 缺失信息提示

用明显缺字段的输入测试：

```text
我想去厦门玩。
```

检查：

- [ ] 顶部出现缺失信息摘要
- [ ] 摘要会显示出发城市
- [ ] 摘要会显示目的地
- [ ] 摘要会显示预算
- [ ] 摘要会显示天数或日期
- [ ] 摘要会显示兴趣
- [ ] 摘要会显示出行风格
- [ ] 点击任一缺失项，会切到对应 step
- [ ] 点击后会滚动到对应字段附近
- [ ] 对应字段有柔和高亮
- [ ] 字段旁边能看到具体提示文案
- [ ] 提交失败时不会请求 `/api/generate-trip`
- [ ] 提交失败时自动滚到第一个错误
- [ ] 页面已有填写内容不会丢失

再补齐字段，确认：

- [ ] 缺失摘要会收缩或消失
- [ ] 字段错误随修复而消失
- [ ] 所有字段补完后可以正常生成

## 5. `/result` 结构和内容

- [ ] 顶部有总览
- [ ] 天气提醒正常显示
- [ ] 预算摘要正常显示
- [ ] 每日行程以 Day Cabinet 展示
- [ ] 每天固定有上午 / 下午 / 晚上三个 Time Slot
- [ ] 每个 ItineraryItem 显示为 Itinerary Block
- [ ] 景点攻略、住宿建议、交通建议、注意事项仍然存在
- [ ] 页面不是一大段纯 AI 文本

## 6. Day Cabinet 与手机端浏览体验

至少检查 360px、390px、430px 和一个桌面宽度。

- [ ] 页面整体无横向滚动
- [ ] 手机端默认先看到多日摘要
- [ ] 点某天摘要可以展开该天
- [ ] 再点同一天可收起
- [ ] 桌面端仍可连续阅读多天
- [ ] 长地点名、长说明文字不会把卡片撑出屏幕
- [ ] 空的时间格子会显示留白状态，不会报错

## 7. 顶部 ResultDayNav

- [ ] 导航可见，且包含总览 / Day / 预算 / 交通 / 修改
- [ ] 当前选中项有清楚状态
- [ ] 手机端按钮区域可以左右滑动
- [ ] 不出现难拖的原生横向滚动条
- [ ] 页面本身不会被导航撑出横向滚动
- [ ] 导航为轻量 sticky，向下滚动后仍可使用
- [ ] sticky 不遮挡主要内容
- [ ] sticky 不盖住修改输入区
- [ ] 点击“总览”回到总览
- [ ] 点击某个 Day 会展开并定位到对应 Day
- [ ] 点击“修改”会跳到修改区

## 8. 积木操作

随便展开一个 Day，检查每个积木：

- [ ] 有“不要这个”
- [ ] 有“换一个”
- [ ] 有“一定保留”
- [ ] 有“加类似”

对每种操作分别检查：

- [ ] 点击后不会自动提交
- [ ] 点击后不会立刻请求 `/api/generate-trip`
- [ ] 页面会跳到修改区
- [ ] RegenerateBox 会出现新的中文修改句
- [ ] 修改句包含 Day
- [ ] 修改句包含时段
- [ ] 修改句包含地点名

## 9. 快捷修改

检查五个按钮：

- [ ] 轻松一点
- [ ] 少走路
- [ ] 预算低一点
- [ ] 加美食 / 夜市
- [ ] 不早起

逐个确认：

- [ ] 点击后不会自动提交
- [ ] 点击后会填入现有 RegenerateBox
- [ ] 生成的是清楚的中文修改句
- [ ] 修改句符合该按钮的真实含义

## 10. RegenerateBox 状态同步

先造出错误状态，再检查：

- [ ] 在修改框为空时点击提交，出现错误
- [ ] 再点击某个积木操作，旧错误会消失

再造出成功状态：

- [ ] 成功重排一次后能看到 successMessage
- [ ] 如果有 `appliedChanges`，会显示在修改区
- [ ] 再点击某个积木操作或快捷修改，旧 success / appliedChanges 会消失

同时确认：

- [ ] 用户手动输入仍然正常
- [ ] 用户仍可改写自动生成的修改句

## 11. 重新生成

在 `/result` 的修改框测试：

- [ ] 空修改句不会提交
- [ ] 超长修改句会有友好提示
- [ ] 正常提交时仍只请求 `POST /api/generate-trip`
- [ ] 请求体仍使用 `tripRequest + previousPlan + modificationRequest`
- [ ] 成功后会更新整份 `TripPlan`
- [ ] `appliedChanges` 与本次变化一致
- [ ] 复制和 Markdown 导出都随新方案更新

失败场景再检查：

- [ ] 失败时保留旧方案
- [ ] 失败时不会把页面变成半成品
- [ ] 可以直接再试一次

## 12. 复制与 Markdown 导出

- [ ] 点击“复制完整方案”成功
- [ ] 粘贴结果是完整 Markdown
- [ ] 中文无乱码
- [ ] 内容基于当前完整 TripPlan
- [ ] 折叠状态、导航状态、未提交修改草稿不会混进导出
- [ ] Markdown 下载成功
- [ ] 下载文件名正常

## 13. localStorage 与空状态

- [ ] 刷新 `/plan` 后，草稿可恢复
- [ ] 刷新 `/result` 后，最近一次结果可恢复
- [ ] 清空 `travel-planning:*` 后访问 `/plan`，有友好空状态
- [ ] 清空 `travel-planning:*` 后访问 `/result`，有友好空状态
- [ ] localStorage 损坏时不白屏

## 14. Network 与边界

完成一轮主流程后检查 Network：

- [ ] 浏览器只直接请求本站页面、静态资源和本站 API
- [ ] 首页解析只请求 `POST /api/parse-trip`
- [ ] 首次生成和重新生成都只请求 `POST /api/generate-trip`
- [ ] 没有新增 `/api/patch-trip`
- [ ] 没有新增 `/api/chat`
- [ ] 浏览器没有直接请求模型平台
- [ ] 浏览器没有直接请求天气平台

## 15. 范围边界核对

确认 v1.1 没有偷偷长成 v1.2 / 2.0：

- [ ] 没有拖拽
- [ ] 没有自定义时间轴
- [ ] 没有新增 / 删除 Time Slot
- [ ] 没有地图 API
- [ ] 没有真实距离计算
- [ ] 没有实时开放时间
- [ ] 没有登录 / 收藏 / 数据库
- [ ] 没有真正局部 Patch
- [ ] 没有完整聊天系统

## 16. 朋友二轮测试重点

邀请 3～5 位自由行新手，不先解释功能，让他们自己操作。

重点记录：

1. 他们能不能在 `/plan` 很快看懂还差什么
2. 他们会不会主动点缺失摘要去补信息
3. 他们在手机上能不能快速找到某一天
4. 他们会不会优先点积木操作，而不是自己从零写修改句
5. 他们能不能理解“先填入修改框，还没真正提交”
6. 他们对五个快捷修改的理解是否一致
7. 他们是否强烈要求保存 / 分享
8. 他们是否强烈要求更自由地改每天节奏和格子结构
