# v1.7 Tech Audit

## 1. 目标与范围

本次审计只关注 `v1.7 Phase 1` 的技术准备，不改任何业务代码、UI 风格、样式、API 或数据库。

目标：

- 盘点当前项目结构
- 确认现有页面与缺失页面
- 评估现有旅行数据结构是否足够支撑 Workspace 升级与 Explore 接入
- 识别过大的页面 / 组件
- 给出 Explore 内容接入建议
- 输出后续 `v1.7 Phase 1` 的实施建议与风险点

## 2. 当前状态

### 2.1 项目目录结构

当前主目录结构清晰，已形成典型的 Next.js App Router 项目分层：

- `app/`
  - 页面与 API route
- `components/`
  - `auth/`
  - `layout/`
  - `map/`
  - `trip/`
  - `trips/`
  - `workspace/`
- `lib/`
  - `ai/`
  - `map/`
  - `poi/`
  - `route/`
  - `supabase/`
  - `trip/`
  - `trips/`
  - `utils/`
  - `weather/`
- `supabase/`
  - 当前主要用于 v1.6 持久化相关
- `docs/`
  - 已有产品与阶段文档

结论：

- 现有目录结构对继续演进是够用的
- `workspace` 能力已经开始组件化
- `explore` 相关目录、类型、数据层目前还不存在

### 2.2 当前页面盘点

| 页面 | 路由 | 状态 | 备注 |
| --- | --- | --- | --- |
| landing | `/` | 已存在 | 由 `WorkspaceLanding` 渲染，当前首页与 workspace 概念混在一起 |
| create | `/create` | 已存在 | 创建页可用 |
| result | `/result` | 已存在 | 结果页可用，但已承载大量工作台逻辑 |
| workspace | `/workspace` | 名义存在，实际缺失 | 当前只是 `redirect("/")` 的兼容入口，不是独立工作台 |
| explore | `/explore` | 缺失 | 无页面、无组件、无数据层 |

| login | `/login` | 已存在 | v1.6 Auth 页面 |

结论：

- `landing`、`create`、`result`、`trips` 已落地
- `workspace` 还没有成为真正的核心入口
- `explore` 目前完全未开始

### 2.3 当前工作台状态

虽然 v1.6 已经有大量 workspace 组件，但系统层面还没有真正进入“Workspace 是核心入口”的阶段：

- `/` 首页直接复用 `WorkspaceLanding`
- `/workspace` 仅做重定向
- `/result` 才是真正承载工作台逻辑的页面
- 当前旅行工作态仍以 `localStorage` 为主
- 云端持久化只覆盖“保存计划 / 打开历史计划”

这意味着：

- 现在的 workspace 更像一组 UI 组件和概念
- 还不是一条独立、稳定、可扩展的产品主入口

## 3. 当前数据结构审计

### 3.1 Trip 相关主结构

当前核心类型位于 [lib/trip/types.ts](/C:/Users/10200/Desktop/travel_planing/lib/trip/types.ts)。

#### `TripPlan`

当前已具备：

- `tripTitle`
- `summary`
- `destination`
- `days`
- `travelStyleSummary`
- `weatherSummary`
- `budgetSummary`
- `hotelAreaAdvice`
- `transportAdvice`
- `dailyItinerary`
- `generalTips`
- `warnings`

结论：

- 对 AI 结果页展示已经足够
- 对“可编辑旅行工作台”也基本够用
- 但对 Explore 内容体系还不够，因为缺少内容资产级元数据

当前缺失的 Explore 倾向字段：

- 内容唯一 slug
- 城市 / 国家 / 区域标签
- 主题标签集合
- 内容来源标记（AI pipeline / 人工精选 / 编辑）
- 发布状态
- 封面图 / 缩略图
- 摘要卡片文案
- Explore 列表排序字段
- 浏览量 / 推荐权重 / 上架时间等内容运营字段

#### `DailyItinerary`

当前已具备：

- `day`
- `date`
- `theme`
- `routeOrder`
- `routeReason`
- `morning`
- `afternoon`
- `evening`
- `dailyTips`

结论：

- 对当前结果页和 Day Cabinet 足够
- `theme` 已存在，这一点对接 pipeline 很关键
- 如果 Explore 只是复用 `TripPlan` 内容本体，这个结构本身不需要先改

### 3.2 POI

当前 POI 结构分两层：

1. `lib/poi/types.ts`
   - `PoiCandidate`
   - `PoiSearchResult`
2. `lib/trip/enrichment-types.ts`
   - `MapPoint`

当前已具备：

- 名称
- 地址
- 城市
- 坐标
- provider
- resolved 状态
- 与 day / slot / item 的关联

结论：

- 支撑结果页地图联动已经足够
- 支撑 Explore 静态内容展示也基本够用
- 但如果以后 Explore 要做“按城市 / 主题 / 片区筛选内容卡”，还需要内容级 POI 聚合字段

### 3.3 Food

当前没有独立 `Food` 数据结构。

现状：

- 美食点只是 `ItineraryItem.type = "food"` 的一种 itinerary item

结论：

- 对当前结果页足够
- 对 Explore 不够理想

原因：

- Explore 往往需要单独抽“美食亮点”或“吃什么”卡片
- 仅依赖 itinerary item 会导致内容展示层难以做轻量聚合

建议：

- `v1.7 Phase 1` 先不要改主 schema
- 但在 Explore 方案中预留 `derived content fields`
  - 例如从 `TripPlan` 派生 `foodHighlights[]`

### 3.4 Weather

当前天气结构有两层：

1. 原始天气能力：`lib/weather/types.ts`
   - `WeatherForecast`
   - `WeatherDay`
   - `WeatherAlert`
   - `TripWeatherSummary`
2. Trip 内部展示结构：`TripPlan.weatherSummary`

当前已具备：

- 预报天级信息
- 预警
- 降雨概率
- 风力
- impacts / warnings

结论：

- 对主流程和结果页够用
- 后续只需要在展示层收束重点，不急着改 schema
- Explore 如果只展示“这条内容对应的天气建议风格”，不需要单独扩结构

### 3.5 Map

当前地图结构分布在：

- `lib/map/`
- `components/map/`
- `components/workspace/InspectorMapPreview.tsx`
- `lib/trip/enrichment-types.ts` 中的 `MapPoint`

当前已具备：

- AMap SDK loader
- 基础地图容器
- marker layer
- active marker
- fit view
- unresolved 点位兜底
- 与 itinerary block / point list 的联动

结论：

- 当前 map 能力足够支撑 result/workspace 继续演进
- Explore 第一阶段不需要独立地图编辑能力
- Explore 如需卡片级地图，只要做轻量只读地图或静态点位摘要即可

## 6. Explore 接入方案分析

### 6.1 方案 A：JSON 直接导入

做法：

- 将 pipeline 生成的 JSON 文件直接放入主站仓库
- 页面在构建时或服务端读取这些 JSON
- `/explore` 直接渲染内容卡片和详情页

优点：

- 起步快
- 不需要立刻建表
- 本地调试简单
- 适合验证内容结构是否可用

缺点：

- 200 份以上内容会迅速变成仓库内的大型静态资产
- 过滤、排序、分页、上线状态管理都不方便
- 更新内容需要重新部署
- 不利于后续内容运营
- 不利于多来源内容接入
- 不适合做“探索内容库”而只适合做“静态样例集”

适用场景：

- 本地 smoke test
- schema 对齐验证
- 内容展示原型

### 6.2 方案 B：Database 存储

做法：

- 主站新增 Explore 内容表
- pipeline 生成 JSON 后，通过导入脚本或后台同步写入数据库
- 主站按查询接口拉取内容列表与详情

优点：

- 适合 200+ 内容规模
- 支持筛选、排序、分页、推荐位
- 支持发布状态和内容治理
- 不把大型 JSON 直接塞进前端 bundle 或仓库主路径
- 便于后续接“主题页 / 城市页 / 内容模板”

缺点：

- 需要新增表结构与导入流程
- phase 1 成本高于直接 JSON

适用场景：

- 真正的 Explore 内容体系
- 后续持续扩充内容
- 需要内容运营、审核、推荐、检索

### 6.3 推荐结论

推荐：**主站正式方案使用 Database 存储**。

理由：

1. Explore 的定位不是 demo，而是长期内容资产层。
2. 当前已有约 200 份初始内容，已经超过“直接塞几个 JSON 文件”的合理规模。
3. 后续 Explore 很可能需要：
   - 按城市筛选
   - 按主题筛选
   - 推荐内容
   - 上下架
   - 内容更新
4. 这些需求天然更适合数据库而不是静态 JSON 目录。

### 6.4 实际落地建议

建议采用“两段式”：

#### 阶段 1：JSON 作为导入中间产物

- 不直接作为线上渲染主源
- 仅作为 pipeline 输出格式
- 用于导入、校验、预览和回归

#### 阶段 2：主站以 DB 为正式内容源

- Explore 页面只读数据库
- pipeline 输出通过脚本写入 DB
- 页面层不直接依赖本地 JSON 文件

## 7. v1.7 Phase 1 实施建议

### 7.1 先做“技术解耦”，不要直接冲 UI

在不改 UI 风格的前提下，优先做这些技术准备：

1. 明确 `/`、`/workspace`、`/result` 的职责边界
2. 抽离 result 页的状态控制层
3. 定义 Explore 独立内容 schema
4. 设计 Explore 内容读取层
5. 为 pipeline -> 主站导入预留接口

### 7.2 Workspace 侧建议

- 不要继续把 workspace 概念混在 landing 里
- `/workspace` 需要从兼容 redirect 升级为真实入口
- `/result` 应逐步从“结果页”过渡为“工作台的某种工作态”

推荐方向：

- `/`：landing / marketing / create entry
- `/workspace`：真正的旅行工作台入口
- `/result`：过渡态，后续考虑并入 workspace 流程

### 7.3 Explore 侧建议

先补齐以下技术基础：

- `lib/explore/types.ts`
- `lib/explore/schema.ts`
- `lib/explore/repository.ts`
- `app/explore/page.tsx`
- `app/explore/[slug]/page.tsx` 或等价详情路由

推荐 Explore 内容模型分层：

1. `ExploreContentMeta`
   - slug
   - city
   - country / region
   - title
   - summary
   - theme
   - tags
   - cover image
   - status
   - source
2. `ExploreContentPayload`
   - 原始 `TripPlan`
   - 可选 enrichment
   - 可选派生字段（food highlights / poi highlights）

### 7.4 数据策略建议

- 不要直接复用 `trip_plans` 表承载 Explore 内容
- Explore 是“公共内容资产”，不是“用户私有旅行计划”

建议独立内容表，例如：

- `explore_trip_contents`

避免把用户数据模型和内容资产模型混到一起。

- `/workspace` 不是实际入口，只是 redirect

- `Food` 没有独立内容结构，不利于 Explore 内容卡抽取

### P2

- `/trips`、`/create` 等页面仍有阶段性文案残留
- workspace 组件存在一定“概念先行、结构未收敛”的迹象

## 9. 风险点

### 9.1 把 Explore 直接做成静态 JSON 页面

风险：

- 短期看似快，长期维护成本高
- 很快会卡在筛选、排序、分页、上架流程上

### 9.2 把 Explore 内容直接塞进 `trip_plans`

风险：

- 混淆用户私有数据与公共内容资产
- 后续权限、运营、推荐逻辑会变脏

### 9.3 在 `v1.7 Phase 1` 直接大改 UI

风险：

- 会让 workspace 解耦和 explore 数据层准备被打断
- 容易把“视觉施工”和“架构准备”混成一轮

### 9.4 不先拆 result 页

风险：

- 后续 workspace 独立入口建设会持续被 1600+ 行页面拖慢
- map / pending changes / state restore / mobile desktop 双视图会越来越难维护

## 10. 结论

当前项目已经具备：

- AI 生成主链路
- 结果页
- 天气
- 地图基础能力
- 用户保存与历史计划
- 一套初步的 workspace 组件

但还没有具备：

- 真正独立的 workspace 入口
- Explore 页面
- Explore 数据模型
- Explore 内容存储与导入机制

因此，`v1.7 Phase 1` 的最佳策略不是先做 UI，而是先完成三件事：

1. 拆解并稳定 `/result` 的工作台状态层
2. 明确 `/`、`/workspace`、`/result` 的职责分工
3. 为 Explore 建立独立的内容 schema、数据层与导入路径

推荐结论：

- Workspace：继续做，但要从 `/result` 中逐步抽离为真正入口
- Explore：正式方案应走 **Database 存储**
- JSON：保留为 pipeline 输出与导入中间产物，而不是主站正式内容源
