# V1.8.4 Auth, User Center, And Settings Product Plan

## 1. Purpose

本阶段只做方案设计，不修改业务代码，不新增 migration，不 commit，不 push。

目标是基于 `docs/V1.8.4_AUTH_SETTINGS_AUDIT.md`，产出一份可直接进入实施阶段的产品规则与数据方案，确保 v1.8.4 能把 Wanderly 从“本地可用工作台”升级成“正常网站产品”。

本方案重点回答：

- Auth 用什么形态对用户最清楚；
- Guest 与 Logged-in 的权限边界怎么划；
- Supabase 最小要补哪些表和字段；
- localStorage 和云端的真相边界怎么收口；
- 长期偏好和单篇计划偏好如何同时作用于 AI；
- My Trips 和 Settings 需要怎样落地；
- Step 3-8 应按什么顺序施工。

## 2. Auth Strategy

### 2.1 推荐总方案

推荐方案：

- v1.8.4 继续保留 Supabase magic link 作为主登录方式；
- 不在 v1.8.4 强行新增密码注册 / 密码登录；
- 但要把当前 magic link 包装成用户能理解的“注册 / 登录”正常流程；
- UI 上对外表达为：
  - `注册 / 登录`
  - `输入邮箱，发送登录链接`
  - `首次使用会自动创建账号`
  - `已有账号也用同一个入口继续登录`

这是当前最稳妥的方案。

原因：

- 当前项目已有 Supabase Auth + OTP 基础，技术路径现成；
- v1.8.4 主线重点是用户中心、保存、找回、Settings，不是完整认证系统重构；
- 如果现在再补密码模式，会增加页面、错误处理、找回密码、邮箱验证、Supabase 配置和产品解释成本；
- magic link 对早期产品足够可用，只要包装得正常，用户可理解。

### 2.2 是否继续保留 magic link

建议保留，并作为 v1.8.4 默认主方案。

推荐对外文案：

- 主按钮：`注册 / 登录`
- 表单标题：`用邮箱继续`
- 输入框提示：`输入常用邮箱，给你发一封登录链接`
- 辅助说明：`第一次使用会自动创建 Wanderly 账号`

避免继续使用明显开发态表述：

- “这轮只做最轻的 magic link”
- “phase 4 再做保存”
- “先把账号接上”

### 2.3 是否新增密码注册 / 密码登录

v1.8.4 不推荐立即上密码模式。

结论：

- `A. 是否继续保留 magic link`：是，保留
- `B. 是否新增密码注册 / 密码登录`：本版本不新增
- `C. 如果新增需要什么`：见 2.6，作为 v1.9+ 预案
- `D. 如果暂时不新增密码模式怎么包装`：见 2.4 和 2.5

### 2.4 v1.8.4 的 Auth 页面建议

保留轻量，但要变成正常网站模式：

1. `/login`
   - 标题：`注册 / 登录`
   - 说明：`输入邮箱，给你发登录链接。第一次使用会自动创建账号。`
   - 表单：邮箱输入 + `发送登录链接`
   - 成功态：`登录链接已发送，请去邮箱里打开`
   - 页脚小字：`登录后会回到你刚刚的页面`

2. `/login/sent` 可选
   - 如果想让成功态更稳，可做一个发信成功页
   - 不是必须

3. 不单独做 `/register`
   - 可用 `/login` 同页承载注册 / 登录
   - 顶部文案写成“注册 / 登录”，不要强拆两页

### 2.5 登录状态展示规则

Header / UserMenu 中明确三态：

- 未登录：显示 `注册 / 登录`
- 已登录：显示 `我的行程` + 简短账号态
- 展开菜单：
  - 当前邮箱
  - `我的行程`
  - `设置`
  - `退出登录`

同时要向用户解释 session 持久化：

- 不需要额外文案解释技术细节；
- 但退出登录入口必须始终明确；
- 登录页和用户菜单都不要让用户猜“怎么退出”。

### 2.6 如果未来新增密码模式，需要哪些内容

如果 v1.9 以后要加密码模式，最少需要：

页面：

- `/login`
- `/register`
- `/forgot-password`
- `/reset-password`

Supabase：

- Email + Password provider 开启
- Email confirmation / reset 邮件模板
- redirect URL 配置

客户端方法：

- `signUpWithPassword`
- `signInWithPassword`
- `requestPasswordReset`
- `updatePassword`

产品规则：

- 注册成功后的邮箱确认
- 未确认邮箱时的状态提示
- 找回密码失败与过期处理

但这些不建议放进 v1.8.4 主线。

### 2.7 登录后回跳规则

统一规则：

- 所有登录门槛动作都应带 `returnTo`
- 登录成功后优先回到原操作页面
- 如果原操作依赖当前本地草稿，则继续使用本地草稿，不丢内容

推荐回跳目标：

- 从保存按钮去登录：回 `/workspace`
- 从 My Trips 去登录：回 `/trips`
- 从 Settings 去登录：回 `/settings`
- 从历史计划打开去登录：回 `/trips`

## 3. Guest And Logged-In Rules

### 3.1 Guest 用户允许做什么

未登录用户允许：

- 浏览 Landing
- 浏览 Explore
- 使用一次 AI 生成计划
- 查看结果
- 进入本地 Workspace
- 创建空白工作台
- 短期编辑本地草稿
- 在本机刷新后继续当前 local workspace session

这条原则不能被 Auth 打断。

### 3.2 Guest 用户需要登录后才能做什么

未登录用户遇到以下动作时，应提示登录：

- `保存到我的旅程`
- `打开我的旅程`
- `打开历史计划`
- `保存 Settings`
- `跨设备找回计划`
- `长期保存当前草稿`
- `重命名 / 删除已保存计划`

### 3.3 按钮级规则

建议统一成下面这套：

1. Workspace 顶部保存按钮
   - 未登录：点击后去 `/login?returnTo=/workspace`
   - 已登录：直接保存到 Supabase

2. Header 中 `我的行程`
   - 未登录：允许点，但进入登录门槛页或 `/login?returnTo=/trips`
   - 已登录：进 `/trips`

3. Header / 用户菜单中的 `设置`
   - 未登录：点击后去 `/login?returnTo=/settings`
   - 已登录：进 `/settings`

4. `/trips` 页面
   - 未登录：显示登录提示卡
   - 已登录：显示真实列表

5. 历史计划打开
   - 未登录：不开放
   - 已登录：加载 Supabase -> 恢复本地工作态 -> 跳 `/workspace`

### 3.4 登录后本地草稿如何保留

关键原则：

- 登录不应清空 localStorage 草稿
- 登录后回到原页面时，继续读取当前本地 workspace session
- 如果用户刚刚是从“保存计划”去登录，登录回来后应仍停在当前 Workspace，并允许立即保存

建议规则：

1. 登录前不动当前 local draft
2. 登录成功后只刷新 auth state
3. 回到 `returnTo`
4. 页面基于现有 localStorage 草稿继续显示
5. 用户再次点 `保存计划` 时，才把该草稿提升为云端 trip

### 3.5 “长期编辑”产品定义

v1.8.4 里应把“长期编辑”明确解释成：

- 未登录：只保证当前浏览器 / 当前设备的临时工作态
- 已登录并保存：才保证云端可找回、跨设备可继续

不要让用户误以为：

- “我没登录也一定能长期保存”

## 4. Supabase Data Plan

### 4.1 总体策略

继续沿用 Supabase，不切数据库。

最小数据改造原则：

- 尽量在现有 `profiles` 和 `trip_plans` 基础上扩展；
- 新增一张 `user_settings` 表承载长期偏好；
- 单篇计划偏好优先放在 `trip_preferences_json`；
- 暂不把所有偏好字段拆成很多列；
- 优先保证产品可用和查询可实现，再考虑 v1.9 细拆。

### 4.2 profiles 改造建议

当前已有字段：

- `id`
- `email`
- `display_name`
- `avatar_url`
- `created_at`
- `updated_at`

推荐扩展为：

- `display_name`
- `avatar_url`
- `onboarding_completed boolean default false`
- `default_workspace_mode text nullable`
- `updated_at`

字段建议：

- `display_name`
  - 必留
  - 用于账号设置和用户展示
- `avatar_url`
  - 可保留
  - v1.8.4 可不做头像上传，只预留
- `onboarding_completed`
  - 可加，但由于产品规则明确“不做 welcome 引导页”，它不是首要字段
  - 建议仅作为未来扩展预留，可选
- `default_workspace_mode`
  - 不建议放在 `profiles`
  - 更适合进入 `user_settings`

推荐结论：

- `profiles` 只保留“账号身份基础资料”
- 不把大部分产品设置塞进 `profiles`

### 4.3 user_settings / user_preferences 表建议

建议新建 `user_settings` 表，1:1 绑定用户。

推荐字段：

- `user_id uuid primary key references auth.users(id)`
- `travel_budget_preference text`
- `travel_pace text`
- `interests jsonb`
- `companions text`
- `wake_preference text`
- `transport_preference text`
- `ai_detail_level text`
- `ai_style_preference text`
- `use_defaults_for_ai boolean not null default true`
- `workspace_default_mode text`
- `workspace_map_ratio numeric`
- `show_map_detail_overlay boolean not null default true`
- `created_at timestamptz`
- `updated_at timestamptz`

字段说明：

- `travel_budget_preference`
  - 长期预算风格，不是本次旅行预算
  - 例如：`budget_friendly / moderate / premium`
- `travel_pace`
  - 例如：`slow / balanced / packed`
- `interests`
  - 长期兴趣数组
  - 用 `jsonb` 足够
- `companions`
  - 例如：`solo / couple / family / friends`
- `wake_preference`
  - 例如：`early / normal / sleep_in`
- `transport_preference`
  - 例如：`walk_more / transit_first / taxi_ok / train_first`
- `ai_detail_level`
  - 例如：`brief / balanced / detailed`
- `ai_style_preference`
  - 例如：`practical / journal / hidden_gems`
- `use_defaults_for_ai`
  - Create 默认是否自动带入长期偏好
- `workspace_default_mode`
  - `read / edit`
- `workspace_map_ratio`
  - 保存用户常用地图比例
- `show_map_detail_overlay`
  - 控制右侧浮层默认是否展示

推荐原因：

- 这些是“长期默认偏好”
- 与单篇 trip 区分明确
- 查询频率低、修改频率低，独立表最合适

### 4.4 trip_plans 改造建议

建议在 `trip_plans` 上增加：

- `source_type text`
- `status text`
- `trip_preferences_json jsonb`
- `local_draft_id text`
- `last_opened_at timestamptz`
- 继续保留 `updated_at`

推荐取值：

- `source_type`
  - `ai_generated`
  - `blank_manual`
  - `explore_imported`
- `status`
  - `draft`
  - `saved`
  - `archived`

字段作用：

- `source_type`
  - 支撑 My Trips 筛选
  - 支撑后续产品分析
- `status`
  - 区分还在编辑的云端草稿与正常保存计划
- `trip_preferences_json`
  - 存单篇计划偏好
  - 不建议 v1.8.4 全拆成列
- `local_draft_id`
  - 便于首保存时建立 local -> cloud 对应
- `last_opened_at`
  - 便于 My Trips 排序和“最近打开”

### 4.5 哪些字段必须迁移

v1.8.4 必须迁移：

- `user_settings` 新表
- `trip_plans.source_type`
- `trip_plans.status`
- `trip_plans.trip_preferences_json`
- `trip_plans.local_draft_id`
- `trip_plans.last_opened_at`

建议但非强制：

- `profiles.onboarding_completed`

### 4.6 哪些适合先放 JSON

适合先放 JSON：

- `trip_preferences_json`
- `user_settings.interests`

原因：

- 偏好字段后续还会调整；
- v1.8.4 更重要的是先把“长期 vs 单篇”两层关系立住；
- v1.9 再按查询需要拆细。

### 4.7 哪些适合 v1.9 再拆细

建议留到 v1.9 再拆：

- companions 是否拆 enum 列
- AI 风格偏好是否拆多字段
- Workspace 细粒度 UI 偏好
- trip_plans 的统计字段、聚合字段

## 5. localStorage Boundary

### 5.1 Supabase 存什么

Supabase 应存：

- 登录用户 profile
- 用户长期 settings
- 已保存 trip
- trip source / status
- 单篇 trip preferences
- 最近打开时间

### 5.2 localStorage 存什么

localStorage 只存：

- 未登录临时草稿
- 当前 workspace session
- 当前 read / edit 模式
- 地图比例等 UI 状态
- 当前搜索输入 / 临时搜索结果缓存

### 5.3 未登录草稿登录后如何保存到 Supabase

建议流程：

1. 用户未登录时在本地创建 / 编辑草稿
2. 点“保存到我的旅程”
3. 跳登录
4. 登录成功回到 `/workspace`
5. 继续读取本地草稿
6. 再次触发保存时：
   - 创建新的 `trip_plans` 行
   - 写入 `source_type / status / trip_preferences_json / local_draft_id`
7. 返回 `tripId`
8. 把本地 metadata 更新为“该草稿已对应云端 trip”

### 5.4 savedTripId / localDraftId / blankDraftId 对应关系

建议统一成：

- `localDraftId`
  - 本地工作态主标识
  - 所有本地创建草稿都应有
- `blankDraftId`
  - 只作为空白创建来源标记
  - 不是长期主键
- `savedTripId`
  - 云端 `trip_plans.id`

规则：

- 未保存前：只有 `localDraftId`
- 空白创建时：`localDraftId + blankDraftId`
- 首次保存后：
  - 本地 metadata 增加 `savedTripId`
  - 云端 `trip_plans.local_draft_id = localDraftId`
- 再次更新时：
  - 以 `savedTripId` 为主真相

### 5.5 如何避免旧 localStorage 误导用户

建议加三条防线：

1. UI 上明确区分：
   - `仅保存在本机`
   - `已保存到我的旅程`

2. 本地 metadata 增加时间和状态：
   - `updatedAt`
   - `savedTripId`
   - `lastSyncedAt`

3. 打开已保存 trip 时：
   - 以云端内容恢复为主
   - 本地旧缓存只作为临时工作态容器，不作为真相来源

推荐原则：

- 云端 trip 打开后，应该覆盖当前本地 trip 内容
- 不允许用户误把旧 local 缓存当成云端最新版本

## 6. Default Preferences vs Trip Preferences

### 6.1 两层偏好定义

#### Settings 长期偏好

表示用户平时旅行的一般偏好。

来源：

- `user_settings`

用途：

- Create 页面默认带入
- AI 生成时提供长期背景

#### 单篇计划偏好

表示当前这趟旅行的具体偏好。

来源：

- Create / Plan 填写
- Explore 导入草稿
- Workspace 当前草稿
- `trip_request`
- `trip_preferences_json`

用途：

- 当前 trip 的真实生成约束

### 6.2 冲突规则

明确规则：

- prompt 同时接收两层偏好
- 如果冲突，以单篇计划偏好为准
- 单篇计划偏好默认不写回 Settings
- 只有用户明确点击 `保存为我的默认偏好` 才更新长期偏好

### 6.3 Create 页面规则

Create 页面应提供轻量开关：

- `使用我的默认偏好`
- `本次不使用`

默认建议：

- 已登录且 `use_defaults_for_ai = true` 时，默认勾选
- 未登录时不显示此能力

### 6.4 Prompt merge 方案

推荐在 server 侧生成前做 merge，不在前端拼 prompt。

#### 输入层

1. `userDefaults`
   - 来自 `user_settings`
2. `tripSpecificPreferences`
   - 来自 `TripRequest + trip_preferences_json`

#### 归一化层

先转成统一偏好对象，例如：

```ts
interface ResolvedTripPreferences {
  budgetPreference?: string;
  pace?: string;
  interests?: string[];
  companions?: string;
  wakePreference?: string;
  transportPreference?: string;
  aiDetailLevel?: string;
  aiStylePreference?: string;
  mustVisitPlaces?: string[];
  avoidPlaces?: string[];
  specialRequirements?: string;
}
```

#### merge 规则

```ts
resolved = {
  ...userDefaults,
  ...tripSpecificScalarFields,
  interests: mergeList(userDefaults.interests, tripSpecific.interests),
}
```

其中：

- 标量冲突：单篇优先
- 数组：
  - `mustVisitPlaces / avoidPlaces` 直接以单篇为主
  - `interests` 可做“默认 + 单篇去重合并”，也可完全以单篇覆盖

推荐简单版本：

- 若单篇字段存在，直接覆盖长期字段
- 若单篇字段缺失，则回退长期默认

#### 传给模型的结构

建议不要只把 merged 结果直接塞进原 `TripRequest`，而是显式传：

```json
{
  "tripRequest": {...},
  "userDefaults": {...},
  "resolvedPreferences": {...}
}
```

这样模型和后续调试都更清楚。

## 7. My Trips Product Rules

### 7.1 最小功能范围

v1.8.4 的 My Trips 最小应包含：

- 列表
- 打开
- 删除
- 重命名
- 搜索
- 筛选

### 7.2 筛选维度

推荐筛选：

- 全部
- 草稿
- 已保存
- AI 生成
- 空白手搓
- Explore 导入

对应依赖字段：

- `status`
  - `draft / saved / archived`
- `source_type`
  - `ai_generated / blank_manual / explore_imported`

### 7.3 搜索规则

搜索范围：

- `title`
- `destination_city`

v1.8.4 先做前端输入 + API query 即可，不必做复杂全文检索。

### 7.4 重命名规则

建议：

- 在 My Trips 卡片或详情入口中提供 `重命名`
- 实际更新 `trip_plans.title`
- 不改 `source_type`
- 不改原 request / plan 内容

### 7.5 打开规则

打开已保存计划时：

- 从 Supabase 拉取 trip
- 覆盖恢复到本地 workspace session
- 跳 `/workspace`
- 默认 Read Mode
- 更新 `last_opened_at`

### 7.6 删除规则

删除规则：

- 仅删除云端 `trip_plans` 行
- 如果当前本地正在打开的就是这条 trip，则清空 `savedTripId` 绑定
- 不要误删 unrelated 本地 draft

## 8. Settings IA

### 8.1 页面定位

`/settings` 应该像旅行档案设置页，不是 SaaS dashboard。

要求：

- 全中文
- 轻、清楚、像个人旅行默认偏好整理页
- 不做复杂管理后台感

### 8.2 信息架构

#### 1. 账号设置

- 邮箱
- 登录状态
- 昵称
- 退出登录
- 删除数据入口预留

#### 2. 旅行偏好

- 预算偏好
- 出行节奏
- 兴趣偏好
- 同行人
- 作息偏好
- 交通偏好

#### 3. Workspace 偏好

- 默认进入阅读 / 编辑
- 默认地图比例
- 默认天数
- 是否显示地图详情浮层

#### 4. AI 偏好

- 生成详细程度
- 是否自动使用默认偏好
- 是否偏小众
- 是否少走路
- 是否优先交通方便

### 8.3 页面交互建议

设置页交互原则：

- 分组保存或整页保存都可以
- v1.8.4 推荐整页保存优先，降低实现复杂度
- 顶部给出保存状态：
  - `未保存`
  - `保存中`
  - `已更新`

### 8.4 文案风格

文案要像整理旅行习惯，不像后台表单。

例如：

- `你平时更喜欢哪种节奏？`
- `默认想多走一点，还是更省体力？`
- `以后新建计划时，要不要先套用这些习惯？`

## 9. Migration And API Impact

### 9.1 需要新增的 migration

v1.8.4 预计需要：

1. 新建 `user_settings`
2. 扩展 `trip_plans`
3. 必要时小幅扩展 `profiles`
4. 补对应 RLS policy
5. 补 `handle_new_user` 后的设置初始化逻辑（可选）

### 9.2 需要新增或改造的 API

建议新增：

- `GET /api/me`
- `PATCH /api/me/profile`
- `GET /api/settings`
- `PATCH /api/settings`

建议改造：

- `POST /api/trips`
  - 支持 `source_type / status / trip_preferences_json / local_draft_id`
- `PATCH /api/trips/[tripId]`
  - 支持更新 title、status、preferences、last_opened_at
- `GET /api/trips`
  - 支持搜索和筛选参数

### 9.3 需要新增的 client 方法

建议新增：

- `getCurrentUserProfile()`
- `updateCurrentUserProfile()`
- `getCurrentUserSettings()`
- `updateCurrentUserSettings()`
- `renameSavedTrip()`
- `listSavedTrips({ query, status, sourceType })`

### 9.4 Auth 侧 client 方法

若 v1.8.4 继续 magic link 主方案，客户端只需规范化现有接口：

- `requestAuthLink(email, returnTo)`
- `signOut()`
- `getAuthStatus()`

不用在本阶段强加 password auth client。

## 10. Risks And Guardrails

### 10.1 不能破坏的主链路

必须保护：

- Landing -> Explore
- Landing / Create -> AI 创建 -> Workspace
- Landing / Create -> 空白创建 -> Workspace
- Explore -> 导入到工作台 -> Workspace
- My Trips -> Workspace
- Workspace Read/Edit 切换
- `/result` 兼容入口

### 10.2 Auth 改造护栏

- 不要把登录门槛前置到 Landing / Explore / 首次 AI 生成
- 不要让未登录用户无法进入 Workspace 本地编辑
- 不要让登录过程清掉当前本地草稿

### 10.3 数据层改造护栏

- 不切数据库
- 不重写 trip 核心 schema
- 不把 workspace session 真相整体搬进 Supabase
- 不让旧 localStorage 继续和云端真相抢主导权

### 10.4 Settings 护栏

- 不把单篇 trip 偏好自动回写长期设置
- 不把 Settings 变成 Welcome/onboarding 流程
- 不让 `/settings` 变成传统 SaaS 配置中心

## 11. Recommended Step 3-8 Plan

### Step 3：注册 / 登录 / 退出正常化

修改范围：

- `app/login/*`
- `components/auth/*`
- Header / UserMenu
- Supabase auth client 封装

目标：

- 把 magic link 包装成正常“注册 / 登录”页面
- 明确退出登录
- 明确登录态显示
- 保留 `returnTo`

风险：

- 容易误伤现有 Header 和 Workspace 顶部登录状态
- 容易把开发说明文案残留到正式 UI

验收标准：

- 用户能理解如何注册 / 登录 / 退出
- 登录成功后能回到原页面
- 页面文案全中文

### Step 4：登录门槛与保存规则

修改范围：

- Workspace 保存按钮
- `/trips` 未登录态
- `/settings` 未登录态
- 登录回跳逻辑
- local draft -> cloud save 流程

目标：

- 把需要身份的动作统一挂到登录门槛上
- 保留 guest 的轻体验
- 登录后不丢当前草稿

风险：

- 最容易破坏未登录 Workspace 使用体验
- 最容易出现回跳后草稿丢失

验收标准：

- 未登录仍能创建和编辑本地草稿
- 保存时提示登录
- 登录后继续保存，不丢内容

### Step 5：My Trips 完善

修改范围：

- `trip_plans` 字段
- `/api/trips`
- `/trips`
- 相关 client / list 组件

目标：

- 增加重命名、搜索、筛选
- 支持 source_type / status 分类

风险：

- trips API 改动会影响 Workspace 打开历史计划
- 旧数据兼容需要处理

验收标准：

- My Trips 可按类型筛选
- 可搜索、重命名、删除、打开
- 不破坏历史计划进入 `/workspace`

### Step 6：Settings 页面骨架

修改范围：

- 新建 `/settings`
- settings API
- user settings client 方法
- 用户菜单入口

目标：

- 先把账号设置、旅行偏好、Workspace 偏好、AI 偏好页面搭起来

风险：

- 容易做成传统后台配置页
- 容易和单篇偏好混淆

验收标准：

- 已登录用户可读写 settings
- 页面全中文
- 风格仍像 Wanderly，不像 SaaS 后台

### Step 7：用户偏好接入 AI

修改范围：

- `user_settings` 读取
- Create 默认值
- plan / generate-trip server 侧 merge
- prompt 输入结构

目标：

- AI 同时读取长期偏好与单篇偏好
- 冲突时单篇优先

风险：

- 最容易影响当前 AI 生成结果稳定性
- merge 规则不清楚会出现“用户明明选了这次偏好，却还被默认偏好覆盖”

验收标准：

- 已登录用户可选择是否使用默认偏好
- AI 生成能体现长期偏好，但不会压过本次旅行需求
- 单篇偏好不会自动污染 Settings

风险：

验收标准：

- Landing / Create / Explore / Workspace / My Trips / Settings 都可用
- Guest / Logged-in 权限边界清楚
- 保存、找回、偏好影响 AI 全链路成立
- 不破坏 `/workspace` 主入口与 `/result` 兼容
