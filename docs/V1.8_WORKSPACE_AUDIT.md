# V1.8.3 Workspace Audit

## 1. Audit Scope

本次只做 Workspace 当前状态审计。

- 不修改任何业务代码。
- 不删除任何业务文件。
- 不 commit。
- 不 push。
- 只允许新增或覆盖本文件 `docs/V1.8_WORKSPACE_AUDIT.md`。

本次审计主要基于以下代码范围：

- 路由入口：`app/workspace/page.tsx`、`app/result/page.tsx`、`app/trips/page.tsx`、`app/create/page.tsx`、`app/plan/page.tsx`
- Workspace 主体：`components/workspace/*`
- 地图能力：`components/map/*`、`lib/map/*`
- 状态与存储：`hooks/useTripWorkspaceState.ts`、`hooks/useTripMap.ts`、`lib/trip/storage.ts`、`lib/trips/*`

## 2. Corrected Workspace Target

本次先明确校正目标：

Workspace 不是“左侧详情页 + 右侧小地图插件”。

Workspace 应该是：

- 左侧计划工作区
- 右侧完整地图工作区

左侧负责：

- 行程阅读
- Day Timeline
- 地点卡片
- 后续编辑入口与积木编辑

右侧负责：

- 完整地图容器
- marker 展示与选中态
- Day 联动
- 地点空间关系理解
- 后续搜索、距离、路径、路线优化入口

因此，右侧地图如果只是 map preview、inspector、辅助卡片或插件位，就还没有达到 v1.8.3 的 Workspace 目标。

## 3. Current Route And Entry Flow

### 3.1 当前 Workspace 页面在哪里

当前存在两个相关路由：

- `/workspace` -> `app/workspace/page.tsx`
- `/result` -> `app/result/page.tsx`

两者都渲染同一个页面壳：

- `components/workspace/WorkspaceRoutePage.tsx`

差异仅在于传入：

- `/workspace` 使用 `mode="workspace"`
- `/result` 使用 `mode="result"`

结论：

- 当前没有两套不同的 Workspace / Result 页面。
- `/workspace` 与 `/result` 只是同一壳层的两个入口。

### 3.2 `/workspace` 是否存在，实际入口是什么

`/workspace` 已存在，而且 AI 生成完成后已经进入这里。

当前 AI 主链路：

1. `/create`
2. `/plan`
3. `/workspace`

证据：

- `app/plan/page.tsx` 成功生成后执行 `router.push("/workspace")`

### 3.3 `/result` 是否仍作为结果页或兼容入口存在

是，而且不只是兼容壳，当前仍承担真实入口职责。

证据：

- `lib/trips/open-flow.ts`：历史计划恢复后导航到 `/result`
- `lib/trips/save-client.ts`：保存登录回跳默认是 `/result`
- `components/trips/SavedTripCard.tsx`：文案仍写“回到现有 `/result` 工作台”
- `app/trips/page.tsx`、`app/login/page.tsx` 仍保留 `/create -> /plan -> /result` 的旧链路叙事

结论：

- `/result` 当前仍是有效真实入口。
- Workspace 主入口并没有统一。

### 3.4 AI 创建完成后现在跳转到哪里

当前 AI 生成完成后跳转到：

- `/workspace`

这条链路已经对齐目标方向。

### 3.5 My Trips / 历史计划点击后现在跳转到哪里

当前历史计划打开链路是：

1. `/trips`
2. 调用 `openSavedTripIntoWorkspace(trip.id)`
3. 拉取 `/api/trips/[tripId]`
4. 把 `trip_request_json`、`trip_plan_json`、`enrichment_json`、`weather_summary_json` 回填到 `localStorage`
5. 最后跳转 `/result`

结论：

- 历史计划并不是通过明确的 Workspace trip route 打开
- 而是“先回填本地存储，再打开共用页”
- 历史计划目前仍未统一到 `/workspace`

### 3.6 是否已有“空白创建计划 / 手动创建计划 / 从 0 开始”的入口

当前没有真正可用的“从 0 直接进入 Workspace”的入口。

现状：

- `/create` 页面有 `ai-assisted` 与 `self-directed`
- 但 `self-directed` 在 `lib/trip/create-modes.ts` 中 `available: false`
- `/create` 当前也不会创建 blank workspace draft
- `WorkspaceRoutePage` 的空状态只会引导去 `/create` 或 `/explore`，不会建立空白计划

结论：

- 空白创建入口缺失
- 手动创建入口缺失
- 从 0 进入 Workspace 的真实链路缺失

这必须列为后续必补入口问题。

## 5. Read Mode And Edit Mode Status

### 5.1 当前是否默认进入 Read Mode

是。

证据：

- `useTripWorkspaceState.ts` 默认 `workspaceMode = "read"`

### 5.2 Read Mode 当前实际完成到什么程度

Read Mode 已有较完整骨架：

- 顶部 WorkspaceTopBar
- 左侧 Day Tabs
- 左侧 Day Timeline / DayCabinet
- 右侧地图相关信息
- itinerary block 与 map point 的联动

但它完成的是：

- “左计划 + 右路线检查面板”

还不是：

- “左计划 + 右完整地图工作区”

### 5.3 Read Mode 是否已经形成“左计划 + 右完整地图”的阅读体验

否。

当前 Read Mode 已形成：

- 左计划 + 右地图预览 / route inspector

没有形成：

- 左计划 + 右完整地图工作台

差距主要在右侧：

- 地图不是主容器
- 地图不占主要高度
- 地图周围被多块说明卡分割

### 5.4 Edit Mode 是否存在入口、状态、组件或接口

存在。

当前已有：

- `workspaceMode = "edit"`
- `WorkspaceTopBar` 切换按钮
- `ModificationPanel`
- `useTripModification`
- `RegenerateBox`
- `ExportActions`

### 5.5 Read/Edit 切换逻辑是否真实存在

存在，且是实际状态切换，不是占位文案。

切换会影响：

- itinerary block action 是否显示
- `ModificationPanel` 是否渲染
- 顶部按钮状态

### 5.6 Edit Mode 目前是否只是占位

不是纯占位，但也还不是目标态。

当前 Edit Mode 本质是：

- 收集修改意图
- 通过 AI 重新生成完整 TripPlan

它不是：

- 真正的手帐积木编辑器
- 真正的块级操作工作台

### 5.7 后续最小可用 Edit Mode 需要补哪些东西

至少需要补：

- Day 级编辑上下文
- block 选中后的明确编辑面
- 计划块与地图点位的共用选中上下文
- 手动空白计划进入后的可编辑骨架
- 不依赖“全量 AI 重生”的最小编辑状态

### 5.8 空白计划进入 Workspace 后，理论上应该进入 Read Mode 还是 Edit Mode

建议：

- 空白计划进入 Workspace 时默认进入 `Edit Mode`

原因：

- 空白计划没有足够内容支撑 Read Mode
- Read Mode 更适合已有 itinerary 的阅读与校对
- 空白计划的第一任务是创建内容，不是阅读内容

更合理的策略是：

- AI 生成计划 / 历史计划 -> 默认 Read Mode
- 空白手动计划 -> 默认 Edit Mode，或 Edit-ready 空状态

## 6. Day And Map Synchronization

### 6.1 Day Tab 是否存在

存在。

- `WorkspaceDayTabs.tsx`

### 6.2 Day Tab 切换是否真实影响左侧日程

是。

切换 `activeWorkspaceDayNumber` 后，左侧 `activeWorkspaceCabinet` 会切换到对应 Day。

### 6.3 Day Tab 切换是否真实同步右侧地图

是，但同步的是“右侧当前 Day 路线信息 + preview map”，不是完整地图工作区。

同步链路：

1. 切换 Day
2. `desktopSelectedInsight` 重算
3. `MapPanel` 收到新的 `insight`
4. `InspectorMapPreview` 改为当前 Day 的 `mapPoints`

### 6.4 地图展示的是全部地点、当天地点，还是 mock 数据

当前展示的是：

- 当前 Day 的地点

不是全部 trip 地点，也不是静态 mock 卡片。

### 6.5 marker 编号是否和左侧地点顺序一致

大体一致，但依赖 enrichment 输出顺序。

当前 marker 编号来源：

- `buildAmapMarkerLayer(points)` 中按 `points.forEach((point, index) => order = index + 1)`

如果 `mapPoints` 顺序与左侧一致，则 marker 编号也一致。

### 6.6 地点顺序是否和左侧日程一致

通常接近一致，但不是强约束。

原因：

- 顺序真源在 `insight.mapPoints`
- block 与 point 匹配依赖 `day + slot + itemIndex + placeName`
- generic placeName 或 unresolved point 会降低稳定性

### 6.7 点击左侧地点是否能高亮地图 marker

可以，已实现。

链路：

- 点击左侧 itinerary block
- `useTripMap.selectItineraryBlock`
- 找到 matching map point
- 设置 `activeMapPointId`
- 地图 marker active

### 6.8 点击地图 marker 是否能同步选中左侧地点卡

可以，已实现。

链路：

- 点击 marker
- `onMarkerClick(pointId)`
- `useTripMap.selectMapPoint`
- 反推对应 block id
- 左侧 block active

### 6.9 若目前没有地图同步，请明确指出阻塞点

当前不是“没有同步”，而是“同步已存在，但有明显阻塞”：

- `tripPlan` 是主数据源，map enrichment 是后置异步结果
- 历史 trip 虽写入 `trip-enrichment`，但页面不直接复用，仍重新请求 `/api/enrich-trip`
- `findMatchingMapPoint()` 依赖名字与索引规则，非 POI 或泛化名称会失配
- 目前只有当前 Day 视角，没有稳定 trip-level map workspace state

### 7.12 目前哪些能力已实现，哪些只是预留，哪些完全没有

- 当前 Day marker

- active marker

完全没有成品入口：

- 搜索
- 图层
- 定位
- 两点距离查询
- 时间查询

即使这一阶段不实现搜索、距离、路径优化，也建议右侧地图工作区预留：

  保证地图成为右侧主视觉

- 底部地点详情带
  显示当前 active point、距离、建议、下一跳

- 侧边折叠信息区
  route stats / warnings 可收纳，而不是永远挤压地图

## 8. Data Sources And Trip Identity

### 8.1 Workspace 当前使用的数据来自哪里

当前 Workspace 主数据来自：

- `localStorage`

具体读取：

- `travel-planning:trip-plan`
- `travel-planning:trip-request`

即：

- `loadTripPlan()`
- `loadTripRequest()`

页面不是从 URL 参数直接拿 trip。

### 8.2 trip identity 当前如何识别

当前 trip identity 很弱。

已有：

- `savedTripId` metadata
- `savedTripTitle`
- `restoredAt`
- `savedAt`

缺失：

- URL 级 `tripId`
- `slug`
- `externalId`
- `local draft id`
- `blank draft id`

结论：

- 已保存 trip 有弱 identity
- AI 新生成 trip 主要是匿名本地状态
- 空白计划 identity 目前不存在

### 8.3 AI 生成计划、历史计划、空白计划三类数据是否能统一到同一个 Workspace 数据结构

当前只能统一前两类：

- AI 生成计划
- 历史计划恢复

两者都能落到同一组：

- `trip-plan`
- `trip-request`

但空白计划还进不来。

所以当前并没有真正完成三入口统一。

### 8.4 localStorage 与 Supabase trips 的关系是否清晰

部分清晰，但还不够。

当前逻辑：

- Supabase 是云端存储层
- localStorage 是当前工作态读取层
- 历史 trip 打开时，先从 Supabase 拉，再写回 localStorage
- 保存当前 trip 时，根据 `savedTripId` 判断 create / update

问题：

- Workspace 自身不知道现在打开的是云端 trip、匿名 trip 还是未来 blank draft
- localStorage 与真实 trip identity 没有强绑定

### 8.5 地图点位数据来自哪里

地图点位来自：

- `/api/enrich-trip`
- `buildTripResultEnrichment()`
- `enrichTripPlanWithRoutes()`
- 生成 `TripResultEnrichment.enrichment.mapPoints`

点位不是直接来自原始 `TripPlan`。

### 8.6 enrichment 数据是否是地图展示的必要依赖

是。

当前右侧地图展示依赖：

- `desktopSelectedInsight`
- `insight.mapPoints`

而 `insight` 的来源是：

- `tripEnrichment`

没有 enrichment，就没有当前这套地图点位与 route summary。

### 8.7 是否存在数据结构混乱、字段命名不一致、空状态缺失的问题

存在。

主要问题：

- `TripSourceType` 同时有 `manual`、`ai_generate`、`ai_generated`、`user_created`，边界不清
- `TripPlanDraft` 有 source meta，但正式 `TripPlan` 不带同层来源语义
- `savedTripId` 与当前 `tripPlan` 没有强绑定校验
- `WorkspaceRoutePage` 的 empty state 不等于 blank workspace support
- 空白计划缺少 draft id / state / schema
- `trip-enrichment` 已恢复到 localStorage，但页面不直接消费

## 9. UI/UX Fit Against Wanderly Direction

### 9.1 当前哪些地方符合

符合的部分：

- 纸张、水彩、书签、旅行档案感素材已经在用
- 左侧计划阅读区已经成立
- 整体不黑、不赛博、不像传统后台
- 左右分栏方向已经对了

### 9.2 哪些地方偏离

偏离主要集中在右侧和整体角色定义：

- 右侧不是完整地图工作区，而是 inspector 侧栏
- 页面仍保留明显 result page 演化痕迹
- 顶部更像工具条，不像旅行档案展开页
- 右侧被多张说明卡过度卡片化

### 9.3 当前是否误把地图做成了右侧小插件

是，基本可以这么判断。

更准确地说：

- 不是“小插件”那么轻
- 但确实还是“右侧地图预览 + inspector 组件”

还不是：

- 和左侧并列的完整地图工作区

### 9.4 哪些组件最需要优先精修

优先级最高的组件：

- `TripWorkspace`
  原因：它定义了左右角色分配，目前右侧仍被组织成 inspector

- `MapPanel`
  原因：它目前就是 route insight 卡片壳，不是 map workspace 壳

- `WorkspaceInspector`
  原因：当前右侧几乎都被它定义成了信息检查区

- `InspectorMapPreview`
  原因：名字和职责都在证明当前地图只是 preview

- `WorkspaceRoutePage`
  原因：入口、空状态、`/workspace` / `/result` 收口都在这里

### 9.5 Read Mode 第一眼是否像“打开一本旅行杂志 + 旁边展开完整地图”

否。

当前更像：

- 左边是旅行计划阅读稿
- 右边是路线检查栏

还不像：

- 左边打开一本私人旅行杂志
- 右边展开一整块可工作的地图台面

## 10. Issues

### P0

- P0.1 缺少空白创建入口
  当前不存在“从 0 进入 Workspace 手动创建计划”的真实链路。

- P0.2 `/result` 仍承担真实入口
  历史计划打开、登录回跳、保存回跳仍大量依赖 `/result`，导致 Workspace 主入口不统一。

- P0.3 右侧地图当前只是 preview / inspector，不是完整工作区
  这直接偏离你本次修正后的 Workspace 目标。

- P0.4 trip identity 无法支撑 AI / 历史 / 空白三种入口
  当前主要依赖同一组 localStorage 键，缺少 URL 级或 draft 级 identity。

- P0.5 空白计划没有 local draft id / blank draft state
  因此无法作为第三种正式入口接入同一 Workspace。

### P1

- P1.1 Day 与地图同步已存在，但仅限“当前 Day 的 inspector map”
  还不是 trip-level 完整地图工作流。

- P1.2 地图点位强依赖 enrichment
  enrichment 失败时，右侧地图能力会明显降级。

- P1.3 marker / block 匹配规则有脆弱性
  generic placeName、non-POI、unresolved point 都会影响联动稳定性。

- P1.4 Edit Mode 仍以 AI 重新生成整份计划为中心
  距离手帐积木式编辑仍有明显差距。

- P1.5 `WorkspaceSidebar`、`WorkspaceLanding`、相关 state 未收口
  说明当前结构还在过渡态。

### P2

- P2.1 多处文案仍引用 `/result`
  会持续强化旧结果页心智。

- P2.2 右侧内容过度卡片化
  让地图更像辅助组件，不像并列核心工作区。

- P2.3 顶部 Header 与 WorkspaceTopBar 还可以进一步清理角色边界
  避免后续再次出现导航层级混杂。

- P2.4 `TripSourceType` 命名边界松散
  后续扩展三入口时会增加理解成本。

## 11. Recommendations For Step 2-6

### Step 2：Workspace 入口与数据流梳理

- 统一 `/workspace` 为正式工作台入口
- 将 `/result` 降为兼容入口，不再承接新能力
- 打通三入口：
  AI 生成、历史计划、空白计划
- 为三入口定义统一 trip identity：
  saved trip id、local draft id、blank draft id
- 明确 `trip-plan`、`trip-request`、`trip-enrichment`、Supabase row 的主从关系

### Step 3：Workspace 主布局修正，确立左计划 + 右完整地图双工作区

- 保留 55 / 45 左右比例
- 左侧保持计划主叙事区
- 右侧从 `inspector` 结构重构为 `map workspace` 结构
- route stats / warnings / point list 改为可折叠附属面，而不是持续压地图
- 地图高度提升到接近右侧全高主容器

### Step 4：Read Mode 骨架与 UI 精修

- AI / 历史计划进入默认 Read Mode
- 提升旅行档案首屏气质
- 左侧强化杂志 / 手帐式阅读节奏
- 右侧形成“大地图 + 轻工具 + 选中详情”的阅读工作流

### Step 5：右侧地图工作区基础增强

至少完成：

- Day / Place / Marker 联动
- 明确 active point 状态
- 地图专属工具栏预留位
- 地图内或地图下方 detail band

并预留后续入口：

- 搜索
- 距离 / 时间查询
- 路线模式
- 多点路径优化
- 图层 / 定位

### Step 6：Edit Mode 最小可用骨架与 Workspace 收口

- blank trip 默认进入 Edit Mode
- 已有 trip 默认进入 Read Mode
- 让 Edit Mode 先成为最小可用的块级编辑面
- 保证 Read / Edit 共用同一 trip identity
- 清理 `/result` 旧文案与 inspector-only 叙事

## Audit Summary

当前 v1.8.3 Workspace 最重要的现实状态是：

- 左侧计划工作区已经初步成立
- 右侧底层地图能力也已经具备真实基础

但右侧成品形态还不是“完整地图工作区”，而是：

- `MapPanel -> WorkspaceInspector -> InspectorMapPreview`
- 本质仍是 route inspector 侧栏

因此，这一阶段最大的 P0 不是单个 UI 细节，而是四个结构问题：

- 缺少空白创建入口
- `/result` 仍承担真实入口
- 右侧地图仍是 preview / inspector，不是完整工作区
- trip identity 还不足以统一 AI / 历史 / 空白三入口

结论很明确：

Step 2 之后最该优先做的，不是继续给右侧叠更多路线卡，而是先把 Workspace 主入口、trip identity、双工作区布局和右侧地图角色彻底校正。
