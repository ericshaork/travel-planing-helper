# v1.7 Phase 1.2 Data Architecture

## 1. 本轮目标

v1.7 Phase 1.2 聚焦两件事：

1. 稳定 Workspace 架构，把 `TripWorkspace` 中的状态、effect 和 handler 从视图层抽离。
2. 为 Explore 建立独立数据基础，包括类型、schema、repository、Supabase migration 和 JSON importer。

本轮明确不做：

- 不改 UI
- 不改样式
- 不接入图片资源
- 不开发 Explore 页面视觉

## 2. Workspace 状态拆分

### 2.1 调整目标

在 Phase 1.1 中，`/result` 已经拆出 route/page 结构，但 [components/workspace/TripWorkspace.tsx](/C:/Users/10200/Desktop/travel_planing/components/workspace/TripWorkspace.tsx) 仍然承载了大量状态与交互逻辑。

本轮继续拆分，目标是让 `TripWorkspace` 回到“薄视图层”：

- 保留 JSX 结构与现有交互表现
- 将工作台状态收敛到 hooks
- 让后续 `/workspace`、`/result`、`Explore` 互相演进时更容易复用

### 2.3 各 hook 职责

#### `useTripSave`

负责：

- 当前工作台中的 `currentTripPlan`
- 重新生成后的本地持久化
- 重新生成后的当前计划替换

这层不是云端保存按钮逻辑，而是“当前工作台里的旅行计划状态”。

#### `useTripModification`

负责：

- `pendingChanges`
- `modificationDraft`
- `externalDraftVersion`
- block action 入篮
- quick modification
- 把 pending changes 合并进 regenerate draft
- regenerate 成功后的修改态重置

#### `useTripMap`

负责：

- `activeMapPointId`
- `activeItineraryBlockId`
- `unmatchedBlockPlaceName`
- map point 与 itinerary block 的双向选中联动
- map 选中态的归一化与清空

#### `useTripWorkspaceState`

这是聚合层 hook，负责：

- desktop / mobile workspace 状态
- sidebar 状态
- 当前 day 选择
- enrichment 拉取
- workspace 对外 handler 编排
- 串起 `useTripSave` / `useTripModification` / `useTripMap`

### 2.4 拆分后的结果

`TripWorkspace.tsx` 现在主要负责：

- 渲染 mobile workspace
- 渲染 desktop workspace
- 组合 `TripEditor` / `ModificationPanel` / `MapPanel` / `WorkspaceUtilityPanel`
- 用 hook 返回值驱动 UI

这样做的好处：

- 视图层更稳定
- handler 更集中
- 后续可继续拆出 `TripWorkspaceMobileView` / `TripWorkspaceDesktopView`

### 2.5 当前仍保留在视图层的内容

当前还保留在 `TripWorkspace.tsx` 的部分包括：

- `MobilePageShell`
- 一些 summary helper

这是有意保留的。因为本轮目标是先把“状态和行为”拆开，而不是继续大范围重写视图层。

## 3. Explore 数据层

### 3.1 设计原则

本轮明确采用：

- **Explore 独立内容模型**
- **不复用 `TripPlan` 作为 Explore 主模型**
- **不把 Explore 公共内容写进 `trip_plans`**

原因：

1. `TripPlan` 是 AI 生成结果页与用户私有计划的主 schema。
2. Explore 是公共内容资产，不是用户私有旅行记录。
3. Explore 需要额外的内容元数据、审核状态、发布状态、导入来源。

- [lib/explore/types.ts](/C:/Users/10200/Desktop/travel_planing/lib/explore/types.ts)

### 3.4 为什么不复用 `TripPlan`

`TripPlan` 缺失这些 Explore 关键维度：

- slug
- 发布 / 归档状态
- 审核状态
- 导入来源元数据
- 内容级 poi / food highlights
- 公共内容索引字段

所以这里采用：

- `TripPlan` 继续服务 AI 生成工作台
- `ExploreTripContent` 服务 Explore 公共内容平台

## 5. JSON Importer 设计

### 5.1 目标

为 `travel-content-pipeline` 输出的 JSON 建立“导入前规范化层”。

当前新增：

- [lib/explore/importer.ts](/C:/Users/10200/Desktop/travel_planing/lib/explore/importer.ts)

### 5.2 Importer 能力

已提供：

- `parsePipelineExploreTrip`
- `buildExploreTripSlug`
- `buildExploreTripImport`
- `buildExploreTripImportBatch`

### 5.3 当前 importer 处理方式

Importer 会：

1. 解析 pipeline JSON 必填字段
2. 兼容 `POI` / `poi`
3. 归一化 `daily_itinerary`
4. 归一化 `poiHighlights`
5. 归一化 `foodHighlights`
6. 自动生成 slug
7. 注入 source metadata
8. 输出 `ExploreTripContentInsert`

### 5.4 设计边界

当前 importer 只负责：

- 结构校验
- 内容归一化
- 导入前标准化

当前 importer 不负责：

- 实际命令行执行
- 文件扫描
- 数据库 migration 执行
- Explore 页面渲染

- `listPublished`
- `getBySlug`

### 6.2 设计意图

Repository 层的作用是：

- 隔离 Supabase row 与前端 domain model
- 避免 Explore 页面直接拼 SQL / row 字段名
- 为后续 `/explore` 与 importer 共用一套数据访问边界

## 8. 风险点

### 风险 1：`TripWorkspace` 视图层仍偏大

虽然状态和 handler 已经拆出，但 `TripWorkspace.tsx` 仍然包含大量 JSX。

后续建议：

- `TripWorkspaceMobileView`
- `TripWorkspaceDesktopView`

### 风险 2：Explore importer 目前是基础归一化，不是最终运营导入工具

当前 importer 已足够作为工程基础，但还不是完整的内容运营工具。

后续可能还需要：

- CLI 脚本
- 批量导入日志
- 重复 slug 检查
- 审核前 diff

### 风险 3：发布与审核后台还不存在

本轮只做数据层准备，没有做：

- admin 审核界面
- 内容发布工作流
- 手工修订界面

这意味着表结构已准备好，但运营闭环还没建。

## 10. 本轮结论

v1.7 Phase 1.2 已经完成两项关键准备：

1. Workspace 从“单个大组件堆行为”进一步演进为“视图层 + hooks 编排”
2. Explore 从“仅有规划”升级为“已有独立数据模型、schema、repository、migration 和 importer 基础”

- Phase 1.3 可以继续做工作台与页面边界清理
