# AGENTS.md - AI 自由行出行规划网站开发规范

## 1. 项目目标

本项目是一个面向自由行新手的 AI 出行规划网站。

用户通过一句话输入和分步补充信息，提供：

- 出发城市；
- 目的地城市；
- 出行日期或天数；
- 预算；
- 兴趣偏好；
- 出行风格；
- 必去地点；
- 不想去地点；
- 住宿、交通、作息等特殊要求。

系统需要结合天气 API 和 LLM，生成一份个性化、结构清晰、可执行的城市自由行方案。

MVP v0.1 的核心功能：

1. 一句话输入旅行需求；
2. 解析并展示核心字段；
3. 分步补充缺失字段；
4. 查询目的地天气；
5. 生成 Day by Day 行程；
6. 展示景点攻略和路线顺序；
7. 展示天气提醒和预算摘要；
8. 支持复制/导出 Markdown；
9. 支持补充要求后重新生成。

## 2. 开发优先级

开发时严格围绕 MVP，不要主动扩展非核心功能。

### 2.1 必须优先完成

- 首页自然语言输入；
- 分步补充页；
- 结果页；
- 天气 Provider；
- LLM 调用；
- 结构化 JSON 输出；
- 复制/导出；
- 重新生成。

### 2.2 暂时不要开发

除非用户明确要求，否则不要开发以下功能：

- 用户登录；
- 用户注册；
- 收藏行程；
- 历史记录；
- 支付；
- 下单；
- 真实酒店预订；
- 真实票务购买；
- 地图实时路线；
- 多人协作；
- App 端；
- 复杂后台管理系统。

## 3. 推荐技术栈

默认使用：

- Next.js；
- React；
- TypeScript；
- Tailwind CSS；
- Next.js Route Handlers；
- LLM API；
- QWeather 或 OpenWeatherMap。

MVP 暂不需要数据库。如果需要临时保存状态，优先使用前端状态和 `localStorage`。

## 4. 代码规范

### 4.1 TypeScript

- 所有业务数据必须定义 TypeScript 类型；
- 不要大量使用 `any`；
- 外部 API 返回值必须经过转换后再进入业务层；
- AI 输出必须经过 schema 校验后再渲染。

### 4.2 文件组织

请尽量按以下结构组织：

```text
/app
  /page.tsx
  /plan/page.tsx
  /result/page.tsx
  /api
    /parse-trip/route.ts
    /generate-trip/route.ts
/components
  /layout
  /trip
/lib
  /ai
  /weather
  /trip
  /utils
/docs
```

不要把所有逻辑写在一个大文件里。

### 4.3 组件规范

- 页面组件只负责组合，不放复杂业务逻辑；
- 表单组件负责输入和校验；
- 结果展示组件只负责展示 `TripPlan`；
- API 调用逻辑放在 `/lib` 或 API route 中；
- 组件命名使用 PascalCase；
- 函数命名使用 camelCase。

### 4.4 样式规范

- 默认使用 Tailwind CSS；
- 不要引入大型 UI 框架套用现成模板；如使用组件库能力，必须重做视觉样式，不能保留 Shadcn 或 Material UI 的默认外观；
- 页面气质应轻松、可信、有生活感，接近旅行手账和城市便签，而不是企业 SaaS 或炫技型 AI 工具；
- Tailwind 只作为样式工具，不使用其默认蓝紫色板作为主视觉；
- 颜色、圆角、阴影、纸张纹理等视觉常量应集中在 `styles/globals.css` 或统一 theme 配置中；
- 组件样式必须服务内容层级，结果页优先清楚、好读、方便复制；
- 不要为了炫技增加复杂动画；必要动画应轻微、自然，并避免机械的线性运动；
- 功能图标优先使用 Iconify，不使用 Emoji 充当主要功能图标。

## 5. AI 输出规范

### 5.1 必须结构化输出

LLM 生成旅行方案时必须返回 JSON，不要直接返回纯 Markdown 或长文本。

前端应基于 JSON 渲染结果页。

### 5.2 不得编造实时信息

AI 不允许编造以下内容：

- 实时票价；
- 实时酒店价格；
- 实时酒店余量；
- 景点实时开放状态；
- 实时排队情况；
- 未调用 API 的天气预警。

如果没有数据，应使用保守说法，例如：

```text
建议出行前在官方平台再次确认。
```

### 5.3 行程生成原则

生成行程时必须遵守：

1. 不要安排过满；
2. 尊重用户必去地点；
3. 避开用户不想去的地点；
4. 路线顺序要尽量合理；
5. 对自由行新手要写清楚怎么走；
6. 要结合天气信息做提醒；
7. 预算估算要符合用户预算范围；
8. 不确定的信息必须提示用户核实；
9. 景点攻略要实用，不要写成百科介绍。

## 6. 天气 API 规范

### 6.1 Provider 抽象

天气能力必须通过 `WeatherProvider` 抽象，不要在业务代码里直接绑定某一个 API。

推荐接口：

```ts
export interface WeatherProvider {
  getForecast(input: WeatherQuery): Promise<WeatherForecast>;
}
```

### 6.2 API 失败处理

如果天气 API 失败：

- 不要让整个生成流程失败；
- 继续生成基础行程；
- 在结果中提示“暂时无法获取实时天气”；
- 不要让 AI 声称已经结合实时天气。

## 7. Prompt 开发规范

### 7.1 Prompt 文件位置

Prompt 统一放在：

```text
/lib/ai/prompts.ts
```

不要散落在组件或 API route 中。

### 7.2 解析 Prompt

解析用户输入时：

- 只提取用户明确说出的内容；
- 不要猜测日期、预算、城市；
- 可以识别“轻松一点”“不想早起”“喜欢拍照”等隐含偏好；
- 返回缺失字段和追问问题；
- 输出 JSON。

### 7.3 生成 Prompt

生成行程时：

- 输入包括 `TripRequest` 和 `WeatherForecast`；
- 输出必须符合 `TripPlan` 结构；
- 需要包含每日行程、景点攻略、路线顺序、天气提醒、预算摘要、住宿区域建议、交通平台建议；
- 预算和交通价格只能做估算，不能伪装成实时数据。

### 7.4 重新生成 Prompt

重新生成时：

- 输入包括原始需求、旧方案和用户修改要求；
- 保留未被修改的约束；
- 应用新增要求；
- 直接返回完整新方案，不做局部 patch；
- 返回 `appliedChanges` 字段说明修改了什么。

## 8. 错误处理规范

API 错误返回统一结构：

```ts
export interface ApiErrorResponse {
  error: {
    code: string;
    message: string;
    details?: unknown;
  };
}
```

不要把原始错误堆栈直接展示给用户。

常见错误码：

- `INVALID_INPUT`
- `WEATHER_API_FAILED`
- `AI_GENERATION_FAILED`
- `AI_OUTPUT_INVALID`
- `UNKNOWN_ERROR`

用户看到的错误信息要友好，例如：

```text
暂时无法生成方案，请稍后重试。
```

## 9. 环境变量和安全

### 9.1 环境变量

使用 `.env.local` 存储密钥。

必须提供 `.env.example`，但不要填真实 Key。

示例：

```env
LLM_BASE_URL=
LLM_API_KEY=
LLM_MODEL=
USE_MOCK_AI=true

WEATHER_PROVIDER=qweather
QWEATHER_BASE_URL=
QWEATHER_API_KEY=
USE_MOCK_WEATHER=true
```

### 9.2 安全要求

- 不要把 API Key 写死在代码中；
- 不要在前端暴露服务端 Key；
- `.env.local` 必须加入 `.gitignore`；
- 对用户输入做长度限制；
- 对外部 API 错误做兜底处理。

## 10. UI/UX 注意事项

### 10.1 产品气质

本项目不是企业 SaaS，也不是炫技型 AI 工具。它应该像一个懂旅行的朋友，帮用户把复杂攻略整理成一份清楚、好读、能照着走的行程。

关键词：

- 轻松；
- 可信；
- 有生活感；
- 像旅行手账；
- 像城市便签；
- 不像模板站。

在视觉表达和交互文案之间做取舍时，优先保证自由行新手看得懂、敢相信、能执行。

### 10.2 视觉禁止项

不要使用以下常见 AI/SaaS 模板风格：

- 大面积紫色、靛蓝色或蓝紫渐变；
- 默认 Tailwind 蓝紫色板作为主视觉；
- Hero + 三张功能卡片的首页结构；
- 完美居中的大标题、副标题、CTA 按钮套路；
- 玻璃拟态卡片堆叠；
- 一屏三个等宽功能模块；
- 空泛的 AI 光效、发光球或科技网格背景；
- Shadcn / Material UI 默认外观；
- Emoji 作为主要功能图标；
- Lorem Ipsum 或空泛占位文案。

### 10.3 视觉建议

- 优先使用米白、纸张色、浅沙色、浅灰绿和墨色文字；
- 卡片可以借鉴行程便签、车票、小票、地图标注等旅行场景，但不能妨碍阅读；
- 背景可以使用轻微纸张纹理、噪点或柔和渐变，不要喧宾夺主；
- 页面布局允许轻微不对称，重要信息仍需保持稳定对齐和清晰层级；
- 组件要有定制感，不直接套用组件库默认样式；
- 结果页先保证阅读、复制和移动端浏览，再考虑装饰性细节。

### 10.4 文案规则

文案要像朋友说话，不要像 AI 产品发布会。

避免使用：

- “赋能”；
- “智能化解决方案”；
- “一站式”；
- “革新你的旅行体验”；
- “AI 驱动的下一代平台”；
- “无缝衔接”；
- “极致体验”。

文案应口语化、句子短、具体、有场景感，可以轻微幽默，但不要油腻或过度营销。

推荐方向：

- “别再开 12 个网页查攻略了”；
- “先把想去的地方丢进来”；
- “我帮你排个不绕路的版本”；
- “下雨的话，就别硬上海边了”；
- “预算紧？那就少折腾一点”；
- “这天别安排太满”。

### 10.5 首页

首页要让用户一眼知道这个网站是做什么的。

首页不要做传统 SaaS 三段式，也不要使用 Hero + 三张功能卡片。建议结构：

1. 一个偏口语的开场标题；
2. 一句话旅行需求输入框；
3. 几个真实示例输入；
4. 兴趣和风格快捷标签；
5. 一段很轻的能力说明，写清楚会生成什么，不夸大能力。

示例标题方向：

- “攻略太多？先扔给我。”；
- “你说想去哪，我来排路线。”；
- “别查到凌晨了，先生成一版。”。

输入框必须给出真实、具体的旅行示例，降低用户不知道怎么写的问题。

### 10.6 分步补充页

不要做成很长的复杂表单。优先展示已经识别出的信息，再让用户补充缺失信息。

### 10.7 结果页

结果页不要只展示一大段 AI 文本，必须使用卡片结构。

结果页重点顺序：

1. 旅行总览；
2. 天气提醒；
3. 预算摘要；
4. 每日行程；
5. 景点攻略；
6. 住宿建议；
7. 交通建议；
8. 复制/导出/重新生成。

结果页可以采用行程手账、城市便签、每日路线卡、小票式预算和天气提醒贴纸等表达，但不能影响阅读、信息层级和复制。

### 10.8 Loading 与错误提示

Loading 文案不要写“AI 正在赋能你的旅程”等空泛表达，可以使用：

- “先看天气”；
- “再排路线”；
- “避开绕路”；
- “把攻略压缩一下”。

错误提示要像人话，并明确告诉用户系统将如何继续：

- “天气接口刚刚没接上，先给你一版不含实时天气的。”；
- “这个预算有点紧，我会按省一点的方式排。”；
- “模型刚刚没吐出合格格式，再试一次。”。

## 11. 测试建议

至少使用以下测试输入：

```text
我想 7 月从深圳去厦门玩 3 天，预算 2500，喜欢海边、美食和拍照，不想太累。
```

```text
我想从广州去成都玩 4 天，喜欢美食和城市漫步，不想去太商业化的景点。
```

```text
我想去杭州玩两天，不想早起，预算 1500，想轻松一点。
```

需要验证：

- 字段解析是否正确；
- 缺失信息是否能追问；
- 天气失败是否能兜底；
- 结果是否结构化；
- 行程是否明显过满；
- 是否尊重必去/不想去地点；
- 复制/导出是否可用；
- 重新生成是否能应用修改要求。

## 12. 开发完成标准

MVP v0.1 可以认为完成，当且仅当：

1. 本地可以正常启动；
2. 用户可以完成从输入到生成的完整流程；
3. 天气 API 可以接入或有明确兜底；
4. LLM 可以返回结构化 `TripPlan`；
5. 结果页能按卡片展示方案；
6. 复制和导出 Markdown 可用；
7. 重新生成可用；
8. 没有明显硬编码 API Key；
9. 没有把所有逻辑堆在一个文件；
10. 非 MVP 功能没有被过度开发。

## 13. MVP v1.1 Development Rules

- v1.1 的主线是点击式行程积木编辑，轻量自然语言修改只作为辅助，不开发完整聊天系统；
- 继续复用现有 `TripPlan`、`DailyItinerary`、`ItineraryItem` 和 `/api/generate-trip`；
- `dailyItinerary[].morning / afternoon / evening` 分别映射为上午、下午、晚上的固定 Time Slot，不为展示概念复制一套 AI 输出 Schema；
- “不要这个 / 换一个 / 一定保留 / 加类似”和快捷修改只能先生成 `modificationRequest` 或轻量 constraints，不得自动提交；
- 用户确认后必须基于当前 `TripRequest`、完整 `previousPlan` 和修改要求重新生成完整 `TripPlan`，不做局部 Patch；
- 当前完整 `TripPlan` 始终是页面展示、复制和 Markdown 导出的唯一真实来源；
- v1.1 必须优先完成 `/plan` 缺失项摘要、字段级错误、点击定位、首错滚动，以及手机端多日摘要、Day 导航和修改入口；
- v1.1 不做拖拽、格子高低调整、隔板增删、精确时间轴、地图 API、真实距离、实时开放时间、真实票务/酒店、登录、收藏、数据库、历史版本或多人协作；
- 具体范围、数据流和阶段顺序以 `docs/PRD_v1.1.md`、`docs/TECH_DESIGN_v1.1.md` 和 `docs/V1.1_PHASE_PLAN.md` 为准；不得覆盖 MVP 1.0 的 `docs/PRD.md` 与 `docs/TECH_DESIGN.md`。
